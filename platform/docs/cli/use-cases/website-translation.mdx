# Use Case: Translating a Website

This guide walks through a complete workflow for translating website content using the Bowrain CLI and Bowrain's visual editor.

## Overview

1. **CLI**: Initialize a Bowrain project and configure file mappings
2. **CLI**: Run translation flows (AI, TM, QA)
3. **Bowrain**: Open local project in visual editor for manual review
4. **CLI**: Push to Bowrain Server for team collaboration (optional)

## Step 1: Initialize Project

Create a Bowrain project in your website repository:

```bash
# Initialize project with source and target locales
cd my-website/
bowrain init --name "Website Translation" --source en-US --targets fr-FR,de-DE,ja-JP

# Check word count for cost estimation
bowrain word-count content/**/*.md

# Configure file mappings in .bowrain/config.yaml
# Edit to map your local files to translation items
```

Example `.bowrain/config.yaml`:

```yaml
project:
  name: Website Translation
  source_locale: en-US
  target_locales:
    - fr-FR
    - de-DE
    - ja-JP

mappings:
  - local: content/**/*.md
    remote: pages/{path}
    format: markdown
  - local: src/locales/**/*.json
    remote: ui/{path}
    format: json
```

## Step 2: Run Translation Flows

Execute translation workflows from the CLI:

```bash
# List available flows
bowrain flow list

# Run AI translation flow
bowrain flow run ai-translate

# Or create a custom flow with QA checks
# .bowrain/flows/translate-website.yaml:
# name: translate-website
# description: Translate website content with AI and QA
# steps:
#   - tool: ai-translate
#     config:
#       provider: anthropic
#       model: claude-sonnet-4.5
#   - tool: qa-check
#   - tool: term-enforce

bowrain flow run translate-website
```

### Flow-Based Translation

Bowrain CLI flows are composable pipelines that run on your local files:

- **ai-translate**: Use LLMs (Anthropic, OpenAI, Ollama) for machine translation
- **tm-leverage**: Pre-fill translations from translation memory
- **qa-check**: Rule-based quality checks (whitespace, punctuation, placeholders)
- **term-enforce**: Validate terminology compliance
- **segmentation**: Split source text into sentences

Flows automatically process all files matching your `.bowrain/config.yaml` mappings.

### Editor Views

Bowrain offers multiple views for different translation tasks:

| View | Best For |
|------|----------|
| **Grid View** | Overview, batch operations |
| **Split Preview** | Context-sensitive translation with live rendering |
| **Focus View** | Distraction-free, segment-by-segment work |

### Using Terminology

1. Import a termbase (TBX, CSV, JSON) into `.bowrain/termbase.tbx`
2. Add `term-lookup` to your flow to show terminology suggestions
3. Add `term-enforce` to validate required terms are used correctly
4. The Context panel in Bowrain shows matched terms per block

## Step 4: Sync with Bowrain Server (Optional)

For team collaboration, connect your project to a Bowrain Server instance:

```bash
# Connect to server
bowrain init --server https://bowrain.example.com --project abc123

# Check sync status
bowrain status

# Push translations to server
bowrain push -m "Translate landing page content"

# Pull translations from teammates
bowrain pull
```

The push/pull workflow is similar to git:

| Git | Bowrain CLI |
|-----|------------|
| `git status` | `bowrain status` |
| `git diff` | `bowrain diff` |
| `git pull` | `bowrain pull` |
| `git push` | `bowrain push` |

Bowrain CLI uses content-addressed blocks for efficient incremental sync — only changed blocks transfer over the network.

## Tips

- **Initialize projects with `bowrain init`** to set up `.bowrain/` configuration
- **Define custom flows** in `.bowrain/flows/` for your specific translation workflow
- **Use `tm-leverage` flow** to auto-fill translations from translation memory
- **Import terminology** as `.bowrain/termbase.tbx` and add `term-lookup` to your flow
- **Run `pseudo-translate` flow** first to identify potential UI truncation issues
- **Use pre-push hooks** to enforce quality gates before pushing to server
- **Commit `.bowrain/config.yaml`** to git, but not `.bowrain/.sync-cache` (auto-gitignored)
- **Check `bowrain status`** before pushing to see what changed
