# Automation Walkthrough

This guide walks through setting up fully automated translation for a Docusaurus
site using the Bowrain toolchain. By the end, you will have:

- A project configuration with file mappings and automation rules
- Pseudo-translated content for layout validation
- One-command sync that pushes, translates, and pulls in a single step
- Local QA gates that run automatically before content reaches the server
- Server-side automation that triggers translation on push
- A GitHub Actions workflow for continuous translation in CI/CD

We use the same example Docusaurus site from the
[E2E Walkthrough](/docs/bowrain/e2e-walkthrough) (available at
[github.com/gokapi/docusaurus-e2e](https://github.com/gokapi/docusaurus-e2e)),
but focus entirely on the automation story.

## Step 2: Project Configuration

The `.bowrain/config.yaml` file defines how your project connects to the Bowrain
Server. It maps local files to their formats and declares automation rules.

The config uses an envelope format with `apiVersion`, `kind`, and `spec`:

```yaml
apiVersion: v1
kind: ProjectConfig
metadata:
  name: Acme Docs
spec:
  project:
    name: Acme Docs
    source_locale: en
    target_locales:
      - nb
      - qps
  mappings:
    - local: "i18n/en/**/*.json"
      remote: "{path}/{filename}"
      format: json
      target_path: "i18n/{locale}/{path}/{filename}"
    - local: "docs/**/*.md"
      remote: "{path}/{filename}"
      format: markdown
    - local: "blog/**/*.md"
      remote: "{path}/{filename}"
      format: markdown
  automations:
    - name: qa-before-push
      trigger: pre-push
      actions:
        - type: run_flow
          config:
            flow: qa-check
    - name: sync-on-push
      trigger: post-push
      actions:
        - type: wait_translate
          config:
            timeout: 5m
        - type: pull
```

### Key sections

**`mappings`** tell Bowrain which files contain translatable content and what format
to use when parsing them. The `local` pattern matches source files, `remote`
defines the server-side path, and `target_path` uses `{locale}` to write
translations (e.g., `i18n/nb/code.json` for Norwegian).

**`target_locales`** declare which languages you want translations for. When you
run `bowrain sync` or `bowrain pull`, these are the locales that get pulled.

**`automations`** define hooks that run at specific lifecycle points. Each rule
has a `trigger` (when to run) and a list of `actions` (what to do):
- **`pre-push`** runs _before_ content is uploaded. Use this for QA checks that
  gate the push — if they fail, content is not sent to the server.
- **`post-push`** runs _after_ content is uploaded. Use this to wait for
  translations and pull them automatically.

## Step 3: Pseudo-Translation for Testing

Before involving real translators or translation services, generate
pseudo-translations to validate your i18n setup. Pseudo-translation replaces
characters with accented variants and adds length padding to expose truncation
and layout issues:

```bash
bowrain pseudo-translate i18n/en/code.json \
  --target-lang qps \
  --output i18n/qps/code.json
```

The result looks like this:

```json
{
  "homepage.hero.title": {
    "message": "[!!! Büüüîîîld Àmàààzîîîng Thîîîngs wîîîth Àcmëëë !!!]"
  }
}
```

This makes problems visible immediately:
- **Truncation**: Pseudo-translated text is ~30% longer, exposing UI elements that
  clip or overflow
- **Missing i18n keys**: Any hardcoded strings stand out against the accented text
- **Encoding issues**: Accented characters reveal encoding problems before real
  translations arrive

## Step 4: One-Command Sync

The `bowrain sync` command combines push, wait, and pull into a single operation.
It uploads your source content, waits for the server to finish processing
(including any server-side automations like AI translation), and then pulls the
translated content back:

```bash
bowrain sync --timeout 2m
```

This runs three phases:

1. **Push** — uploads source content to the server based on your mappings
2. **Wait** — polls the server until all target locales have translations ready
3. **Pull** — downloads translated content and writes it to the mapped file paths

After sync completes, you have translated files in your project:

```bash
$ ls i18n/
en/  fr/  nb/  qps/

$ head -20 i18n/nb/code.json
{
  "homepage.hero.title": {
    "message": "Bygg fantastiske ting med Acme"
  },
  "homepage.hero.subtitle": {
    "message": "Acme gir deg verktoyene og infrastrukturen du trenger for a levere raskere og mer palitelig."
  }
}
```

:::note
The `--timeout` flag sets a maximum wait time for translations. If the server

want a longer timeout (e.g., `--timeout 5m`).

## Step 5: Local Automation Rules

The `automations` section in `config.yaml` defines hooks that run automatically
at specific points in the push/pull lifecycle. This gives you QA gates without
any manual steps.

### Pre-push QA

The `pre-push` automation runs a flow before content is uploaded:

```yaml
automations:
  - name: qa-before-push
    trigger: pre-push
    actions:
      - type: run_flow
        config:
          flow: qa-check
```

The referenced flow is defined in `.bowrain/flows/qa-check.yaml`:

```yaml
name: qa-check
description: Run quality checks on source content

steps:
  - tool: qa-check
  - tool: term-check
```

When you run `bowrain push`, the QA flow executes first. If any check fails,
the push is blocked and you see a report of the issues:

```bash
$ bowrain push
Running pre-push automation: QA checks
  ✓ leading-trailing-whitespace  (27 blocks checked)
  ✓ missing-placeholders         (27 blocks checked)
  ✓ empty-target                 (27 blocks checked)

Pushing to http://localhost:8080...
  i18n/en/code.json                 27 blocks
  docs/intro.md                      8 blocks
Done. 74 blocks pushed.
```

This prevents broken content from reaching the server or your translators.

The `post-push` hook triggers after a successful push. In the example above,

generated as soon as content arrives.

## Step 6: Server-Side Automation

The Bowrain Server has its own automation engine that can respond to events like
content pushes, translation completions, and schedule-based triggers. While local
automations (in `config.yaml`) run on the CLI side, server-side automations run
on the server and can coordinate across multiple projects and locales.

### The automation rule editor

In the Bowrain web app, navigate to **Automations** in the project sidebar to
configure server-side rules:

![Automation rules editor](pathname:///img/web-app/dark/automations.png)

### Event types

Server-side automations respond to events:

| Event | Description |
|-------|-------------|
| `content.pushed` | New source content was uploaded |
| `translation.completed` | A locale reached 100% translation |
| `translation.reviewed` | All translations in a locale are reviewed |
| `schedule` | Time-based trigger (cron expression) |

### Actions

Each rule can trigger one or more actions:

| Action | Description |
|--------|-------------|
| `ai-translate` | Translate all untranslated blocks using AI |
| `mt-translate` | Translate using a configured MT provider |
| `run-flow` | Execute a named flow (QA, formatting, etc.) |
| `notify` | Send a notification (email, webhook, Slack) |
| `export` | Export translated files to a connector (Git, S3, etc.) |

### Example: auto-translate on push

A common server-side rule: when content is pushed, automatically translate all
untranslated blocks using AI, then notify the team:

```
Event:   content.pushed
Actions: ai-translate → notify (Slack)
```

This means developers push content and translations appear automatically. The
team gets a Slack notification when translations are ready for review.

## Step 7: CI/CD with GitHub Actions

The final piece of the automation puzzle: running `bowrain sync` in CI so
translations stay up to date automatically whenever source content changes.

Here is the GitHub Actions workflow (``.github/workflows/bowrain-sync.yml``):

```yaml
name: Translation Sync

on:
  push:
    branches: [main]
    paths:
      - 'i18n/en/**'
      - 'docs/**'
      - 'blog/**'

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Bowrain CLI
        run: |
          curl -fsSL https://bowrain.dev/install.sh | bash

      - name: Sync translations
        env:
          BOWRAIN_SERVER_URL: $\{{ secrets.BOWRAIN_SERVER_URL }}
          BOWRAIN_TOKEN: $\{{ secrets.BOWRAIN_TOKEN }}
        run: |
          bowrain sync --timeout 5m

      - name: Commit translations
        run: |
          git config user.name "bowrain-bot"
          git config user.email "bot@bowrain.dev"
          git add i18n/
          git diff --staged --quiet || \
            git commit -m "chore: sync translations"
          git push
```

### How it works

1. **Trigger**: The workflow runs on pushes to `main` that modify source content
   files (English i18n JSON, docs, or blog posts).
2. **Sync**: `bowrain sync` pushes the changed content, waits for server-side
   automation to translate it, and pulls the results.
3. **Commit**: If new translations were pulled, they are committed and pushed
   back to the repository.

The result is a fully automated pipeline:

```
Developer pushes code
  → GitHub Actions detects source content changes
    → bowrain sync pushes to server
      → Server auto-translates via AI
        → bowrain sync pulls translations
          → Bot commits translated files back to repo
```
