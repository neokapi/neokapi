# Walkthrough

This guide takes you from zero to a fully automated, multilingual Docusaurus site
in under 10 minutes. You will:

1. Set up a Bowrain project with file mappings and automation rules
2. Generate pseudo-translations to validate your i18n setup
3. Push content and sync real translations with one command
4. Claim the project and set up an API token
5. Automate the entire pipeline with GitHub Actions

We use a real Docusaurus site as the example project:
[github.com/gokapi/bowrain-example-docusaurus](https://github.com/gokapi/bowrain-example-docusaurus).
This project is connected to [Bowrain Cloud](https://dev.bowrain.cloud) with live
badges showing real translation progress.

## Prerequisites

- **Bowrain CLI** installed ([Installation](/docs/bowrain/installation))

```bash
# Verify bowrain is installed
bowrain --version
```

## Step 1: Initialize the Project

Clone the example project and create a Bowrain project:

```bash
git clone https://github.com/gokapi/bowrain-example-docusaurus.git
cd bowrain-example-docusaurus

bowrain init --anonymous --name "Acme Docs" --source en --server https://dev.bowrain.cloud
```

This creates a `.bowrain/` directory — think of it as a git-like project root
for localization:

```
bowrain-example-docusaurus/
├── .bowrain/
│   ├── config.yaml          # Project configuration
│   └── flows/
│       └── qa-check.yaml    # QA flow definition
├── .github/
│   └── workflows/
│       └── bowrain-sync.yml # CI/CD workflow
├── docs/                    # Markdown documentation
├── blog/                    # Blog posts
├── i18n/
│   ├── en/                  # English source strings
│   │   └── code.json
│   ├── nb/                  # Norwegian (AI-translated)
│   └── qps/                 # Pseudo-translated
└── docusaurus.config.ts
```

The init command prints two important things — save them both:

- **Claim URL** — use in [Step 4](#step-4-claim-the-project-and-create-an-api-token)
  to take ownership of the project
- **Project ID** — stored in `.bowrain/config.yaml`

:::tip

All commands work from any subdirectory in your project.

The init command stores a **claim token** in `.bowrain/config.yaml`. This token
grants push/pull access to the project until it is claimed. **Do not commit the
claim token to a public repository.** Add it to `.gitignore` or claim the project
first (Step 4), which removes the token from the config.
:::

## Step 2: Configure File Mappings and Automation

The `.bowrain/config.yaml` file maps your source files to Bowrain Cloud and
declares automation rules that run at lifecycle hooks:

```yaml
apiVersion: v1
kind: ProjectConfig
metadata:
  name: Acme Docs
spec:
  project:
    name: Acme Docs
    source_locale: en
    target_locales:
      - nb
      - qps
  server:
    url: https://dev.bowrain.cloud
  mappings:
    - local: "i18n/en/**/*.json"
      remote: "{path}/{filename}"
      format: json
      target_path: "i18n/{locale}/{path}/{filename}"
    - local: "docs/**/*.md"
      remote: "{path}/{filename}"
      format: markdown
    - local: "blog/**/*.md"
      remote: "{path}/{filename}"
      format: markdown
  automations:
    - name: qa-before-push
      trigger: pre-push
      actions:
        - type: run_flow
          config:
            flow: qa-check
    - name: sync-on-push
      trigger: post-push
      actions:
        - type: wait_translate
          config:
            timeout: 5m
        - type: pull
```

### Key sections

**`server`** points to Bowrain Cloud. The `project_id` is added automatically
by `bowrain init`. No credentials are stored in the config — authentication
is handled separately via `bowrain auth login` or the `BOWRAIN_AUTH_TOKEN`
environment variable.

**`mappings`** define which files contain translatable content and what format to
use. The `local` pattern matches source files, `remote` defines the server-side
path, and `target_path` uses `\{locale\}` to write translations to the right
directory (e.g., `i18n/nb/code.json` for Norwegian).

**`automations`** define hooks that run at specific lifecycle points:
- **`pre-push`** runs before content is uploaded — use for QA gates that block
  broken content from reaching the server
- **`post-push`** runs after upload — use to wait for AI translation and pull
  results automatically

The QA flow referenced by `pre-push` is defined in `.bowrain/flows/qa-check.yaml`:

```yaml
name: qa-check
description: Run quality checks on source content

steps:
  - tool: qa-check
  - tool: term-check
```

## Step 3: Pseudo-Translate for Testing

Before involving real translators, generate pseudo-translations to validate your
i18n setup. Pseudo-translation replaces characters with accented variants and adds
padding to catch truncation bugs:

```bash
bowrain flow run pseudo-translate
```

The result:

```json
{
  "homepage.hero.title": {
    "message": "[!!! Büüüîîîld Àmàààzîîîng Thîîîngs wîîîth Àcmëëë !!!]"
  }
}
```

Any hardcoded strings, truncation, or encoding issues become immediately visible
against the accented text. Preview with `npm run build -- --locale qps`.

## Step 4: Push, Translate, and Pull with One Command

The `bowrain sync` command combines push, wait, and pull into a single operation:

```bash
bowrain sync --timeout 2m
```

Three phases run automatically:

1. **Push** — uploads source content to Bowrain Cloud based on your mappings
2. **Wait** — polls the server until translations are ready (server-side
   automation translates via AI)
3. **Pull** — downloads translated content and writes it to the mapped paths

Because the `pre-push` automation is configured, QA checks run first:

```
Running pre-push automation: qa-before-push
  ✓ qa-check        (27 blocks checked)
  ✓ term-check      (27 blocks checked)

Pushing to https://dev.bowrain.cloud...
  i18n/en/code.json                 27 blocks
  docs/intro.md                      8 blocks
Done. 74 blocks pushed.

Waiting for translations...
  ✓ nb: 74/74 blocks translated

Pulling translations...
  i18n/nb/code.json                 27 blocks
Done. 74 blocks pulled.
```

After sync completes, the Norwegian translations are in your project:

```bash
$ head -10 i18n/nb/code.json
{
  "homepage.hero.title": {
    "message": "Bygg fantastiske ting med Acme"
  },
  "homepage.hero.subtitle": {
    "message": "Acme gir deg verktøyene og infrastrukturen du trenger for å levere raskere og mer pålitelig."
  }
}
```

## Step 5: Claim the Project and Create an API Token

The project was created anonymously. To manage it and set up CI, you need to
claim it and create an API token.

### Claim the project

Open the claim URL from Step 1 in your browser:

1. Sign in (or create an account) on [Bowrain Cloud](https://dev.bowrain.cloud)
2. The project appears in your workspace dashboard
3. The claim token is automatically removed from `.bowrain/config.yaml`

<video controls autoPlay loop muted width="100%" style={{maxWidth: '900px'}}>
  <source src="/video/web-app/dark/login-and-workspace.webm" type="video/webm" />
</video>

### Create an API token for CI

After claiming, create a long-lived API token for use in GitHub Actions:

```bash
bowrain auth login                           # authenticate with Bowrain Cloud
bowrain auth token create --name "CI" --expire-days 90
```

The command prints the token once — copy it and store it as a GitHub repository
secret named `BOWRAIN_AUTH_TOKEN`:

```bash
# Or set it via the GitHub CLI
gh secret set BOWRAIN_AUTH_TOKEN --repo your-org/your-repo
```

:::warning
API tokens (`bwt_...`) are shown only once at creation. Store the token
immediately as a GitHub secret. You can revoke and recreate tokens at any time
with `bowrain auth token list` and `bowrain auth token delete`.
:::

### Invite translators

From the Bowrain web app:

1. Navigate to **Settings** → **Members**
2. Click **Invite member**
3. Enter the translator's email and select the **Member** role

The translator receives an email with an invitation link. After accepting, they
can work on translations in the **Translation Editor** — with inline TM
suggestions, terminology lookup, and AI-assisted drafts.

## Step 6: Server-Side Automation

While the CLI runs local automation (QA gates, sync), Bowrain Cloud has
its own event-driven automation engine. Navigate to **Automations** in the
project sidebar to configure server-side rules.

### Built-in: auto-translate on push

The server ships with a default rule: when content is pushed, automatically
translate all untranslated blocks using AI. This is what makes `bowrain sync`
return translations — the server translates in the background while the CLI
waits.

### Custom rules

Create rules that respond to events:

| Event | Description |
|-------|-------------|
| `content.pushed` | New source content was uploaded |
| `translation.completed` | A locale reached 100% translation |
| `translation.reviewed` | All translations in a locale are reviewed |

Each rule triggers one or more actions:

| Action | Description |
|--------|-------------|
| `ai-translate` | Translate using AI/LLM |
| `run-flow` | Execute a named flow (QA, formatting) |
| `notify` | Send notification (email, webhook, Slack) |
| `export` | Export to a connector (Git, S3) |

Example: translate on push, then notify Slack when all locales are done.

## Step 7: Automate with GitHub Actions

The final piece: run `bowrain sync` in CI so translations stay up to date
whenever source content changes.

Two actions work together:
- [`gokapi/setup-bowrain`](https://github.com/gokapi/setup-bowrain) — installs the Bowrain CLI
- [`gokapi/bowrain-action`](https://github.com/gokapi/bowrain-action) — runs sync and commits translations

```yaml
name: Bowrain Sync

on:
  workflow_dispatch:
  push:
    branches: [main]
    paths: ['i18n/en/**', 'docs/**', 'blog/**']

permissions:
  contents: write

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: gokapi/setup-bowrain@v1
        with:
          token: $\{{ secrets.GOKAPI_REGISTRY_TOKEN }}
          auth-token: $\{{ secrets.BOWRAIN_AUTH_TOKEN }}
          server: https://dev.bowrain.cloud

      - uses: gokapi/bowrain-action@v1
        id: sync

      - name: Summary
        if: steps.sync.outputs.committed == 'true'
        run: echo "Translations committed at $\{{ steps.sync.outputs.commit-sha }}"
```

The pipeline:

```
Developer pushes code
  → GitHub Actions detects source content changes
    → setup-bowrain installs CLI + configures auth
      → bowrain-action syncs translations + commits
        → Translated files committed back to repo
```
