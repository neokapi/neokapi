# Maturity-Ladder Governance in Mature Ecosystems — Hardening neokapi's L0–L4 Format-Maturity Model

Research date: 2026-06-11. Focus: mechanics of multi-level support/maturity ladders — graduation criteria, demotion/sunsetting, automated vs human certification, and how support promises are published — as prior art for a per-format maturity ladder in a many-format processing engine maintained by one human plus AI agents.

## Findings

### 1. Rust target tiers — the closest analog: per-unit support tiers with explicit user promises

Source: [Target Tier Policy, rustc book](https://doc.rust-lang.org/rustc/target-tier-policy.html) (current, originally [RFC 2803](https://rust-lang.github.io/rfcs/2803-target-tier-policy.html), merged via [rfcs#2803](https://github.com/rust-lang/rfcs/pull/2803)).

- **The promise per tier is defined by what CI enforces, not by aspiration**: Tier 1 = "guaranteed to work" — builds **and passes the full test suite** in CI; regressions block. Tier 2 = "guaranteed to build" — CI builds it, tests may not pass. Tier 3 = "the codebase has support" — no CI, no guarantee it even builds. The tier label is literally a name for a CI gate.
- **Promotion is evidence-against-checklist**: a promotion proposal "must quote the corresponding requirements verbatim and respond to them" — the rubric text itself is the application form. Tier 3→2 goes via Major Change Proposal to the compiler team (+ infra team); →Tier 1 requires a **full RFC** approved jointly by compiler + infrastructure teams. A target "must have already received approval for the next lower tier, and spent a reasonable amount of time at that tier" — "multiple stable releases of Rust should typically occur between promotions." No tier-skipping.
- **Maintainer-count requirements scale with tier**: Tier 3 ≥1 named maintainer, Tier 2 ≥2, Tier 1 ≥3, plus Tier 1 demands "multiple production users … across multiple organizations." A tier is partly a measure of *bus factor*, not just code state.
- **Externality protections**: lower tiers must not tax the project — "must not impose burden on the authors of pull requests"; tier 2/3 maintainers may not block PRs over their target's test failures; tier 2 CI "must not take substantially longer than the current slowest target." Tier 1 flips the burden: maintainers must *provide* CI resources ("available for continuous exclusive use").
- **Demotion is normal and documented**: tier 2/3 removal requires the PR be CCed to target maintainers and "communicated widely to the Rust community before being dropped from a stable release"; tier 1 demotion requires a full RFC. Historical demotions: Apple 32-bit targets tier 2→3 ([RFC 2837](https://github.com/rust-lang/rfcs/pull/2837), 2019); all tier-2 MIPS targets demoted to tier 3 in Rust 1.72 ([rust-lang/rust#115238](https://github.com/rust-lang/rust/pull/115238), 2023 — trigger: unmaintained, LLVM bugs, no active maintainers). Demoted targets are not deleted — they drop to tier 3 (code stays, support promise withdrawn).
- **Anti-self-dealing rule**: "If any member of an approving Rust team serves as one of the maintainers of a target … they must recuse themselves from any approval decisions regarding the target's tier status." Promotion is judged by a party other than the one doing the work.
- **Non-promissory escape hatch**: "Neither this policy nor any decisions … shall create any binding agreement or estoppel"; requirements use RFC-2119 MUST/SHOULD/MAY, and "approving teams may grant exceptions" for SHOULDs — judgment is explicitly retained over the checklist.
- **Publication form**: [Platform Support page](https://doc.rust-lang.org/rustc/platform-support.html) — one row per target under tier headings, with **per-target capability columns alongside the tier** (`std` support ✓/*/?, `host` tools yes/no, notes), and dedicated per-target sub-pages listing maintainers and build/test instructions. I.e., a single headline tier *plus* a small capability vector — not a single scalar.

### 2. Kubernetes — graduation criteria as a template, anti-perma-beta deadlines, machine-tracked feature stages

- **KEP template institutionalizes graduation criteria**: every enhancement files a KEP whose template has a mandatory "Graduation Criteria" section plus a machine-readable `kep.yaml` (stage, milestone alpha/beta/stable) ([KEP template](https://github.com/kubernetes/enhancements/blob/master/keps/NNNN-kep-template/README.md)). Release sign-off requires concrete evidence per stage: e2e tests for all beta API operations; for GA, "all GA Endpoints must be hit by Conformance Tests within one minor version of promotion" and a "Minimum Two Week Window for GA e2e tests to prove flake free." A separate **Production Readiness Review** questionnaire (rollback, monitoring, scalability) is "required prior to targeting to a milestone."
- **Beta is a deadline, not a resting state**: [KEP-1635 "Prevent perma-beta"](https://github.com/kubernetes/enhancements/tree/master/keps/sig-architecture/1635-prevent-permabeta) and the [Moving Forward From Beta blog (2020-08)](https://kubernetes.io/blog/2020/08/21/moving-forward-from-beta/): a REST API may stay in a single beta version at most 9 months / 3 releases; if it doesn't progress it is deprecated with announced removal. [KEP-3136](https://github.com/kubernetes/enhancements/blob/master/keps/sig-architecture/3136-beta-apis-off-by-default/README.md) (stable in v1.24) additionally made **new beta APIs off-by-default** — beta stopped being a de-facto GA, restoring meaning to the level.
- **Deprecation policy = demotion contract with users** ([API deprecation policy](https://kubernetes.io/docs/reference/using-api/deprecation-policy/)): numbered rules — Rule #3: an API "may not be deprecated in favor of a less stable version" (no silent demotion of stability); Rule #4a: GA may be deprecated but **not removed** within a major version; beta removable 9 months/3 releases after deprecation; alpha removable anytime. Deprecation is **machine-communicated in-band**: since v1.19, deprecated API calls return an RFC 7234 `Warning` header, an audit annotation `"k8s.io/deprecated":"true"`, and an `apiserver_requested_deprecated_apis` metric labeled with `removed_release` — users can query *their own usage* of things about to be removed.
- **Feature status is machine-tracked and generated**: the [feature gates reference](https://kubernetes.io/docs/reference/command-line-tools-reference/feature-gates/) tables (feature, default, stage, since, until) are generated from source-code metadata, with a separate [removed-gates page](https://kubernetes.io/docs/reference/command-line-tools-reference/feature-gates-removed/) preserving history. GA gates are themselves removed after a deprecation window — the ladder cleans up after itself.

### 3. TC39 stages — tests as the graduation gate; champions; honest deaths

Source: [TC39 process document](https://tc39.es/process-document/); [inactive proposals list](https://github.com/tc39/proposals/blob/main/inactive-proposals.md).

- **Entrance criteria per stage are about artifact completeness**, not effort: Stage 1 = champion + problem prose + repo; Stage 2 = draft spec text (placeholders OK); Stage 2.7 = **complete spec text + sign-off from assigned reviewers and editors**; Stage 3 = "sufficient testing and appropriate pre-implementation experience"; Stage 4 = **two compatible implementations passing test262 tests** + shipping experience + spec PR. The 2.7→3 transition exists precisely to make *conformance tests* the gate: test262 tests must be authored and submitted during 2.7.
- **The executable test suite is the certification instrument**: stage 4 is not "the committee likes it" but "two independent implementations pass the shared test corpus." This is the strongest prior art for neokapi's spec.yaml-as-graduation-gate.
- **Champions = named per-unit owners**: each proposal has champion(s) who present advancement and "may request regressing proposals to earlier stages or withdrawal entirely, requiring committee consensus and documented rationale." If a champion disappears, another delegate may take over — or the proposal stalls.
- **Demotion/death is recorded, with reasons**: the inactive-proposals list catalogs withdrawn/rejected proposals. `Object.observe` was withdrawn after years at stage 2 (2015); `SIMD.js` was **removed from Stage 3** because implementers chose to pursue SIMD in WebAssembly instead ([tc39/ecmascript_simd](https://github.com/tc39/ecmascript_simd)). Stage regression from 3 is rare but real, and the rationale is preserved publicly.

### 4. CNCF — institutionalized periodic re-certification and a deliberate archiving ramp

- Levels: sandbox → incubating → graduated; criteria now live as application **issue templates** (the old [graduation_criteria.md](https://github.com/cncf/toc/blob/main/process/graduation_criteria.md) redirects to `.github/ISSUE_TEMPLATE/template-{incubation,graduation}-application.md`) — i.e., the checklist *is* the application form, like Rust.
- **Annual review with a default-to-demote rule** ([sandbox annual review](https://github.com/cncf/toc/blob/main/process/sandbox-annual-review.md)): sandbox projects must self-file a yearly review PR; "If an annual review isn't submitted within two months of notification, the TOC will take this as a sign that the project is not under active maintenance and the TOC is likely to decide to archive the project." Failure to re-certify is itself the demotion trigger — no fresh evidence of failure needed.
- **Archiving is slow, public, and reversible** ([archiving.md](https://github.com/cncf/toc/blob/main/process/archiving.md)): triggers include health-dashboard metrics, failure to meet *current* acceptance criteria, security unresponsiveness, or being feature-complete; process = months-long health inquiry → stakeholder notification → summary statement → 2 weeks public comment → 2-week TOC vote needing 2/3 majority. Archived projects: repos remain available, trademarks stay with the foundation, services withdrawn, "cannot actively advertise being a CNCF project," and may be reactivated via the normal lifecycle. The artifact survives; the *endorsement* is withdrawn.

### 5. Automated scoring — OpenSSF Scorecard and Best Practices badge: what keeps scores credible

- **Scorecard** ([checks.md](https://github.com/ossf/scorecard/blob/main/docs/checks.md), [scorecard.dev](https://scorecard.dev/)): ~20 fully automated checks, each 0–10, each assigned a risk level; the aggregate is a **risk-weighted average** (Critical=10, High=7.5, Medium=5, Low=2.5 weights — [TrustSource summary](https://support.trustsource.io/hc/en-us/articles/8528908713362-Understanding-the-OpenSSF-Score), [scorecard README](https://github.com/ossf/scorecard)). Checks with graded sub-tiers (e.g. Branch-Protection's 5 cumulative tiers → 3/6/8/9/10 points) show how to score a single axis as a mini-ladder.
- **Honesty mechanisms**: scores are recomputed continuously by a central service over the top ~1M repos (no self-reporting); checks docs explicitly flag low-confidence checks ("A low score is therefore not a definitive indication that the project is at risk" for CI-Tests/Fuzzing/SAST) — the rubric documents its own blind spots.
- **Criticism / measurement validity**: the Scorecard team's own paper ([arXiv:2208.03412](https://arxiv.org/pdf/2208.03412), 2022) concedes scores are *practice* proxies, not outcome measures; [arXiv:2210.14884](https://arxiv.org/pdf/2210.14884) ("Do Software Security Practices Yield Fewer Vulnerabilities?") found weak and sometimes inverse correlation between aggregate score and vulnerability outcomes — the aggregate number is the least defensible part of the system. Gaming is acknowledged: checks reward specific detectable configurations, so "check the boxes without real security" is possible ([Ry Walker research note](https://rywalker.com/research/openssf-scorecard)). Lesson: **automated scores stay credible per-check; the scalar aggregate is where validity collapses.**
- **Best Practices badge** ([bestpractices.dev/criteria](https://www.bestpractices.dev/en/criteria)): three levels (passing ~50 criteria, silver ~+30, gold ~+20, each level a hard prerequisite of the next); MUST/SHOULD/SUGGESTED grading; mostly **self-certification but with mandatory evidence fields** — criteria require `{Met URL}` or written `{Met justification}` / `{N/A justification}`, so even self-assessment leaves an auditable evidence trail. A minority of criteria are auto-verified by the BadgeApp. Weakness: no real staleness/revocation mechanism — badges effectively permanent, which is exactly what CNCF's annual review fixes.

### 6. Web-platform Baseline — computing one top-line status from per-implementation data

Source: [Baseline definition, web-platform-dx/web-features](https://github.com/web-platform-dx/web-features/blob/main/docs/baseline.md); [repo README](https://github.com/web-platform-dx/web-features/blob/main/README.md).

- **Two-tier computed status with a time-decay promotion**: "newly available" (interoperable) when a feature is supported in **all** current stable releases of the core browser set (Safari iOS/macOS, Chrome Android/desktop, Edge, Firefox Android/desktop); the **keystone date** is "the last release date on which a browser introduced support" — i.e., **the worst implementation sets the clock**. "Widely available" is purely mechanical: "keystone date is on or before today's date minus 30 months." Everything below is "limited availability."
- **Lowest-axis-wins aggregation**: the top-line status is the *minimum* across implementations, computed from [@mdn/browser-compat-data](https://github.com/mdn/browser-compat-data), with partial implementations excluded from counting as support. No averaging — one lagging browser keeps the feature out of Baseline entirely.
- **Hand-authored feature boundaries, computed status**: humans (the W3C WebDX Community Group, via the web-features-set owners group) decide what counts as "a feature" (the grouping/keys); the status per feature is then derived mechanically and **republished as a versioned npm package** (`web-features`) consumed by MDN, caniuse, webstatus.dev. Status changes propagate by data release, not by announcement.

### 7. Other ladders — distinctive mechanics only

- **Go port policy** ([go.dev/wiki/PortingPolicy](https://go.dev/wiki/PortingPolicy)): binary split — first-class ports ("broken builds block releases"; Google's Go team owns the builders) vs secondary ("a change that breaks a secondary port will not necessarily be rolled back… port maintainers are responsible"). Entry: accepted proposal + **2 named maintainers** + a named builder maintainer + a builder already running *before* merge. Demotion: a port is "broken" if its builder fails repeatedly in a dev cycle with no active fix; "If a port is broken in release 1.N, then the core Go team can choose to remove the port from release 1.N+1" — but "Removing a formerly working port should be a last resort. Finding a new maintainer is always preferable."
- **Node.js platform tiers** ([BUILDING.md](https://github.com/nodejs/node/blob/main/BUILDING.md)): Tier 1 and Tier 2 *both* block releases on test failures; tier 2 differs only in that "Infrastructure issues may delay the release of binaries." Experimental: "Test failures … do not block releases." Tier moves are allowed **only at major release lines** — the support matrix is per-release-line and immutable within it. A support promise is scoped to a version, not to "now."
- **Debian release-architecture qualification** ([arch_policy](https://release.debian.org/testing/arch_policy.html)): **numeric** qualification bar — "Five Developers need to certify … actively developing on this architecture, and … at least 50 users"; ~98% of the archive must build; redundant 24/7 buildds; security/sysadmin/release teams hold a **veto**. Re-qualification happens *every release cycle*; failing ports get "multiple warnings with verbose explanation" then move to debian-ports (a lower-promise home, not deletion).
- **Apache Project Maturity Model** ([community.apache.org](https://community.apache.org/apache-way/apache-project-maturity-model.html)): pure human self-assessment checklist with stable criterion IDs (CD10, QU30, CS20…) across code/licenses/releases/quality/community/consensus/independence; binary compliance, "a guide; not a requirements document." Distinctive: criterion IDs make assessments diffable across years.
- **FINOS lifecycle** ([lifecycle docs](https://community.finos.org/docs/governance/software-projects/project-lifecycle/), [archived stage](https://community.finos.org/docs/governance/software-projects/stages/archived/)): everything enters at Incubating regardless of maturity; activation via checklist + TOC approval; **mechanical inactivity trigger** — "no activity in the last 180 days" makes a project eligible for archiving.

### 8. Cross-cutting answers

**(a) Promise vs work-tracking.** The successful ladders define each level as *what the user may rely on and what the project will do when it breaks* — Rust tier 1 / Go first-class / Node tier 1 all reduce to "regressions at this level block release." Internal progress states (K8s alpha→beta, TC39 stages 0–2.7) are explicitly *not* user promises; K8s learned the hard way that letting an internal stage (beta-on-by-default) function as a de-facto promise corrupts the ladder, and fixed it with KEP-3136 + the perma-beta deadline. Conflation fails in both directions: users treat work-states as promises (perma-beta), or promises get diluted into aspirations (Best Practices badges that never expire). Keep two artifacts: a *support tier* (promise, changes rarely, demotion is a public event) and a *scorecard* (work-tracking, changes on every audit run).

**(b) Demotion/sunsetting mechanics.** Triggers seen: failing CI for a cycle with no fix in progress (Go), no maintainer/upstream-dead (Rust MIPS), failure to file re-certification (CNCF 2-month rule), mechanical inactivity (FINOS 180 days), failing the current acceptance criteria after a rubric update (CNCF archiving). Grace periods: Debian "multiple warnings"; Go one release; K8s beta 9 months/3 releases with in-band telemetry warnings; CNCF months of inquiry + 2-week comment + 2-week vote. Destination is almost never deletion: Rust → tier 3 (code stays, promise withdrawn); Debian → debian-ports; CNCF → archived-but-forkable; K8s → removed-gates history page. Communication: CC the named maintainer, announce before the release that drops the promise, and (K8s) make the *user's own usage* of the demoted thing observable.

**(c) Re-certification cadence and automatability.** CNCF annual reviews and Debian per-release re-qualification are human-heavy because the evidence is social (community health, maintainer counts). Everything Rust/Go/Node/K8s/Baseline gate on is **CI-shaped** — build, test-suite pass, conformance corpus, flake-free window, telemetry — and automates completely. For a solo maintainer + agents: the only ritual that resists automation is *judgment about the rubric itself* and promotion approval; the evidence collection, scoring, dashboard regeneration, and "stale level" detection are all cron-able. The CNCF default-to-demote rule ("no review filed → archive") is the key import: absence of a fresh green audit should itself decay the published status, so a dead automation can't silently preserve stale promises.

**(d) Single level vs vector.** Evidence: Rust publishes headline tier **plus** per-target std/host columns and per-target doc pages — tier for the promise, vector for capability detail. Baseline collapses a per-browser vector into one top-line status via **minimum (keystone = last browser to ship), never averaging**, and still exposes the underlying per-browser data. Scorecard publishes both per-check scores and a weighted-average aggregate — and the aggregate is precisely the part academic evaluation found weakly valid ([arXiv:2210.14884](https://arxiv.org/pdf/2210.14884)). Pattern: *promises aggregate by minimum; averages are for dashboards only.* Publish the vector, derive the headline as min (possibly min over a designated "core axes" subset), never as a weighted mean.

**(e) Anti-gaming when scorer == worker.** Mechanisms found: (1) Rust's recusal rule — approver may not be the target's maintainer; (2) TC39 — graduation evidence is an *external, shared* test corpus (test262) plus *independent* second implementation, not the champion's own claims; (3) Scorecard — central re-computation from raw artifacts rather than self-reported status; (4) Best Practices — even self-certification must attach evidence (`{Met URL}`) that a third party can audit; (5) parity against an independent reference implementation (Baseline's multi-browser requirement; TC39's two-implementation rule) makes the gate something the worker cannot edit. For an AI agent that both remediates and re-scores: keep the rubric + scorer code change-controlled separately from remediation runs (agent may not touch the audit script in the same change that improves a score), require evidence artifacts (fixtures, corpus files, parity outputs) checked into the repo, and anchor the top levels on oracles outside the agent's control (Okapi-Java parity harness, external spec corpora, real-world reference files).

## Design implications for neokapi

**Split the promise from the score.** Publish two things per format: (1) a **support tier** — a small set of user-facing promise levels with release-gating semantics, changed only by an explicit promotion/demotion event; (2) the **maturity vector** — the multi-axis audit output (engine fidelity, vocabulary fidelity, editor embedding, knowledge assets, corpus), recomputed on every audit run. The current L0–L4 conflates these (Rust separates them; K8s's perma-beta is the failure mode). Concretely: keep L0–L4 as the audit rubric, and *derive* 3 promise tiers from it.

**Per-level guarantee wording (recommended, modeled on Rust/Go/Node):**
- **Supported** (≈L3–L4 on all core axes): "Guaranteed to round-trip. Regressions in this format's test corpus, malformed-input suite, and Okapi parity block any release. Byte-exact output for unmodified content. Bugs here are release-blocking."
- **Maintained** (≈L2): "Guaranteed to parse and write. The format builds and its conformance corpus runs in CI; round-trip fidelity is tested but byte-exactness and parity are not guaranteed. Regressions are fixed in the next release, not hotfixed."
- **Available** (≈L0–L1, and the bridge/plugin tier): "Exists and may work. Not exercised by release-gating CI. Use with verification; file issues with samples." — Use Rust's RFC-2119 register and its non-estoppel disclaimer ("this policy involves human judgment; it creates no binding agreement").
- State each promise as *what CI enforces* — a tier claim that isn't backed by a CI gate is marketing, per the Rust model.

**Promotion mechanics.** Adopt Rust's "quote the requirement verbatim and respond" format: a promotion is a PR/issue using a template that embeds the rubric text per criterion with an evidence link per line (Best Practices' `{Met URL}` pattern). Require time-in-grade for the top tier (e.g., one release cycle at Maintained, mirroring "multiple stable releases between promotions") and a flake-free window for the corpus (K8s's "two week window … to prove flake free") before Supported. The audit script can pre-fill the template; the human approves. New formats enter at Available regardless of how complete they land (FINOS: "all projects initially land in incubating state").

**Anti-perma-L2 rule.** Import KEP-1635: a format may not sit indefinitely in a "promotion intended" state. Tag each non-Supported format with `target_tier` + `target_by` (a release, in the format's spec.yaml or maturity manifest, exactly like kep.yaml's `stage`/`milestone` fields); the audit flags overdue ones, and overdue means either re-commit (new date, logged) or formally park it as Available. This turns the work queue into a self-cleaning system.

**Automated re-certification with default-to-decay.** Run the audit (the existing format-triage skill) on a schedule; every published tier carries a `last_certified` date + audit-run link. Import CNCF's 2-month rule inverted for automation: **if a format's tier hasn't been re-certified within N days (audit didn't run, or ran red), the dashboard automatically shows it as "certification stale" and after a grace period demotes the displayed tier** — a dead cron can't preserve a stale promise. All evidence neokapi gates on is CI-shaped (corpus pass, parity, byte-exact round-trip), so unlike CNCF this re-certification is ~free for a solo maintainer.

**Demotion triggers and exit ramp.** Triggers, in order of automation: (1) red audit on a gating axis persisting past a grace window (Go: "failed multiple times in a development cycle … and maintainers are not actively working on a solution"); (2) stale certification (above); (3) upstream/spec dead or corpus unobtainable (Rust MIPS analog); (4) rubric upgraded and format fails the *new* bar (CNCF archiving explicitly checks "whether the project meets current acceptance criteria"). Grace period: one release with a published warning (Debian's "multiple warnings with verbose explanation"; Go's 1.N→1.N+1). Destination: never delete — demote Supported→Maintained→Available, and for formats that can't sustain even Available, move the reader/writer to a **plugin/bridge tier** (the debian-ports / Rust-tier-3 move: code survives, promise withdrawn, installable on demand). Communicate demotions in release notes *before* the release that drops the promise, and keep a `demotions.md` ledger with rationale (TC39's inactive-proposals.md; K8s's removed-gates page).

**Vector vs single level: publish the vector, headline by minimum.** Follow Baseline + Rust, reject Scorecard's weighted mean: the per-format dashboard row shows the headline tier computed as **min over the gating axes** (engine fidelity, corpus, malformed-input; decide explicitly which axes gate) with the non-gating axes (editor embedding, knowledge assets) shown as columns that inform but don't cap the tier — exactly Rust's `std`/`host` columns, which qualify but don't redefine the tier. Baseline's keystone-date idea also ports: a format's "Supported since" date is the date its *last* gating axis went green, and "battle-tested" status could be time-derived (Supported + 2 releases with no fidelity regressions ≈ "widely available" = newly + 30 months). Never publish a 0–10 average per format — [arXiv:2210.14884](https://arxiv.org/pdf/2210.14884) is the cautionary evidence that aggregate scalars lose validity.

**Rubric versioning and re-scoring.** Version the rubric itself (`rubric_version` in the audit output and in each format's recorded score), and on any rubric edit re-run the audit fleet-wide and record level changes attributable to the rubric change, not the format (CNCF re-checks archived candidates against *current* criteria; Debian re-qualifies every cycle; Apache's stable criterion IDs make assessments diffable across years — give every neokapi criterion a stable ID like `EF-30`, `RC-10`). A rubric change that would demote published tiers triggers the same grace-period + release-note machinery as a code regression.

**Scorer/worker separation for AI agents.** (1) The audit script, rubric, and spec.yaml assertion semantics live behind a change-control boundary: a remediation agent may add tests/fixtures/code but may not modify the scorer or relax an assertion in the same change that improves a score — enforce with CODEOWNERS-style path rules and a CI check that flags PRs touching both `audit` paths and format code. (2) Anchor the top tier on **oracles the agent can't edit**: the Okapi-Java parity harness (the independent second implementation, TC39's two-implementation rule), externally sourced reference corpora, and spec citations resolvable to the published spec. (3) Require every "criterion met" claim to point at a checked-in evidence artifact (fixture, parity report, corpus manifest) — the `{Met URL}` discipline — so a human (or a *different* reviewing agent) can audit cheaply. (4) Human approval remains the gate for tier *changes* in either direction (Rust's recusal rule, adapted: the agent that remediated may not be the approver), while everything else — scoring, dashboards, staleness detection, demotion warnings, promotion-template pre-fill — runs unattended.

**Machine-readable status, generated publication.** Mirror K8s feature gates and web-features: a single checked-in machine-readable status file per format (or one manifest) with `tier`, `tier_since`, `target_tier`, `target_by`, `rubric_version`, `last_certified`, per-axis levels — from which the /format-maturity dashboard, docs badges, and CLI (`kapi formats list` showing tier) are all *generated*. In-band communication on demotion paths: when a user processes a format whose tier is below Maintained or scheduled for demotion, emit a one-line warning (K8s's `Warning` header + deprecation metrics pattern) so users discover support changes at the point of use, not on the website.
