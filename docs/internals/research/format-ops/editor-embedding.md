# In-native-editor integration surfaces for a localization/review tool

Research stream for neokapi format-ops. Researched June 2026; dates noted where sources carry them. Focus: for each editor ecosystem that "owns" a format neokapi processes, what content access the embedding API actually grants, the auth model, the distribution/review gate, and which TMS competitors already sit there — distilled into a defensible "editor embedding" maturity ladder.

## Findings

### 1. Microsoft Office Add-ins (Word / PowerPoint / Excel, office.js) — owns docx/pptx/xlsx (neokapi `openxml`)

**Content access — Word is deep, PowerPoint is shallow, and the gap is the load-bearing fact.**

- Word has a strongly-typed application-specific object model (`Word.Body`, `Range`, `Paragraph`, `ContentControl`) layered over the Common API ([Word JS API overview, updated 2025-05-29](https://learn.microsoft.com/en-us/office/dev/add-ins/reference/overview/word-add-ins-reference-overview)). Ranges support `insertText`, OOXML fragment get/insert, and search. [WordApi 1.4](https://learn.microsoft.com/en-us/javascript/api/requirement-sets/word/word-api-1-4-requirement-set) added bookmarks, **change tracking** (whether tracking is on, `getTrackedChanges()` on ranges/content controls, filtering content controls by tracking state), comments, custom XML parts, and fields — i.e., a translation-review add-in in Word can write target text *as tracked changes with comments*, which is the native review idiom.
- Any Office add-in can pull the **entire current file** via `Office.context.document.getFileAsync(Office.FileType.Compressed)` — the live .docx/.pptx as byte slices (default 4 MB slices, configurable; `ReadWriteDocument` permission) and POST it anywhere ([Get the whole document, updated 2026-03-23](https://learn.microsoft.com/en-us/office/dev/add-ins/develop/get-the-whole-document-from-an-add-in-for-powerpoint-or-word)). There is **no symmetric whole-file write-back**; writes must go through the object model (or `insertFileFromBase64`-style insertion). This means: *full-fidelity extraction from inside the editor is trivial; full-fidelity merge from inside the editor is not* — merge must be reconstructed via ranges/OOXML fragments or done out-of-band.
- PowerPoint's app-specific API lagged for years and only recently became usable for text: 1.1 (2018) was just `createPresentation`; shape add/format arrived in 1.4 (2022); **select slides/text ranges/shapes only in 1.6 (Dec 2024)**; bindings/tables in 1.8 (2025), table formatting in 1.9, accessibility/background in 1.10 (Jan 2026). Volume-licensed perpetual Office is frozen at 1.5 (Office 2024); iPad supports only 1.1 ([PowerPoint requirement sets, updated 2025-12-16](https://learn.microsoft.com/en-us/javascript/api/requirement-sets/powerpoint/powerpoint-api-requirement-sets)). So an embedded pptx review surface is feasible on current Microsoft 365 but cannot assume the install base Word can.

**Auth.** SSO is moving to **Nested App Authentication (NAA)** against Entra ID; legacy Exchange tokens were turned off August 2025 ([NAA FAQ](https://learn.microsoft.com/en-us/office/dev/add-ins/outlook/faq-nested-app-auth-outlook-legacy-tokens), [SSO in Office Add-ins](https://learn.microsoft.com/en-us/office/dev/add-ins/develop/sso-in-office-add-ins)). Add-ins declare permissions in the manifest (`ReadWriteDocument` / unified-manifest `Document.ReadWrite.User`).

**Distribution.** AppSource requires Microsoft Marketplace certification, and **re-certification only on manifest changes — the add-in's web code can be redeployed at any time without review** (certification policies referenced from the publishing docs above). Sideloading and centralized admin deployment bypass the store entirely for enterprise pilots.

**Existing translation tools.** Word's first-party Review→Translate (Microsoft Translator) sets user expectations, but **major TMSes do not ship embedded Word add-ins** — Smartling/Phrase/Crowdin treat .docx as a file format ingested by connectors, and Smartcat's Word story is bilingual-docx round-trip ([Smartcat bilingual files](https://www.smartcat.com/blog/bilingual-files-in-smartcat-supported-not-thwarted/)). The embedded-review slot in Word is comparatively **unoccupied** — relevant because neokapi/bowrain already prototyped exactly this surface (the in-repo Microsoft 365 add-in workstream, PR #776: task pane + Check/Terms/Translate backend).

### 2. Google Workspace add-ons (Docs/Slides/Sheets) — owns gdocs-as-docx exports, sheets

Two coexisting models with very different power:

- **Editor add-ons** (legacy Apps Script model): HTMLService sidebar/dialog UI plus full server-side document access — `DocumentApp.getActiveDocument()` walks body/paragraph/text elements of the open doc ([Extend Google Docs](https://developers.google.com/apps-script/guides/docs)). All logic runs in Google's Apps Script runtime (quotas, no arbitrary backend runtime), desktop-only.
- **Google Workspace add-ons** (current strategic model): one deployment across Gmail/Calendar/Drive/Docs/Sheets/Slides, but **card-based UI only ("HTML/CSS interfaces … can't be used"), no document-context methods (`getActiveSpreadsheet()` etc. unavailable), no simple triggers, ≤100 widgets/sections per card** ([Workspace add-on restrictions](https://developers.google.com/apps-script/add-ons/guides/workspace-restrictions), [add-on types](https://developers.google.com/workspace/add-ons/concepts/types)). Editor access is mediated by per-file scope grants (`drive.file` flow) and then the Docs/Sheets/Slides **REST** APIs ([editor scopes](https://developers.google.com/workspace/add-ons/concepts/editor-scopes)).
- Net: in Google's editors, a rich in-pane review UI forces you onto the legacy Editor-add-on track; the modern track pushes you toward "card UI + REST round-trip", i.e., structurally a **workflow connector, not an embedded editor**. Distribution is the Workspace Marketplace plus OAuth verification (restricted scopes trigger security assessment).
- Existing tools: TMS vendors integrate Google Docs via Drive connectors server-side, not in-editor; in-editor offerings are mostly thin MT add-ons. Same "slot unoccupied" pattern as Word. (bowrain PR #776 also built the Google Card/Apps-Script add-on pair.)

### 3. Figma plugins — owns design text (UI copy upstream of every string-catalog format)

- Plugin code runs in a sandboxed realm with **full document-tree access**: `TextNode.characters` read/write, ranged styling via `setRange*`, style-run discovery via `getStyledTextSegments()`, mixed-style sentinel `figma.mixed`; edits require `figma.loadFontAsync()` per font and are blocked when `hasMissingFont` ([Working with text](https://developers.figma.com/docs/plugins/working-with-text/), [TextNode](https://www.figma.com/plugin-docs/api/TextNode/)). Crucially, **Figma's REST API cannot write file content** — the plugin sandbox is the *only* programmatic write path into a design, so every TMS must embed to round-trip.
- Distribution: Community publishing with human review, nominal **5–10 business days** (community reports of 20+), guidelines penalize plugins that "read or modify a Figma file without a user's explicit awareness" ([publishing](https://help.figma.com/hc/en-us/articles/360042293394-Publish-plugins-to-the-Figma-Community), [review guidelines](https://help.figma.com/hc/en-us/articles/360039958914-Plugin-and-widget-review-guidelines)). Private org plugins skip Community review. Network egress is declared in the manifest (`networkAccess.allowedDomains`).
- **This is the most saturated embedded-TMS surface in existence**: [Crowdin for Figma](https://store.crowdin.com/figma) (push strings + screenshots, key-naming patterns, pull translations into design copies), [Phrase Strings Figma plugin](https://support.phrase.com/hc/en-us/articles/5819515701916-Figma-Strings) ("connect keys" links layers to TMS keys), Lokalise, Smartling. The established pattern is **key-binding**: text layer ↔ TMS key, with pseudo/translated copies generated as parallel frames. A new entrant must match key-binding + screenshot context to be credible.

### 4. Adobe UXP for InDesign — owns the IDML-relevant editor

- UXP (the CEP/ExtendScript successor) is public in InDesign since v18.5 (2023), with UXP v8.0 shipping in InDesign v20.0; it gives a single modern JS runtime with **direct DOM access without the CEP CSInterface marshalling bridge** — third-party measurements cite a 5,000-text-frame iteration dropping from ~45 s (CEP) to ~2 s (UXP) ([Adobe UXP for InDesign](https://developer.adobe.com/indesign/uxp/), [CEP vs UXP 2026](https://metadesignsolutions.com/blog/custom-adobe-cep-vs-uxp-plugin-development)).
- Status, June 2026: **not yet at full ExtendScript parity; Adobe keeps CEP supported until parity**, but recommends UXP for all new panel work; the DOM API surface is "mostly the same" for migrating ExtendScript developers ([migration guides](https://developer.adobe.com/indesign/uxp/resources/migration-guides/), [Mapsoft UXP status](https://mapsoft.com/posts/uxp-indesign-workflow.html)). Distribution is a signed `.ccx` via Adobe Exchange review **or direct file distribution** — no forced store ([factory.dev UXP guide](https://factory.dev/blog/custom-indesign-plugins-uxp-react)).
- Practical meaning for neokapi: the live InDesign document (stories, text frames, conditional text) is scriptable in-process; **IDML file round-trip remains the dominant l10n integration** (e.g., Redokun, classic TMS connectors all consume IDML), so an embedded UXP panel that previews/merges translations into the *open* document is differentiated, and IDML stays the offline contract.

### 5. WordPress Gutenberg — owns post HTML/blocks

- The block editor is deliberately extensible via **SlotFill**: `registerPlugin` + `PluginSidebar` / `PluginDocumentSettingPanel` inject React UI into the editor shell, and the `wp.data` stores expose the full block tree for read/write ([SlotFills reference](https://developer.wordpress.org/block-editor/reference-guides/slotfills/), [PluginSidebar doc](https://github.com/WordPress/gutenberg/blob/trunk/docs/reference-guides/slotfills/plugin-sidebar.md)). There is no sandbox: a WP plugin is trusted PHP+JS, so content access is total. Distribution: wordpress.org review (one-time, then self-updated) or self-hosted commercial.
- Existing tools define three archetypes: **WPML** (block-aware duplication + its own translation editor, XLIFF export to TMSes — [WPML Gutenberg docs](https://wpml.org/documentation/getting-started-guide/translating-content-created-using-gutenberg-editor/)); **TranslatePress** (front-end visual in-context editing); **Weglot** (source-level interception + cloud dashboard, [plugin page](https://wordpress.org/plugins/weglot/)). I.e., WordPress already hosts all rungs of the ladder, which makes it a good reference ecosystem rather than an open slot.

### 6. Headless CMS app frameworks — own structured content (JSON-shaped)

- **Contentful App Framework**: apps are iframe SPAs in **seven locations** (entry field, entry sidebar, entry editor, page, home, config, dialog), with an App SDK for field-level read/write and **app identities** (non-user-bound backend credentials, event subscriptions); the marketplace is **partner-gated** ([App Framework docs](https://www.contentful.com/developers/docs/extensibility/app-framework/)). The canonical TMS apps live exactly here: [Phrase TMS app](https://www.contentful.com/marketplace/phrase-tms/) and [Smartling app](https://www.contentful.com/marketplace/smartling/) put "submit entry for translation + live status" in the entry sidebar.
- **Sanity**: the Studio is your own React app; plugins are npm packages with unrestricted document access — no marketplace gate. [sanity-plugin-transifex](https://github.com/sanity-io/sanity-plugin-transifex) serializes documents to **HTML fragments** for the TMS and deserializes on import (an explicit, documented lossy-bridge design); [sanity-plugin-studio-smartling](https://www.sanity.io/plugins/sanity-plugin-studio-smartling) is the same pattern; both support document-level and field-level i18n shapes, and the Transifex plugin documents a hard gap (internationalized-array fields export zero strings).
- **Storyblok**: **field plugins** (iframe + `postMessage`, [@storyblok/field-plugin SDK](https://www.storyblok.com/docs/libraries/js/field-plugin-sdk)) and **tool plugins** that embed in the Visual Editor and "can access the story while editing" ([tool plugins](https://www.storyblok.com/docs/plugins/tool-plugins)). Existing l10n: [Localazy field plugin](https://localazy.com/storyblok), Lokalise sync integration, DeepL field plugin.
- Pattern across all three: TMS embedding in CMSes is a **sidebar status + submit button** over a server-to-server connector; the heavy lifting (segmentation, TM, QA) never runs in the iframe.

### 7. VS Code extensions — own the code-adjacent formats (JSON/YAML/PO/MDX/properties)

- Extensions get unsandboxed workspace file access plus rich editor affordances: inline decorations, hovers, CodeLens, custom editors, language servers. Marketplace publishing is lightweight (publisher verification + automated scanning), no per-release human review.
- The benchmark is **i18n-ally** (maintained under Lokalise's org): inline annotation of keys with translations, in-place editing, missing-translation detection, MT, review flows, supporting dozens of frameworks/file formats ([repo](https://github.com/lokalise/i18n-ally), [marketplace](https://marketplace.visualstudio.com/items?itemName=Lokalise.i18n-ally)); **Sherlock/inlang** does inline visualize/edit/lint of translations ([marketplace](https://marketplace.visualstudio.com/items?itemName=inlang.vs-code-extension)). This surface is contested but commoditized — table stakes are inline annotations + jump-to-key + edit-in-place.

### 8. Canva apps — owns Canva designs

- The **Content Querying API** "retrieve[s] and edit[s] all text content" of the current design page, with **translation apps as a first-listed use case**; explicit limits: cannot identify which element holds a given text run, **reads text in random order (not linear)**, cannot highlight on canvas, unavailable in Canva Docs ([Content Querying API](https://www.canva.dev/docs/apps/design-guidelines/content-querying-api/)). The Design Editing API adds element-level manipulation, but **elements an app inserts become invisible to that app** afterwards ([Design Editing](https://www.canva.dev/docs/apps/design-editing/), [creating app elements](https://www.canva.dev/docs/apps/creating-app-elements/)). Public apps pass a human review team with a submission checklist ([app review process](https://www.canva.dev/docs/apps/app-review-process/)).
- Canva is therefore capped below "faithful" embedding: no stable text anchoring → no reliable segment↔node binding → translation apps there are batch rewrite tools, not review surfaces.

### 9. Browser-overlay / in-context web editors (no editor API at all)

- Four architectures observed across the TMS market ([locize comparison, 2025](https://www.locize.com/blog/in-context-translation-editor-tools-comparison/)): (a) **SDK/script injection** — [Crowdin In-Context](https://support.crowdin.com/developer/in-context-localization/) ships a JS snippet plus a **pseudo-language whose "strings" are identifiers**; the script finds those identifiers in the DOM and overlays an editor (requires regular DOM — "the Shadow DOM is invisible to our scripts" — and one Crowdin project per app); (b) **reverse proxy** — Smartling GDN / Localize ([ICE docs](https://help.localizejs.com/docs/in-context-editor)); (c) **browser extension** — Lokalise ([LiveJS](https://docs.lokalise.com/en/articles/1592525-livejs-web-in-context-editor)); (d) iframe preview inside the TMS. All of them need either app cooperation (pseudo-locale build) or traffic interception; none has stable anchors — they re-derive string identity from rendered text.

### 10. The maturity ladder the ecosystems actually support

Grounded in the above, "editor embedding" is a five-rung ladder, and each rung is gated by a specific API capability:

- **E0 — none**: format processed as files only; the owning editor never sees the tool.
- **E1 — faithful preview**: the tool renders the editor's output for context (screenshots from Figma plugin, HTML preview, TMS visual context). Gate: any read path (REST export, `getFileAsync`, screenshot API). Every ecosystem above permits E1.
- **E2 — round-trip workflow**: explicit push/pull between the editor's native store and the l10n pipeline with **stable identity binding** (Figma layer↔key à la Crowdin/Phrase; Contentful entry↔job; IDML story IDs; bilingual docx). Gate: addressable content units + a write path (plugin sandbox in Figma, REST in Google/Contentful, file insert in Office). Canva fails the stable-identity test; browser overlays fail it without pseudo-locale cooperation.
- **E3 — embedded add-in (live panel)**: a panel inside the editor reads the *open, unsaved* document and writes targets back in the editor's native idiom (Word tracked changes + comments; Figma `setRange*` with font loading; UXP DOM writes; Gutenberg SlotFill + `wp.data`; Sanity/Storyblok field write). Gate: app-specific object model with granular write + review semantics. Word (rich), Figma, InDesign-UXP, Gutenberg, Sanity/Storyblok/Contentful, VS Code all permit E3; PowerPoint only partially (text APIs ≥1.6, Dec 2024+); Google's modern add-on track and Canva structurally cap below E3.
- **E4 — continuous bidirectional sync**: editor events drive the pipeline without user push/pull (Contentful app identities + environment event subscriptions; WP hooks; CMS webhooks; Office shared-runtime event handlers). Only the CMS frameworks and WordPress make E4 first-class today; document editors (Office/Figma/InDesign) have no robust server-side change feed for content, so E4 there means polling exported files.

Cross-cutting distribution facts worth retaining: Office re-reviews only manifest changes (code redeploys free); Figma reviews every Community publish (5–10+ days) but private org plugins are instant; Adobe allows direct `.ccx` distribution; Contentful's marketplace is partner-gated while Sanity's is open npm; VS Code is effectively unreviewed. Auth models split into platform-identity (Office NAA/Entra, Google OAuth + restricted-scope audits) vs bring-your-own-token (Figma plugin → your backend, CMS app tokens).

## Design implications for neokapi

- **Treat "editor embedding" as a new ladder orthogonal to the L0–L4 format-maturity rubric.** Add an `embedding: E0–E4` field per format in spec.yaml/maturity data with the gate evidence (which API rung the owning editor permits), and surface it on the /format-maturity dashboard. The rubric above is defensible because each rung maps to a documented API capability, not vendor marketing.
- **Prioritize the unoccupied E3 slots: Word and Google Docs.** Figma/CMS slots are saturated by Crowdin/Phrase/Smartling/Lokalise; Word-embedded translation review (targets as tracked changes + comments via WordApi 1.4) and Docs sidebars are nearly empty. This aligns with the existing bowrain add-in workstream (PR #776, `bowrain/addin`) — the research says double down there, not on a Figma plugin first.
- **Exploit the asymmetry `getFileAsync` gives Office**: extract full-fidelity .docx/.pptx from inside the task pane → run the real neokapi openxml reader server-side (or in-pane via WASM) → write targets back through the object model. This keeps neokapi's faithful-engine moat doing the parsing while the add-in only does anchoring + UI — never reimplement extraction against office.js object trees.
- **The WASM (cgo-less) kapi build is the embedding payload.** Office task panes, Contentful/Storyblok iframes, Sanity Studio, VS Code, and Figma's UI iframe are all standard web contexts where the existing WASM engine runs; only Figma's logic sandbox and Apps Script exclude it (bridge via postMessage / REST respectively). Browser limits already documented in-repo (pure-Go SRX only, no ICU/FTS5) apply.
- **Stable anchoring is the moat feature at E2/E3.** Every incumbent re-derives identity (Figma plugins store key bindings in pluginData; Sanity serializes to HTML; overlays match rendered text). neokapi's block/run model + overlays can define a canonical `editor-anchor` overlay (e.g., Word content-control tag / Figma node-id+range / Docs named-range / CMS field path) so the same Block round-trips through both the file path and the embedded path. Spec.yaml could assert anchor survivability per format.
- **PowerPoint and Canva: cap expectations explicitly.** Record in the maturity data that pptx in-editor review requires PowerPointApi ≥1.6 (Microsoft 365 from Dec 2024; perpetual-license and iPad excluded) and that Canva cannot exceed E2-minus (no element identity, random text order). Avoid committing roadmap items the platform cannot honor.
- **For Google, build the dual-track consciously**: card-based Workspace add-on (modern, multi-host, marketplace-strategic) for submit/status (E2/E4), plus the Docs REST API for content; only adopt the legacy Editor-add-on track if an in-pane review UI proves necessary, since Google's strategic track forbids HTML UI and direct document context.
- **CMS apps should stay thin** (sidebar submit + status + diff preview), pushing all segmentation/TM/QA to the bowrain server — exactly the Phrase/Smartling Contentful pattern users already understand; differentiate on the faithful diff/preview (neokapi can render block-level diffs the incumbents can't).
- **InDesign: keep IDML as the contract, add a UXP panel later.** UXP is the right 2026 target (direct DOM, `.ccx` direct distribution, no store gate), but parity is still incomplete; an E1/E2 UXP panel (preview + merge-into-open-doc from a .klf/.kapi job) beats trying to re-host the pipeline in-process.
- **Distribution cadence shapes architecture**: put all changing logic behind your own URL (Office re-certifies only manifests; Figma re-reviews every version). Design add-ins as stable shells + server/WASM-driven behavior so the format engine can iterate weekly without store review cycles.
- **Auth: reuse the existing keychain/device-flow plumbing.** Office NAA/Entra and Google OAuth verification are platform-mandated; everywhere else (Figma, CMS iframes, VS Code) the pattern is "panel talks to your backend with your token" — i.e., bowrain's existing auth (device flow + `BOWRAIN_AUTH_TOKEN`) transfers directly.
