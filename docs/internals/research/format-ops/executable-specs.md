# Executable specifications & conformance suites for formats — prior art survey

Research date: 2026-06-11. Focus: example-based specs that machines (and AI) can generate/port tests from; how shared conformance corpora are structured, cited, versioned, and consumed across implementations.

## Findings

### 1. CommonMark: the spec *is* the test corpus (`spec.txt` + `spec_tests.py`)

- The CommonMark spec source is a single Markdown file, `spec.txt`, in which every normative example is an executable test embedded with a fenced shorthand: a long backtick fence with the info string `example`, containing the Markdown input, a line with a single `.` separator, then the expected HTML output ([commonmark/commonmark-spec](https://github.com/commonmark/commonmark-spec), [spec.txt](https://github.com/commonmark/commonmark-spec/blob/master/spec.txt)). The rendered spec (e.g. [spec.commonmark.org/0.31.2](https://spec.commonmark.org/spec)) shows the same examples side-by-side with a "Try it" link into the dingus.
- [`test/spec_tests.py`](https://github.com/commonmark/commonmark-spec/blob/master/test/spec_tests.py) is a spec-agnostic runner: `--dump-tests` emits the whole corpus as JSON; `--program $PROG` pipes each example's Markdown into any executable and diffs normalized HTML; `-p`/`-s` report pass detail and per-section pass percentages; `-t N` / `--pattern` select tests.
- Dumped test shape: `{"markdown": "...", "html": "...", "section": "Setext headings", "number": 65}` — note the **spec-section citation and stable example number ride along with every test**. The corpus is also republished as an npm package ([commonmark-spec](https://www.npmjs.com/package/commonmark-spec)) so third-party implementations consume it without cloning the repo.
- Versioning: the spec is released as a whole (0.31.2 current, Jan 2024); example numbers are stable *within* a spec version, and implementations advertise "CommonMark 0.31.2 compliant". Implementations pin a spec version and re-baseline on spec upgrades.
- Key mechanism to steal: **single-file, human-readable spec where prose, citation anchor, input, and expected output are physically adjacent**, plus a dump-to-JSON escape hatch so any runner/language can consume it.

### 2. JSON Schema Test Suite: language-agnostic JSON corpus shared by 30+ implementations

- [json-schema-org/JSON-Schema-Test-Suite](https://github.com/json-schema-org/JSON-Schema-Test-Suite) is "meant to be language agnostic and should require only a JSON parser"; converting cases into the local test framework is the implementer's job ([README](https://github.com/json-schema-org/JSON-Schema-Test-Suite/blob/main/README.md)).
- Layout: `tests/<draft-version>/` per released spec draft (draft3 → draft2020-12, plus a `latest` symlink), one `.json` file per keyword/feature (e.g. [`allOf.json`](https://github.com/json-schema-org/JSON-Schema-Test-Suite/blob/main/tests/draft2020-12/allOf.json)), with `optional/` for non-required behavior, `proposals/` for in-flight spec proposals, and `remotes/` for `$ref` resolution fixtures served at a well-known fake origin (`http://localhost:1234/...`).
- Case shape: a **test case** is `{description, schema, tests: [{description, data, valid: true|false}]}` — one shared setup (the schema) amortized over several positive/negative instances. Vocabulary is defined precisely (suite / case / test / runner).
- The suite validates *itself*: [`test-schema.json`](https://github.com/json-schema-org/JSON-Schema-Test-Suite/blob/main/test-schema.json) formally defines the case structure and `bin/jsonschema_suite check` runs in CI — a meta-schema guarding corpus quality.
- Consumption: "clone the main branch as a git submodule or git subtree"; main is guaranteed stable; old drafts (3/4) are frozen with minimal backports. 30+ language ecosystems (Go, Rust, Python, .NET, …) run the same corpus.
- Key mechanisms: **per-spec-version directories**, **required vs `optional/` split**, **meta-schema for the corpus**, **frozen old versions**, **shared fake-remote fixtures**.

### 3. toml-test: neutral exchange encoding + valid/invalid split + per-spec-version manifests

- [toml-lang/toml-test](https://github.com/toml-lang/toml-test) divides tests into `tests/valid/` (input `.toml` + expected `.json` sibling) and `tests/invalid/` ("decoders that reject 'invalid' tests pass").
- Deliberate **oracle-neutrality**: "To work as a language-agnostic test suite, a data exchange format is needed. TOML cannot be used, as it would imply that a particular parser has a blessing of correctness" — so expected output is JSON with a tagged-value encoding: every scalar is `{"type": "integer|string|float|bool|datetime|datetime-local|date-local|time-local", "value": "<string>"}`; tables→objects, arrays→arrays. No implementation is ground truth, so spec changes only require corpus updates.
- Harness protocol: a decoder-under-test is any executable that reads TOML on stdin and writes tagged JSON on stdout (exit 0/nonzero); encoders do the reverse. The `toml-test` Go binary orchestrates, so a new implementation needs only a ~30-line shim.
- **Spec-version manifests**: `tests/files-toml-1.0.0` and `tests/files-toml-1.1.0` are plain file lists declaring which tests belong to which TOML version; the runner takes `-toml=1.1`. One corpus, multiple spec versions, no duplication.
- Naming discipline: "Invalid tests … should try to test one thing and one thing only" and be **named after the fault they expose**.

### 4. yaml-test-suite: one source of truth compiled into multi-view per-test directories

- [yaml/yaml-test-suite](https://github.com/yaml/yaml-test-suite) authors tests as `src/*.yaml` (one or more tests per file, 4-char alphanumeric stable IDs like `AB3D`), then `make data-update` compiles them to a generated `data` branch where each test becomes a directory of orthogonal artifact files: `===` (name), `in.yaml` (input), `test.event` (canonical parser **event-stream DSL** — the semantic oracle), `in.json` (JSON equivalent), `out.yaml` (canonical dumper output), `emit.yaml` (emitter output), and `error` (presence marks expected-invalid).
- The event DSL is the crucial trick: it tests the *parse semantics* independent of any data model, while `in.json` tests the load semantics and `out.yaml`/`emit.yaml` test writing — **one example, multiple assertion planes**.
- Tags categorize tests (see [doc/tag-usage.md](https://github.com/yaml/yaml-test-suite/blob/main/doc/tag-usage.md)); tags also encode spec-version divergence (`1-3-*` marks where YAML 1.3 will differ) and normative disposition (`ok`, `err`, plus aspirational `want`/`dont` for "valid but shouldn't be" cases — the suite can encode *spec bugs*).
- Consumption: implementations pin a tagged data release branch (`git clone -b data-YYYY-MM-DD`); the raw `data` branch is squashed/force-pushed, releases are immutable.
- Cross-implementation comparison is institutionalized: the [YAML Test Matrix](https://matrix.yaml.info/) ([perlpunk/yaml-test-matrix](https://github.com/perlpunk/yaml-test-matrix)) runs ~15 parsers over the suite and renders an HTML matrix of per-test, per-implementation outcomes — the de-facto interop dashboard for YAML.

### 5. Web Platform Tests: directory-per-spec, in-file spec links, vendor expectations kept *outside* the shared corpus

- [WPT](https://github.com/web-platform-tests/wpt) (~50k+ tests) organizes by spec shortname; within a spec dir, a nested structure where "each subdirectory corresponds to the id of a heading in the specification … provides implicit metadata about the tested part" ([test-suite-design](https://web-platform-tests.org/test-suite-design.html)).
- Test types: [testharness.js](https://web-platform-tests.org/writing-tests/testharness.html) JS tests, reftests (render two pages, assert visual equality), wdspec, crashtests, manual. Per-directory `META.yml` / `WEB_FEATURES.yml` carry ownership/feature metadata.
- In-file spec citation: tests carry `<link rel=help>` pointing at the exact spec section anchor; CSS-suite lint verifies the anchors actually exist in the (bikeshed-generated) spec ([wpt#10053](https://github.com/web-platform-tests/wpt/issues/10053), [W3C testing minutes](https://www.w3.org/2020/10/21-testing-minutes.html)) — anchors are how tooling slices the suite by spec section.
- An auto-generated `MANIFEST.json` (from filenames + file contents) indexes test type/refs/timeouts ([Firefox source docs](https://firefox-source-docs.mozilla.org/web-platform/index.html)).
- **The decisive pattern for shared suites**: implementation-specific *expectations* never live in the corpus. wptrunner keeps a parallel `metadata/` tree of `.ini` files mirroring test paths (`<metadata>/spec/section/file.html.ini`), with `expected: FAIL|PASS|TIMEOUT|...`, conditional on `os`/`version`/`product` (`if os == "linux": FAIL`), and crucially **expected-FAIL tests still run and report** — failures are documented, not skipped ([expectation metadata docs](https://web-platform-tests.org/tools/wptrunner/docs/expectation.html)). `wpt update-expectations` regenerates the metadata from run logs (`wptreport.json`) — expectation maintenance is automated, review happens in diff.
- Vendors two-way sync the corpus into their trees (e.g. Chromium `web_tests/external/wpt`, [Igalia overview](https://blogs.igalia.com/gyuyoung/2024/04/26/web-platform-test-and-chromium/)); shared CI runs all engines daily and publishes results to wpt.fyi.
- Adjacent prior art, **test262** (ECMAScript): every test file carries YAML frontmatter between `/*---` and `---*/` with `esid` (the **spec heading anchor ID** in the ECMAScript draft), `description`, `features` (feature tags), `flags` (`onlyStrict`, `module`, `async`, `raw`), and `negative` (expected error type/phase) ([tc39/test262 CONTRIBUTING](https://github.com/tc39/test262/blob/main/CONTRIBUTING.md), [INTERPRETING.md](https://github.com/tc39/test262/blob/HEAD/INTERPRETING.md)). `esid` is the cleanest existing example of machine-checkable spec-section citation per test.

### 6. Unicode UCD + CLDR conformance: data-driven testing with executors, schemas, and a verifier taxonomy

- The Unicode Character Database ships algorithm conformance data as plain text fixtures with rigid field grammars — e.g. [NormalizationTest.txt](https://www.unicode.org/Public/17.0.0/ucd/NormalizationTest.txt) (columns c1..c5; the file header *states the invariants* each row must satisfy: `c2 == toNFC(c1) == toNFC(c2) == ...`), and the [UCA CollationTest](https://www.unicode.org/Public/UCA/15.1.0/CollationTest.html). The fixture format embeds the assertion semantics in its preamble — the data file is self-describing ([UAX #44](https://www.unicode.org/reports/tr44/tr44-4.html)).
- [unicode-org/conformance](https://github.com/unicode-org/conformance) ("Data Driven Testing", active 2024–2026) is the most complete modern architecture for cross-implementation conformance:
  - **testgen** generates paired JSON files per component and ICU/CLDR version: `<component>_test.json` (inputs + params, each with a stable `label`) and `<component>_verify.json` (expected results keyed by the same label) under `testData/icu76/...`.
  - **Schema layer**: `schema/<component>/{test_schema.json, verify_schema.json, result_schema.json}` — generated data, expected data, *and executor output* are all JSON-Schema-validated at three checkpoints (after generation, before execution, after runs).
  - **Executors**: each implementation (ICU4C/cpp, ICU4J, ICU4X/rust, node, dart_web) is a small stdin/stdout JSON-protocol shim (`#VERSION`, `#TEST` commands) — identical to toml-test's "thin shim per implementation" idea, scaled to many components (collation, number format, datetime, likely subtags, plural rules, list format, relative time, MessageFormat 2).
  - **Verifier taxonomy**: results classify as Success / Failure / Error / **Unsupported** / **Known issue** — explicitly distinguishing "not implemented" and "documented divergence" from regressions; reports render to per-executor HTML + a master dashboard (`testReports/index.html`).
  - **Multi-version axis**: every test-data set, expected set, and output dir is keyed by ICU/CLDR version; the system regenerates per release ([CLDR test docs](https://cldr.unicode.org/development/running-tests)).

### 7. XLIFF 2.x / TAPICC: l10n's own conformance corpus (small, but the right shape)

- The OASIS XLIFF TC keeps the official test suite in [oasis-tcs/xliff-xliff-22](https://github.com/oasis-tcs/xliff-xliff-22): `xliff-21/test-suite/` and `xliff-22/test-suite/` each split `core/` vs `modules/`, and `valid/` vs `invalid/` (~487 files total). Invalid files are fault-named (`bad_DataIdNotUnique.xlf`, `bad_EcBeforeSc.xlf`, `bad_InvalidDataRef.xlf`) — one constraint violation per file, mapping 1:1 to the spec's normative constraints (verified via repo tree listing, 2026-06).
- XLIFF 2.1 adds a third category beyond valid/invalid: `core/in-out/` **transformation pairs** (`toJoin1_in.xlf`/`toJoin1_out.xlf`, `toSegment1_in.xlf`/`toSegment1_out.xlf`, `toRemoveAnnotations_*`) — expected results of spec-defined *operations* (segment joining, annotation removal), not just parse acceptance. This is the closest existing analog to neokapi's read→transform→write assertions.
- Schema validation alone is insufficient for XLIFF: duplicate ids, broken refs, language codes need Schematron/custom checks; XLIFF 2.1 added Schematron + NVDL, and Okapi **Lynx** is the reference validator ([Raya on XLIFF validation](https://www.linkedin.com/pulse/validation-xliff-20-files-rodolfo-m-raya), [OASIS list](https://lists.oasis-open.org/archives/xliff/201409/msg00014.html)).
- [TAPICC WG3's "XLIFF 2 Extraction and Merging Best Practice"](https://galaglobal.github.io/TAPICC/T1/WG3/XLIFF-EM-BP-ED.xhtml) (GALA, community-driven) is a worked-example catalog for extraction/merge behavior — prose + paired source/XLIFF examples, i.e. the *informal* version of an executable extraction spec. Nobody has mechanized it; that is open ground for neokapi.

### 8. tree-sitter corpus format: minimal text format + snapshot auto-update

- Grammar tests live in `test/corpus/*.txt`: test name between `===` lines, source code, a `---` separator, then the expected parse tree as an **S-expression** ([writing-tests](https://tree-sitter.github.io/tree-sitter/creating-parsers/5-writing-tests.html)).
- Robustness details worth copying: if the language's syntax collides with delimiters, an arbitrary identical suffix disambiguates (`===|||` / `---|||`); the **longest matching `---` line** wins as divider; per-test attributes under the name line: `:skip`, `:error` (expect parse error), `:fail-fast`, `:language(LANG)`, `:platform(...)`, `:cst`.
- `tree-sitter test -u` **rewrites expected outputs from current parser output** — snapshot-update workflow where git diff is the review surface. This is how thousands of community grammars keep corpora maintainable with tiny teams.

### 9. Document-format validators as external oracles (ODF/OOXML)

- [ODF Validator](https://odftoolkit.org/conformance/ODFValidator.html) (ODF Toolkit; 0.13.0 released 2026-01-23 with ODF 1.4 support) validates packages against Relax NG + conformance criteria; LibreOffice wires it (and **office-o-tron** 0.8.8 for OOXML) into round-trip export tests ([LibreOffice dev blog, 2026-01-22](https://dev.blog.documentfoundation.org/2026/01/22/validating-odf-and-ooxml-files/)).
- For OOXML, Microsoft's [Open XML SDK](https://github.com/dotnet/Open-XML-SDK) exposes `OpenXmlValidator` (schema + semantic constraints, per-conformance-class); [openxml-audit](https://pypi.org/project/openxml-audit/) is a pure-Python port claiming "100% parity with Open XML SDK v3.4.1" validation, covering package/schema/semantic checks for DOCX/XLSX/PPTX and staged ODF conformance levels.
- Pattern: when a format has an authoritative external validator, **writer-output validity becomes a cheap machine-checkable assertion** orthogonal to byte-equality round-trips.

### 10. Cross-implementation comparison services

- [Babelmark 3](http://babelmark.github.io/) runs one Markdown fragment through 20+ implementations via a registry of API endpoints and renders outputs side-by-side; built to surface divergence where the spec is ambiguous ([Willison, 2023](https://simonwillison.net/2023/Jan/27/babelmark3/)). The YAML Test Matrix ([matrix.yaml.info](https://matrix.yaml.info/)) does the batch equivalent over a fixed corpus; wpt.fyi does it continuously for browsers. Three tiers of the same idea: ad-hoc probe → corpus matrix → continuous dashboard.

### 11. AI consumption/generation of conformance tests (2024–2026 research)

- **iPanda** ([arXiv 2507.00378](https://arxiv.org/pdf/2507.00378), 2025): LLM agent for protocol conformance testing — generates test cases from spec documents via keyword extraction + RAG over the spec, then iteratively repairs generated test programs against the implementation. Confirms the load-bearing role of *retrieval anchored to spec sections*.
- **DiffSpec** ([arXiv 2410.04249](https://arxiv.org/pdf/2410.04249)): differential testing with LLMs using natural-language specs and code artifacts — generate inputs, compare multiple implementations as mutual oracles (the babelmark idea, automated).
- **Validating Formal Specifications with LLM-Generated Test Cases** ([arXiv 2510.23350](https://arxiv.org/html/2510.23350v1), Springer 2026): frontier models produce positive *and negative* cases from natural-language requirements at >90% oracle conformance with careful prompting — but only with machine validation of the generated cases (syntax checks, schema checks) in the loop.
- Practical synthesis across these: AI test generation is reliable when (a) each generation unit is a *spec section with anchor*, (b) generated cases conform to a *machine-checked case schema*, (c) an *independent oracle* (reference implementation, validator, or N-version differential) adjudicates, and (d) failures are *triaged into a taxonomy* (regression vs unsupported vs documented divergence) rather than pass/fail.

### Cross-cutting structural patterns (the distilled grammar of an executable format spec)

1. **Case shape**: `{stable-id, name, input, expected(s), validity-class, spec-citation, tags, version-applicability}` recurs everywhere. Three validity classes cover all suites: *valid* (input + expected output), *invalid* (input + expected rejection, fault-named, one fault per case), *in-out / operation* (input + operation + expected output — XLIFF `in-out`, CommonMark md→html, tree-sitter src→tree).
2. **Multiple expected views per input** (yaml-test-suite events/json/canonical-dump; Unicode DDT test/verify pairs) decouple parse semantics from data model from emission — each view is a separate assertion plane that implementations can adopt incrementally.
3. **Neutral expected-encoding**: never express expected output in the format under test (toml-test's JSON tagged values; YAML's event DSL) — otherwise one parser gets "blessed" and the suite can't outlive implementations.
4. **Spec citation as machine-checkable anchor**: test262 `esid`, WPT `<link rel=help>` + heading-id directories, CommonMark `section`+`number`. Lint that anchors resolve against the spec document.
5. **Versioning**: per-spec-version directories (JSON Schema), per-version file manifests over one corpus (toml-test `files-toml-1.0.0`), version tags on cases (yaml `1-3-*`), immutable dated data releases (yaml `data-YYYY-MM-DD`), frozen old drafts (JSON Schema). Stable short IDs make results comparable across suite revisions.
6. **Shared corpus, local expectations**: implementation-specific pass/fail status lives *outside* the suite (WPT `.ini` metadata trees, DDT "Known issue"/"Unsupported" classifications), is auto-regenerated from run logs, and expected-fail tests still execute and report.
7. **The suite validates itself**: meta-schema + CI check (JSON Schema `test-schema.json`, Unicode `schema/` triple-checkpoint).
8. **Snapshot update with diff review** (`tree-sitter test -u`, `wpt update-expectations`) is what makes large corpora maintainable solo.
9. **Thin executor shims** (toml-test stdin/stdout, DDT `#TEST` protocol) make the corpus runnable against anything — including a WASM build or a Java bridge — for pennies.

## Design implications for neokapi

- **Converge spec.yaml on the recurring case grammar.** Each case: stable 4–6 char ID (yaml-test-suite style, never reused), `name`, `input` (inline or file ref), `class: valid|invalid|operation`, one or more `expected` views, `spec:` citation with resolvable anchor, `tags`, `since`/`until` format-version bounds. Keep "one fault per invalid case, named after the fault" (toml-test, XLIFF `bad_*`).
- **Adopt the XLIFF triad explicitly**: `valid/` (parse + extraction assertions), `invalid/` (must reject, with error-category assertion), `in-out/` (operation pairs: extract→blocks, merge(blocks′)→native, redact, segment). The in-out pair is neokapi's natural unit — it directly encodes the faithful round-trip moat.
- **Define a neutral expected-encoding for the block stream** — a canonical JSON event/Part-stream form (analog of yaml-test-suite `test.event` + toml-test tagged values), emitted deterministically by `kapi`. This (a) prevents the Go implementation from being the blessed oracle, (b) lets the okapi-bridge, the WASM lab, and future ports run the same corpus via a thin stdin/stdout shim (DDT executor protocol), (c) gives AI a schema-checkable generation target.
- **Multi-view expected outputs per example**: `expected.blocks` (event stream), `expected.roundtrip` (byte-exact or normalized native output), `expected.extracted` (text-only view), optional `expected.valid-by: <external-validator>`. Implementations/maturity levels adopt views incrementally — maps cleanly onto the L0–L4 rubric (L1 = blocks view passes, L3 = roundtrip view, L4 = all views + invalid corpus + external validator).
- **Machine-checkable spec citations**: require `spec: {url, anchor, quote?}` per case; add a lint (CI) that fetches/caches the spec and verifies anchors resolve (WPT CSS-lint model, test262 `esid`). The anchor is then the RAG unit for AI generation: "generate N candidate cases for section X; every case must cite X."
- **Keep parity/xfail expectations out of the corpus.** Mirror WPT: a per-implementation `expectations/` tree (`<format>/<case-id>: FAIL (reason, issue#)`) that the runner consumes; expected-fail cases still run and report; provide `kapi spec-test --update-expectations` regenerating it from run logs. This replaces ad-hoc xfail lists and feeds the /format-maturity and parity dashboards from one source.
- **Adopt the DDT verifier taxonomy** for results: Success / Failure / Error / Unsupported / Known-divergence (documented, with spec-citation for why neokapi intentionally differs — matches the existing "faithful%" framing and the parity-divergence docs).
- **Meta-schema the spec.yaml format itself** and gate in CI (JSON Schema Test Suite `test-schema.json` pattern; Unicode's three checkpoints: after generation, before run, after run). This is the precondition for trusting AI-generated cases: schema-validate → run against reference oracle → only then human review.
- **Snapshot-update workflow**: `kapi spec-test -u <format>` rewrites `expected.*` views from current engine output, with git diff as the review surface (tree-sitter `-u`). Solo-maintainer leverage: corpus maintenance cost ≈ reading diffs.
- **Version the corpus, not just the code**: per-format-version manifests (`files-xliff-2.1`, `files-xliff-2.2` listing case IDs, toml-test style) rather than duplicating cases; immutable dated corpus releases if/when the corpus is shared outside the repo (yaml `data-YYYY-MM-DD`).
- **Wire external validators as oracles where they exist**: ODF Validator 0.13 (ODF 1.4), Open XML SDK validation (or openxml-audit) for docx/xlsx/pptx writer output, Okapi Lynx/Schematron for XLIFF output, `python3 spec_tests.py --program` for the markdown writer. Cheap L3/L4 assertions: "everything we write is independently valid."
- **AI generation/porting loop (grounded in iPanda/DiffSpec findings)**: per spec section anchor → retrieve section text → generate candidate cases (positive *and* negative) → schema-validate → execute against neokapi *and* okapi-bridge (differential oracle) → auto-classify (both agree = candidate-pass; disagree = divergence triage; both reject = candidate-invalid case) → human reviews only the diffs. Track per-section case counts to surface spec-coverage gaps (CommonMark `-s` per-section stats; WPT heading-id directories).
- **Port harvesting**: CommonMark `--dump-tests` JSON, toml-test, yaml-test-suite data releases, JSON Schema suite, XLIFF `test-suite/`, and UCD fixture files are all directly consumable today — neokapi's markdown/toml/yaml/json-schema-adjacent/xliff formats should import these corpora wholesale (with provenance + version recorded per case) instead of authoring parallel examples; spec.yaml needs an `origin:` field (suite, version, upstream id) to keep re-imports idempotent.
- **Longer term: a babelmark/matrix for l10n formats** — same input through neokapi, okapi-bridge (×11 Okapi versions already built), and other tools, rendered as a divergence matrix. The existing parity dashboard is halfway there; the missing piece is making the corpus (not the Go tests) the unit the matrix runs over.
