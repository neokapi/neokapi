# Format-watch & emerging-format trends (research, June 2026)

Stream: (a) how the specs behind neokapi's existing formats evolve and how to watch them mechanically; (b) where new content formats are emerging that a localization/format platform must track. All dates verified against primary sources as of 2026-06-11.

## Findings

### A1. Document-suite standards: OOXML is frozen-at-ISO, alive-at-Microsoft; ODF just shipped 1.4

- **OOXML (ECMA-376 / ISO/IEC 29500).** The latest ECMA edition is the **5th edition (Dec 2021), and only Part 2 (Open Packaging Conventions) was updated in it**; Part 1 "Fundamentals and Markup Language Reference" is still the **December 2016** text ([ecma-international.org/…/ecma-376](https://ecma-international.org/publications-and-standards/standards/ecma-376/)). The Library of Congress format registry confirms the ECMA/ISO pair is "revised in parallel with Office" and last meaningfully revised 2016 ([loc.gov FDD 395](https://www.loc.gov/preservation/digital/formats/fdd/fdd000395.shtml)). Practical consequence: **the standard barely moves; real-world `.docx` evolves through Microsoft's extension documents** ([MS-DOCX], [MS-XLSX], [MS-PPTX] on [learn.microsoft.com/openspecs](https://learn.microsoft.com/en-us/openspecs/office_standards/ms-docx/)) and observed Office output. A parity/maturity process for openxml should diff against Microsoft's open-specs changelogs and real Office fixtures, not wait for ISO editions.
- **ODF.** **ODF 1.4 was approved as a full OASIS Standard on 3 Dec 2025** (TDF announcement: [blog.documentfoundation.org](https://blog.documentfoundation.org/blog/2025/12/03/tdf-announces-odf-v14-as-oasis-standard/); spec: [docs.oasis-open.org/office/OpenDocument/v1.4](https://docs.oasis-open.org/office/OpenDocument/v1.4/OpenDocument-v1.4-part2-packages.html)). ISO is still at ODF 1.2 (ISO/IEC 26300:2015) ([loc.gov FDD 247](https://www.loc.gov/preservation/digital/formats/fdd/fdd000247.shtml)). LibreOffice is the reference producer. ODF moves roughly once a decade; 1.4 is the event of this decade.

### A2. The Unicode release train is the most predictable clock in i18n

- **Cadence is now firmly annual-major + quarterly-dot**: CLDR 48 shipped **2025-10-29** updated for **Unicode 17** ([blog.unicode.org](http://blog.unicode.org/2025/10/unicode-cldr-48-available.html), [release note](https://cldr.unicode.org/downloads/cldr-48)); CLDR 48.1 on **2026-01-08**; **ICU 78.3 + CLDR 48.2 on 2026-03-17** ([blog.unicode.org](http://blog.unicode.org/2026/03/unicode-icu-783-and-cldr-482-released.html)). **CLDR 49 / ICU 79 (with Unicode 18) are planned for October 2026**; CLDR 49 survey-tool submission opened 2026-04-29 ([cldr.unicode.org](https://cldr.unicode.org/), [ICU 78 notes](https://unicode-org.github.io/icu/download/78.html)). Anything in neokapi keyed to ICU (cgo-ICU segmentation, locale data, collation) should expect a **September–October disruption window every year**.
- **MessageFormat 2 (MF2) is now a stable Unicode standard, but adoption hasn't started.** Final Candidate was announced Jan 2025 and approved ~March 2025 ([Unicode blog](http://blog.unicode.org/2025/01/messageformat-20-final-candidate-review.html)); MF2 is a **stable part of LDML as of LDML 47, refined in LDML 48 (Oct 2025)**, recommended for implementation by the CLDR TC ([messageformat.unicode.org](https://messageformat.unicode.org/), [JS reference impl releases](https://github.com/messageformat/messageformat/releases)). It is built into ICU4C/ICU4J and "soon" ECMA-402 ([ICU MF2 guide](https://unicode-org.github.io/icu/userguide/format_parse/messages/mf2.html)). But as of **May 2026 no major i18n framework has switched defaults** — FormatJS/react-intl and Lingui are still MF1 ([locize state-of-migration, May 2026](https://www.locize.com/blog/messageformat-2-i18next/)). Spec home: [github.com/unicode-org/message-format-wg](https://github.com/unicode-org/message-format-wg). **There is no native Go MF2 implementation as of 2025–2026** — existing Go libraries (gotnospirit/messageformat, modern-go/msgfmt, kaptinlin/go-i18n) are all MF1 ([pkg.go.dev search evidence](https://pkg.go.dev/github.com/gotnospirit/messageformat)). That's an open lane for a Go framework.

### A3. Localization-interchange standards: XLIFF 2.2 landed; Fluent is in maintenance

- **XLIFF 2.2 became an OASIS Committee Specification on 13 March 2025**, split into **Part 1: Core** and **Part 2: Extended**, and adds a **Plural/Gender/Select module**; the TC's stated future direction includes **XLIFF as a TM format** ([Core CS](https://docs.oasis-open.org/xliff/xliff-core/v2.2/xliff-core-v2.2-part1.html), [Extended](https://docs.oasis-open.org/xliff/xliff-core/v2.2/xliff-extended-v2.2-part2.html), [OASIS webinar announcement, Sep 2025](https://x.com/OASISopen/status/1966493837011853750), [CSD02 review notice, Dec 2024](https://www.oasis-open.org/2024/12/20/xliff-v2-2-csd02-is-now-available-for-public-review/)). XLIFF 2.1 (2018) remains the last full OASIS *Standard*. A platform whose xliff2 writer targets 2.0/2.1 should plan a 2.2 delta (new module, two-part spec).
- **Fluent (Mozilla, `.ftl`) is effectively in maintenance.** Mozilla's institutional energy has shifted to **ICU4X and Unicode MF2** (community discussion: [Mozilla Discourse "Institutional Support (ICU4X / Unicode MessageFormat 2.0)"](https://discourse.mozilla.org/t/institutional-support-icu4x-unicode-messageformat-2-0/142297)); Firefox is actively migrating intl internals from ICU4C to ICU4X ([meta bug 1713916](https://bugzilla.mozilla.org/show_bug.cgi?id=1713916), [Firefox source docs](https://firefox-source-docs.mozilla.org/intl/icu4x.html)), and **ICU4X 2.0 shipped May 2025** ([Unicode blog](http://blog.unicode.org/2025/05/icu4x-20-released.html)). Fluent files remain ubiquitous in Firefox + many OSS apps ([projectfluent.org](https://projectfluent.org/), [Firefox Fluent docs](https://firefox-source-docs.mozilla.org/l10n/fluent/tutorial.html)) — a stable, low-churn format target with a large installed base and no expected syntax evolution.

### A4. Timed text: WebVTT/IMSC are steady; the action is DAPT (dubbing/AD scripts)

- The W3C **Timed Text WG was rechartered in 2025** ([charter](https://www.w3.org/2025/06/timed-text-wg-charter.html)); **WebVTT remains a Candidate Recommendation** ([w3.org/TR/webvtt1](https://www.w3.org/TR/webvtt1/)); IMSC profiles are stable maintenance items ([TTML profiles overview](https://www.w3.org/AudioVideo/TT/docs/TTML-Profiles.html)).
- **DAPT — Dubbing and Audio description Profiles of TTML2 — is the active spec**: updated **Candidate Recommendation snapshots in July and 18 Dec 2025**, Rec expected no earlier than 15 Jan 2026 ([w3.org/TR/dapt](https://www.w3.org/TR/dapt/), [W3C news, implementations invited](https://www.w3.org/news/2025/w3c-invites-implementations-of-dubbing-and-audio-description-profiles-of-ttml2/), [WG minutes 2025-12-18](https://www.w3.org/2025/12/18-tt-minutes.html)).
- **Netflix has published a TTAL→DAPT converter** ([github.com/Netflix/ttal-dapt-conv](https://github.com/Netflix/ttal-dapt-conv)), signaling the industry's authoring-lineage format (TTAL, Netflix's JSON script-exchange spec — [Netflix partner spec](https://partnerhelp.netflixstudios.com/hc/en-us/articles/360002110287-Dubbing-and-Audio-Description-Archival-Asset-Specifications)) converging on DAPT for dubbing/AD script exchange. This is squarely a localization-workflow format family.

### A5. Markdown-family: CommonMark is dormant; the energy moved to profiles built on it

- **CommonMark's latest spec is 0.31.2 (28 Jan 2024)** — no release in ~2.5 years ([spec.commonmark.org](https://spec.commonmark.org/), [changelog.txt](https://spec.commonmark.org/changelog.txt), [repo](https://github.com/commonmark/commonmark-spec)). MDX is similarly stable at v3 (no MDX 4 signal in 2025–2026 searches). The de-facto evolution is in *profiles*: GFM, **Markdoc** (Stripe, [markdoc.dev](https://markdoc.dev/docs/faq)), MDX, and the new **AI-context markdown conventions** (below). For a format engine, markdown risk is profile/dialect drift, not core-spec churn.
- **AsciiDoc**: the Eclipse **AsciiDoc Language Specification + TCK project is still "incubating"** under the AsciiDoc WG ([projects.eclipse.org](https://projects.eclipse.org/projects/asciidoc.asciidoc-lang), [asciidoc-wg.eclipse.org](https://asciidoc-wg.eclipse.org/projects/asciidoc-lang/)) — a formal spec with a compatibility kit is coming but slow; Asciidoctor remains the de-facto definition. The TCK is interesting precedent for neokapi's own spec.yaml/assertion approach.

### A6. Design tokens: DTCG hit "first stable version" — a dated, versioned release train begins

- The **Design Tokens Specification reached its first stable version, "2025.10", on 28 Oct 2025** ([DTCG announcement](https://www.w3.org/community/design-tokens/2025/10/28/design-tokens-specification-reaches-first-stable-version/), [designtokens.org](https://www.designtokens.org/), [repo](https://github.com/design-tokens/community-group)). Stable adds **theming/multi-brand resolver concepts, CSS Color 4 spaces (Display-P3, Oklch), and richer alias/inheritance semantics**, with reference implementations in **Style Dictionary, Tokens Studio, Terrazzo** and support in Figma, Penpot, Sketch, Framer, Supernova, zeroheight ([zeroheight what's-new](https://zeroheight.com/blog/whats-new-in-the-design-tokens-spec/)). neokapi's existing `designtokens` format almost certainly predates 2025.10 and needs a conformance delta (color object syntax, `$extensions`, resolver/theming files). The version string "2025.10" implies calendar-versioned future releases — a watchable train.

### A7. JSON Schema: stability regime announced, still not shipped — 2020-12 remains the deployed dialect

- JSON Schema is moving off the IETF-draft treadmill to a **self-published "stable" spec where keywords never break again**; the one declared "last breaking change" is forbidding undeclared keywords ([the-last-breaking-change, Feb 2023](https://json-schema.org/blog/posts/the-last-breaking-change), [stable-json-schema](https://json-schema.org/blog/posts/stable-json-schema), [specification page](https://json-schema.org/specification)). As of mid-2026 the stable release has still not shipped; **draft 2020-12 is the de-facto dialect**, including as MCP's default ([MCP SEP-1613](https://github.com/modelcontextprotocol/modelcontextprotocol/issues/1613)). Watch [json-schema.org/specification-links](https://json-schema.org/specification-links); low urgency.

### B1. AI-context formats are the fastest-emerging markdown profiles (2024–2026)

- **llms.txt**: ~**844,000 sites by Oct 2025** (BuiltWith, via [allmo.ai report](https://allmo.ai/articles/llms-txt)); a SE Ranking 300k-domain study found **~10.1% adoption** ([seranking.com](https://seranking.com/blog/llms-txt/)). Crucially, **no major chat-search platform officially reads it, but coding/agent tools (Cursor, Claude Code, Copilot, Aider, Cline) do fetch `/llms.txt` and `/llms-full.txt`** ([presenc.ai state-of-llms-txt 2026](https://presenc.ai/research/state-of-llms-txt-2026), spec: [llmstxt.org](https://llmstxt.org/)). It's a markdown-with-conventions format; multilingual sites will eventually want localized llms.txt variants.
- **AGENTS.md**: introduced mid-2025, **formalized Aug 2025 by OpenAI, Google, Cursor, Factory, Sourcegraph**; **>60k repositories by late 2025/early 2026**; GitHub Copilot added native support Aug 2025; now stewarded under the Linux Foundation's **Agentic AI Foundation** (Dec 2025) ([InfoQ](https://www.infoq.com/news/2025/08/agents-md/), [OpenAI announcement](https://openai.com/index/agentic-ai-foundation/), site: [agents.md](https://agents.md/)). Pattern to note: **the AI ecosystem mints markdown-profile "formats" at high speed and they standardize via foundations within months** — a watch source in its own right.

### B2. Notebooks: the format story is bifurcating

- **Jupyter `.ipynb` (nbformat, JSON)** is stable, but **marimo stores notebooks as pure Python**, explicitly because JSON notebooks are hostile to git and AI agents ([marimo repo](https://github.com/marimo-team/marimo), [vs-Jupyter](https://marimo.io/features/vs-jupyter-alternative), [Jupytext interop](https://docs.marimo.io/guides/coming_from/jupytext/)). Quarto's `.qmd` (markdown + executable cells, [quarto.org](https://quarto.org/)) is the publishing-side analog. Localizable payload = markdown cells + output alt-text; `.ipynb` is tractable today via a configured JSON reader, `.qmd`/marimo via markdown/comment-aware readers.

### B3. Documents/publishing: EPUB 3.4 in flight; Typst is the genuinely new document language

- **EPUB 3.4**: revision started **11 Feb 2025** under the W3C Publishing Maintenance WG (chartered to Feb 2027); FPWDs for EPUB 3.4, Reading Systems 3.4, Accessibility 1.1.1 are out; planned features include **dark-mode support, standardized footnotes/endnotes, webtoon "roll" fixed layout, annotation interchange** ([W3C blog](https://www.w3.org/blog/2025/epub3-3-recommendations-published-work-begins-on-new-features/), [charter](https://www.w3.org/2025/02/pmwg-charter.html), [workspace](https://w3c.github.io/epub-specs/), [EDRLab summary](https://www.edrlab.org/2025/04/08/w3c-pushes-epub-forward/)). EPUB is "zip of XHTML+OPF" — close to what neokapi's html + skeleton machinery already does.
- **Typst**: **second fastest-growing language on GitHub in 2025 by percentage** ([Wikipedia, citing GitHub Octoverse](https://en.wikipedia.org/wiki/Typst), [typst/typst](https://github.com/typst/typst)); production adoption beyond academia exists (Zerodha generates transaction documents with it), though only a couple of journals accept it as of Apr 2026 ([underleaf.ai comparison](https://www.underleaf.ai/blog/typst-vs-latex)). `.typ` is a clean, parseable markup — far more tractable than LaTeX for a faithful reader.

### B4. Structured-content CMS rich text: JSON ASTs are the de-facto "new HTML"

- **Sanity Portable Text** is an open JSON block/span specification with serializers to any target ([sanity.io docs](https://www.sanity.io/docs/presenting-block-text), ecosystem org: [github.com/portabletext](https://github.com/portabletext/contentful-rich-text-to-portable-text)); **Contentful Rich Text** is the comparable JSON AST; converters between them exist. Localization is field-level (Contentful) or document-level (Sanity) ([webstacks comparison](https://www.webstacks.com/blog/contentful-vs-sanity)). These ASTs map naturally onto neokapi's Block/Run model (spans = runs, marks = Pc/Ph) — better served by dedicated readers than by generic JSON config, since inline marks must become runs, not text.

### B5. 3D & spatial content formalized in 2025

- **AOUSD published OpenUSD Core Specification 1.0 on 17 Dec 2025**, the explicit first step toward ISO standardization; **Core Spec 1.1 (animation, massive scenes, compliance testing) is on the 2026 roadmap** ([aousd.org](https://aousd.org/news/core-spec-announcement/), [Linux Foundation press](https://www.linuxfoundation.org/press/alliance-for-openusd-announces-core-specification-1.0-the-universal-language-for-building-3d-worlds)). **glTF 2.0 remains ISO/IEC 12113:2022**, refreshed via the PAS process ([iso.org](https://www.iso.org/standard/83990.html), [Khronos](https://www.khronos.org/news/press/khronos-gltf-2.0-released-as-an-iso-iec-international-standard)). Localizable strings in 3D scenes (labels, configurators, training sims) are a long-tail but now-standardized target.

### B6. Design tooling & e-learning

- **Figma**: `.fig` stays proprietary, but the **REST API exposes the full file as JSON** and is itself specified in an open OpenAPI repo ([developers.figma.com](https://developers.figma.com/docs/rest-api/), [figma/rest-api-spec](https://github.com/figma/rest-api-spec)); Figma shipped an **MCP server in June 2025** and agent write-back workflows ([changelog](https://developers.figma.com/docs/rest-api/changelog/)). For l10n, Figma is a **connector** (API JSON) not a file format; the community `figma-to-json` tooling ([yagudaev/figma-to-json](https://github.com/yagudaev/figma-to-json)) shows the .fig container is reverse-engineerable but unstable.
- **E-learning**: **cmi5 (an xAPI profile) is the designated SCORM successor and is on an IEEE standards track**; xAPI/cmi5 packages still wrap HTML/JS content ([easygenerator overview](https://www.easygenerator.com/en/blog/results-tracking/cmi5-what-it-is-and-why-you-need-it/), [anyforsoft comparison](https://anyforsoft.com/blog/xapi-vs-scorm/)). For localization the unit of work is the package manifest + embedded HTML — connector + existing html format, not a new parser.

### B7. How to watch mechanically (mechanisms that exist today)

- **GitHub Atom feeds**: every repo exposes `…/releases.atom`, `…/tags.atom`, `…/commits/<branch>.atom` — covers MF2 WG, ICU, CommonMark, MDX, DTCG, Typst, glTF, marimo, Netflix/ttal-dapt-conv with zero auth.
- **W3C**: news feed ([w3.org/blog/news/feed](https://www.w3.org/news/2025/updated-candidate-recommendation-dubbing-and-audio-description-profiles-of-ttml2/) entries demonstrate per-spec announcements), per-group publication pages ([Timed Text publications](https://www.w3.org/groups/wg/timed-text/publications/)), and spec GitHub repos (w3c/dapt, w3c/epub-specs).
- **Unicode**: [blog.unicode.org](http://blog.unicode.org/) (Blogger Atom feed at `/feeds/posts/default`) is the single announcement channel for Unicode/CLDR/ICU/ICU4X/MF2; release pages ([cldr.unicode.org/downloads](https://cldr.unicode.org/)) carry the forward schedule.
- **OASIS**: announcements page ([oasis-open.org/news](https://www.oasis-open.org/2024/12/20/xliff-v2-2-csd02-is-now-available-for-public-review/)) + `docs.oasis-open.org` directory listings per TC (xliff, office).
- **Microsoft Open Specifications**: each [MS-*] doc carries a revision history/changelog page — the real OOXML watch target.
- **Library of Congress format registry (FDDs)** tracks format-family status with dated updates ([loc.gov/preservation/digital/formats](https://www.loc.gov/preservation/digital/formats/fdd/fdd000395.shtml)) — useful slow-moving cross-check.
- **TMS vendor format lists as a market signal**: Crowdin/Phrase/Lokalise "supported formats" pages enumerate what paying customers demand (e.g. Crowdin lists `.ftl`, `.xcstrings`, Typst is absent) — a quarterly scrape catches demand before specs do.

## Design implications for neokapi

1. **Pin spec versions in each format's `spec.yaml` and diff against upstream.** Add `spec: {id, version, url, watch: [feed-urls]}` to the executable spec files. A periodic skill (cron/`/loop`-style, like the existing `format-triage`) fetches each watch feed, compares against the pinned version, and opens a GitHub issue per drift ("DTCG 2025.10 vs pinned 2024-draft"). This converts the maturity rubric into a *living* contract: L4 should arguably require "spec watch wired + pinned version current".
2. **Align the maintenance calendar to the Unicode train.** Schedule one heavy "i18n-data upgrade" window each October–November (CLDR/ICU majors, Unicode major) and lightweight dot-release checks quarterly. neokapi's cgo-ICU segmentation and locale code are directly exposed (ICU 79/CLDR 49/Unicode 18 land Oct 2026).
3. **Ship a native Go MF2 implementation.** Spec is stable (LDML 47/48), ICU4C/4J have it, JS reference exists — **Go has nothing**. For a Go localization framework this is a high-leverage, low-competition differentiator, and it future-proofs the message-catalog formats (JSON/ARB/properties) for MF2 syntax. Track [message-format-wg](https://github.com/unicode-org/message-format-wg) releases.
4. **Plan the XLIFF 2.2 delta now.** Core+Extended split and the Plural/Gender/Select module map directly onto neokapi's Plural/Select runs — neokapi can be an early faithful implementation. Also watch the TC's "XLIFF as TM" direction; it intersects sievepen.
5. **Re-validate `designtokens` against DTCG 2025.10** (color object syntax, Oklch/Display-P3, aliases/theming/resolver). Calendar-versioned releases mean this becomes a yearly conformance check, not a one-off.
6. **Treat OOXML maintenance as "watch Microsoft, not ISO".** Add the [MS-DOCX]/[MS-XLSX]/[MS-PPTX] revision pages to the watchlist; refresh Office-generated fixtures yearly. ECMA/ISO editions are near-static.
7. **Top new-format candidates (ranked, with adoption evidence):**
   1. **Fluent `.ftl`** — huge installed base (all of Firefox + OSS ecosystem), frozen syntax = cheap to keep faithful forever; every major TMS supports it. (Maintenance-mode upstream is a *feature* for parity.)
   2. **DAPT (TTML2 dubbing/AD profile) + Netflix TTAL** — W3C CR→Rec in 2026, Netflix-published converter; extends neokapi's subtitle family into dubbing-script exchange, a pure-l10n workflow.
   3. **Portable Text (Sanity) + Contentful Rich Text JSON** — structured-content rich text ASTs whose span/mark model maps 1:1 to Block/Run; generic JSON config can't express inline marks as runs, so these justify bespoke readers despite the "prefer configured readers" rule.
   4. **AsciiDoc `.adoc`** — Eclipse spec+TCK incoming; large docs ecosystems (Antora, Red Hat); fills the markdown-family gap.
   5. **Typst `.typ`** — fastest-growing doc language (GitHub 2025); clean grammar makes a faithful reader feasible where LaTeX never was; zero TMS support today = first-mover.
   6. **EPUB container** — 3.4 revision underway; mostly orchestration over the existing html format + OPF/NCX metadata; high publishing demand.
   7. **ODF (`.odt/.ods/.odp`)** — ODF 1.4 just ratified; okapi-bridge covers it today, but a native reader extends the "faithful format engine" moat to the second office suite.
   8. **Jupyter `.ipynb` / Quarto `.qmd` / marimo `.py`** — docs-as-notebooks; `.ipynb` may be achievable as a configured JSON reader + embedded-markdown child layers (plays to existing strengths).
   9. **llms.txt / AGENTS.md profiles** — trivially parseable as markdown; value is *detection + conventions* (localized llms.txt variants), and it's strong marketing fit for the AI-native positioning.
   10. **MF2 message syntax inside catalogs** — not a file format but a syntax dimension across json/arb/properties; pairs with implication 3.
   Long-tail/connector-tier: Figma (REST JSON connector, not .fig), cmi5/SCORM packages (connector + html), USD/glTF strings (watch only).
8. **Add a "format-watch" source tier for *emergence*, not just spec drift**: GitHub Octoverse fastest-growing languages (caught Typst), Linux-Foundation/foundation announcements (caught AGENTS.md→Agentic AI Foundation, AOUSD), TMS supported-format pages (demand signal), and BuiltWith-style adoption trackers (caught llms.txt). Quarterly scan; each candidate gets scored against an adoption-evidence bar before any implementation issue is opened.
