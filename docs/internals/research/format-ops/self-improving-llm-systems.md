# Self-improving LLM systems & prompt-optimization processes — state of the art (June 2026)

Research stream for neokapi's format-maintenance process: which patterns from DSPy/GEPA, Reflexion/Self-Refine, LLM-as-judge practice, eval-first development, agent-harness engineering, living-runbook systems, and prompt-regression/model-migration practice transfer to a maintainer runbook whose rubrics and prompts must improve over time and survive model upgrades. All claims dated where possible; primary sources preferred.

## Findings

### 1. Programmatic prompt optimization: prompts as compiled, re-compilable artifacts (DSPy, MIPROv2, GEPA)

- **DSPy's core move** is to separate *what* the LLM should do (signatures), *how* it reasons (modules), and *how quality is measured* (metrics), then treat the prompt text itself as an optimizer output rather than hand-maintained source ([DSPy guide overview](https://myengineeringpath.dev/tools/dspy-guide/), [Advancing Analytics intro](https://www.advancinganalytics.co.uk/blog/prompt-optimisation-an-introduction-to-dspy)). The practical consequence: when the model changes, you **re-compile** against the same metric + trainset instead of hand-editing prompts.
- **MIPROv2** jointly optimizes instructions and few-shot demonstrations via bootstrapped teacher traces, instruction proposal, and Bayesian optimization ([The Data Quarry, DSPy optimizers walkthrough](https://thedataquarry.com/blog/learning-dspy-3-working-with-optimizers/)).
- **GEPA (Genetic-Pareto)** — Agrawal et al., arXiv July 2025, revised Feb 2026, ICLR 2026 oral ([arXiv:2507.19457](https://arxiv.org/abs/2507.19457)) — is the strongest current result for *reflective* prompt evolution: it samples execution trajectories, **reflects on them in natural language** to diagnose problems and propose prompt edits, and maintains a **Pareto frontier of candidate prompts**, combining complementary lessons from different candidates. Results: outperforms the GRPO RL baseline by ~6% avg (up to 20%) with **up to 35× fewer rollouts**, and beats MIPROv2 by >10%. The paper's stated reason it works: natural language is "a much richer learning medium" than sparse scalar rewards — i.e., *textual critique is the gradient*.
- **DSPy's GEPA integration** ([dspy.ai GEPA optimization guide](https://dspy.ai/getting-started/gepa-optimization/)) makes the key API decision explicit: the **metric returns a score *plus* feedback text** (e.g. "Don't reference the input season verbatim"), and a separate, typically larger `reflection_lm` rewrites instructions from that feedback. Budgets are coarse (`auto="light"` ≈ 6 candidate prompts, medium/heavy above that). Documented example: 78.1% → 90.1% on the task metric, beating an unoptimized frontier model at lower cost.
- Transferable mechanism, independent of adopting DSPy itself: **(metric with textual feedback) + (execution traces) + (a reflection step that proposes rubric/prompt edits) + (a Pareto archive of candidates scored on a fixed eval set)**. This is exactly the shape of a maintainer loop: failure annotations on format spec assertions are the feedback text; the rubric/skill prompt is the candidate being evolved; the spec.yaml suite is the metric.

### 2. Inference-time self-improvement: Self-Refine and Reflexion — and their hard limits

- **Self-Refine** (Madaan et al., NeurIPS 2023; [selfrefine.info](https://selfrefine.info/), [ACM DL](https://dl.acm.org/doi/10.5555/3666122.3668141)): generate → self-feedback → refine loop with the *same* model, no training; ~20% absolute average improvement across tasks vs one-shot generation.
- **Reflexion** (Shinn et al., 2023): "verbal reinforcement learning" — after a failed attempt the agent writes a self-assessment of what went wrong, stores it in an **episodic memory buffer**, and prepends it to subsequent attempts ([Reflective LLM-based Agent overview, EmergentMind](https://www.emergentmind.com/topics/reflective-llm-based-agent)).
- **Limits** (consistent across the survey literature, e.g. [MAR: Multi-Agent Reflexion, 2025](https://arxiv.org/html/2512.20845), [Agentic RL survey](https://arxiv.org/pdf/2509.02547)): single-agent self-critique inherits the model's own fixed reasoning patterns and blind spots; reflection works best when grounded in **external feedback signals** (test results, execution errors, environment state) rather than pure self-judgment. Reflexion's wins came on tasks with checkable outcomes. Implication: self-improvement loops need an *oracle outside the model* — for neokapi that oracle already exists (round-trip byte-equality, Okapi parity harness, spec.yaml assertions), which is the strongest possible grounding.

### 3. LLM-as-judge: validated practice and catalogued failure modes

- **Hamel Husain's "Critique Shadowing" process** ([hamel.dev LLM-judge guide](https://hamel.dev/blog/posts/llm-judge/)) is the most-cited practitioner method: (1) identify a single **principal domain expert** whose judgment defines success; (2) build a diverse dataset (features × scenarios × personas); (3) expert grades **binary pass/fail with written critiques** (30+ examples, continue until no new failure modes emerge); (4) fix discovered product errors *before* building the judge; (5) build the judge iteratively against the expert until agreement is acceptable (Honeycomb case: **>90% agreement in three iterations**); (6) error-analyze; (7) only then split into specialized judges. Explicit warnings: 1–5 Likert scales are "not actionable" — "if your evaluations consist of a bunch of metrics that LLMs score on a 1-5 scale… you're doing it wrong"; off-the-shelf generic judges cause confusion; terse critiques starve the judge prompt of few-shot material. Re-validate "at regular intervals and whenever something material changes (model updates, significant feature additions)."
- **Judge bias taxonomy** is now well measured: position bias (swapping pairwise order shifts accuracy >10% in code judging — [Brenndoerfer, position-bias measurement](https://mbrenndoerfer.com/writing/position-bias-in-llm-judges)), verbosity bias (embedded in learned representations, harder to prompt away), self-preference/self-enhancement bias ([arXiv:2410.21819](https://arxiv.org/pdf/2410.21819)), plus authority/bandwagon/salience and ~a dozen more catalogued in "Justice or Prejudice?" ([arXiv:2410.02736](https://arxiv.org/pdf/2410.02736)). Frontier models still fail >50% of structured bias tests ([Adaline, judge reliability](https://www.adaline.ai/blog/llm-as-a-judge-reliability-bias)). Mitigations: order randomization, multi-permutation evaluation, rubric shuffling, conciseness terms in the rubric, ensembles.
- **Calibration discipline**: a judge validated on one domain (chat) is not reliable on another (code review, agent tasks) without domain-specific calibration; practitioner guidance is to compare judge verdicts against human spot checks and **recalibrate when divergence exceeds ~20–25%** ([Deepchecks, judge calibration](https://deepchecks.com/llm-judge-calibration-automated-issues/)); maintain a human-labeled golden set and continuously re-anchor ([W&B, exploring LLM-as-a-judge](https://wandb.ai/site/articles/exploring-llm-as-a-judge/)).
- **"Who Validates the Validators?" / EvalGen** (Shankar et al., UIST 2024, [arXiv:2404.12272](https://arxiv.org/abs/2404.12272)) supplies the foundational finding for rubric maintenance: **criteria drift** — graders cannot fully specify evaluation criteria up front, because *the act of grading outputs is what surfaces and refines the criteria*. EvalGen's mixed-initiative loop (LLM proposes candidate assertions/judge prompts; human grades a sample; system selects implementations aligned with the human grades) is the template: rubrics must be treated as living artifacts co-evolved with grading sessions, not fixed specs ([Arize breakdown](https://arize.com/blog/breaking-down-evalgen-who-validates-the-validators/)).
- **Eugene Yan's complement** ([eval-process essay](https://eugeneyan.com/writing/eval-process/), [LLM-evaluators survey](https://eugeneyan.com/writing/llm-evaluators/)): an LLM judge is a tool, not a process — what works is the scientific method as a loop: observe (look at the data) → annotate (≈50:50 pass/fail across the input distribution) → hypothesize failure causes → experiment against a baseline → measure; automated evaluators never remove the need for ongoing human annotation and calibration.

### 4. Eval-first development ("evals are tests") — the practitioner consensus

- **Hamel Husain, "Your AI Product Needs Evals"** (Mar 2024, still the canonical reference; [hamel.dev](https://hamel.dev/blog/posts/evals/), [Simon Willison's note](https://simonwillison.net/2024/Mar/31/your-ai-product-needs-evals/)): build a problem-specific evaluation system, not a generic framework; the eval system is the flywheel that makes iteration fast; unsuccessful AI products "almost always share a common root cause: a failure to create robust evaluation systems." Start with the simplest layer — code-based assertions for things the system must never do — before any LLM judging.
- **Hamel's "Field Guide to Rapidly Improving AI Products"** ([hamel.dev field guide](https://hamel.dev/blog/posts/field-guide/)) adds the operating mechanics: **error analysis is the highest-value activity** — review real traces in a *custom* viewer, capture open-ended notes, let failure categories emerge bottom-up (don't start from canned metrics like "hallucination"), prioritize categories covering 60%+ of failures (NurtureBoss: date-handling success 33%→95% after this). Custom data viewers "can be built in hours with AI-assisted development" and give ~10× iteration speed. **Binary pass/fail + written critique** over scales. **Synthetic data along three dimensions (features × scenarios × personas)**, generating *inputs* not expected outputs, grounded in real system constraints — works "even with zero users" and remains permanent infrastructure. Alignment between automated judge and human "isn't a one-time setup but an ongoing conversation."
- **Eval-driven development (EDD)** is now a named movement ([evaldriven.org](https://evaldriven.org/), [sdarchitect on EDD for agents](https://sdarchitect.blog/2025/10/21/ai-agents-the-case-for-eval-driven-development/), reference architecture in [arXiv:2411.13768](https://arxiv.org/pdf/2411.13768)): define success criteria as executable evals *before* building, evaluate every iteration against the baseline. The academic caveat: pure pass/fail assertions under-fit open-ended agent tasks; the practical answer is layering (assertions → judge → human).
- **"When Better Prompts Hurt"** ([arXiv:2601.22025](https://arxiv.org/pdf/2601.22025), Jan 2026) is the empirical warning shot for prompt-library maintenance: generic prompt "improvements" are **not monotonic** — appending generic rules dropped one RAG compliance suite from 26/30 to 9/30 — so every prompt edit must be treated as a regression risk and run through a **Minimum Viable Evaluation Suite (MVES)** per task before adoption. As prompts are iteratively patched to fix specific issues, **cumulative drift** can quietly degrade other dimensions.

### 5. Prompt regression testing, "prompt rot," and model-swap re-validation

- **Prompt rot, defined** ([Vaibhav Yadav, Jan 2026](https://medium.com/@yadavvaibhav0104/prompt-rot-why-your-perfect-prompts-stop-working-and-how-to-fix-them-3d45dd79cc60)): carefully tuned prompts — especially few-shot examples, rigid phrasing, "magic" wording — are **implicitly overfit to a specific model version**; what looked like reliable prompting was an undocumented interaction with the old model. Rot rarely causes hard failures (outputs still "look reasonable"), so it gets misattributed to randomness. Mitigations: explicit instructions over inferred patterns; few-shot examples to clarify *intent*, not encode rules; enforce constraints **outside the prompt** (validators, schemas) wherever possible.
- **PromptDebt** ([arXiv:2509.20497](https://arxiv.org/pdf/2509.20497)) studies the org-level symptom: "prompt graveyards" — collections of similar mega-prompts nobody dares delete; simple changes require hours of editing and re-testing.
- **Tooling**: [promptfoo](https://github.com/promptfoo/promptfoo) (declarative test configs, 60+ providers, CI gating; founded 2024, **acquired by OpenAI March 2026 but stays MIT/open source** per [mager.co guide, Feb 2026](https://www.mager.co/blog/2026-02-23-promptfoo-llm-validation)) is the de-facto harness for prompt × model matrix regression: same test suite run against old and new prompt/model, diff per test, CI fails on regression ([tutorial with GitHub Actions gating](https://nerdleveltech.com/promptfoo-test-llm-prompts-ci-tutorial)). Lightweight alternatives: [lrts](https://github.com/rufus-SD/lrts) (runs same inputs against two prompt versions, scores drift, CI exit code). [Agenta on prompt drift](https://agenta.ai/blog/prompt-drift): **pin the model/version in CI** (never silently upgrade), and when drift persists, replace exact-match assertions with **contract checks** (schema/invariant assertions) — behavior tests survive model swaps better than string tests.
- **Model-migration runbooks (vendor-grade)**: OpenAI's curated upgrade-guide skill ([openai/skills upgrade-guide.md](https://github.com/openai/skills/blob/724cd511c96593f642bddf13187217aa155d2554/skills/.curated/openai-docs/references/upgrade-guide.md)) prescribes an ordered process directly reusable as a maintainer runbook: **(1) inventory** every model string and prompt-bearing file; **(2) pair prompts to usage**; **(3) classify the source model**; **(4) decide upgrade class** (string-only / light rewrite / blocked); **(5) compatibility gate**; **(6) apply — one change at a time, hold reasoning-effort and API surface constant, leave pinned eval baselines untouched**; **(7) summarize results**. Evals compare against the prior-model baseline on task success, retries, tool calls, tokens, latency, output shape; confirm each prompt edit "is doing real work instead of adding noise."
- **"Trim before rewrite"**: the GPT-5.5 migration guidance ([knightli.com, May 2026](https://knightli.com/en/2026/05/15/gpt-5-5-prompting-guide/), [Simon Willison on the GPT-5.5 prompting guide](https://simonwillison.net/2026/apr/25/gpt-5-5-prompting-guide/)) converges on: don't port the whole old prompt stack — old prompts accumulate **workarounds for the previous model's weaknesses** that actively harm the newer model; first switch the model with prompts frozen, run the eval suite to isolate the model delta, then *remove* stale workarounds incrementally, re-running evals after each small change. Treat each model family "as a new model family to tune for, not a drop-in replacement."

### 6. Agent harness design patterns (the system around the model decides performance)

- "Harness engineering" is now a named discipline ([Addy Osmani, Agent Harness Engineering](https://addyosmani.com/blog/agent-harness-engineering/), [Rajiv Shah](https://rajivshah.com/blog/harness-engineering.html), [Avi Chawla, Anatomy of an Agent Harness](https://blog.dailydoseofds.com/p/the-anatomy-of-an-agent-harness), curated index at [awesome-harness-engineering](https://github.com/ai-boost/awesome-harness-engineering)). Recurring patterns relevant to long-horizon maintenance loops:
  - **Externalized memory over context stuffing**: persistent artifacts (AGENTS.md, todo.md, progress logs) outside the window; the harness summarizes/offloads old context to fight "context rot."
  - **Context resets with structured handoff** beat in-place compaction for coherence: clear the window, start a fresh agent, carry forward an explicit state + next-steps document ([Anthropic, harness design for long-running apps](https://www.anthropic.com/engineering/harness-design-long-running-apps) — included as cross-reference; the same pattern appears vendor-independently in the Ralph-loop writeups).
  - **The Ralph loop** (re-running the agent in a fresh context with the original intent re-injected, plus durable on-disk state) for tasks longer than one session ([Osmani, Self-Improving Coding Agents, Jan 2026](https://addyosmani.com/blog/self-improving-agents/)).
  - **Evaluating the harness, not just the model**: promptfoo's agent-eval guidance stresses "the model might be identical, but the agent harness determines what's possible — you're evaluating the system" ([promptfoo coding-agent evals](https://www.promptfoo.dev/docs/guides/evaluate-coding-agents/)).
- **Four-channel durable memory** (Ryan Carson's Ralph implementation, via [Osmani](https://addyosmani.com/blog/self-improving-agents/)): (1) **git history** as inspectable diffs; (2) **progress log** (chronological attempts/failures); (3) **machine-readable task state** (prd.json with pass/fail fields); (4) **AGENTS.md** semantic knowledge. The compounding principle, from Compound Product's README: "each improvement should make future improvements easier."
- **Voyager** (Wang et al. 2023, [arXiv:2305.16291](https://arxiv.org/abs/2305.16291), [repo](https://github.com/minedojo/voyager)) remains the canonical **skill-library** result: each newly solved task becomes *verified executable code*, stored in a growing library **indexed by natural-language description**; later tasks retrieve and compose verified skills instead of re-deriving them — compounding capability and "alleviating catastrophic forgetting." The 2026 follow-on literature (e.g. [MUSE-Autoskill](https://arxiv.org/html/2605.27366v1), [survey of post-training/memory/skills](https://arxiv.org/pdf/2512.16301)) generalizes this: skill creation + skill *management* (dedup, eviction, evaluation of skills themselves) is the active research edge.

### 7. Self-updating documentation and living runbooks

- **Detection should be cheap and deterministic where possible**: Fiberplane's **drift** linter (Mar 25, 2026; [blog](https://fiberplane.com/blog/drift-documentation-linter/)) deliberately avoids LLM checks ("burning tokens every commit to answer 'did this doc go stale?' is expensive for what should be a trivial check"). Mechanism: docs carry frontmatter **anchors** (file path + optional symbol `#AuthConfig` + git-sha provenance `@<sha>`); tree-sitter parses the anchored code, **AST fingerprints** (normalized, whitespace-insensitive hashes) are compared against the recorded baseline; `drift check` exits 1 in CI when a doc's anchored code changed; `drift link` re-stamps provenance after human review. Their stated design lesson: "the real value isn't the tool, it's the constraint… build or use more tools that can **yell at your agents**."
- **LLM-driven doc repair** sits one layer up: [DeepDocs](https://deepdocs.dev/) runs on every commit, identifies stale docs and proposes updates; the practitioner framing is that doc updates should be generated/blocked as part of the delivery pipeline, like CI/CD replaced manual builds ([Live-Docs overview, Apr 2026](https://www.progressiverobot.com/2026/04/30/live-docs/)).
- **Agent instruction files are now load-bearing docs**: AGENTS.md/CLAUDE.md are *prescriptive execution context*, so they must co-evolve with the codebase; emerging research even derives **executable constraints from instruction files** and enforces them ([ContextCov, arXiv:2603.00822](https://arxiv.org/pdf/2603.00822)). Osmani's maintenance guidance for accumulated-learnings files ([self-improving agents](https://addyosmani.com/blog/self-improving-agents/)): keep entries **brief and factual**; **prune periodically and archive (not delete) superseded lessons**; split knowledge into topical files and **inject only task-relevant sections** (indiscriminate inclusion degrades performance — context bloat); periodically compress old progress logs into distilled insights; and prefer *high-quality examples in the codebase itself* (clean tests, clear patterns) over prose instructions — agents follow what they see (per Simon Willison, via Osmani).

## Design implications for neokapi

Concrete transfers into the format-maintenance runbook (format-triage skill, L0–L4 rubric, spec.yaml suites, parity harness, /format-maturity dashboard):

- **Treat spec.yaml suites as the DSPy metric; treat the rubric/skill prompts as the optimized artifact.** The optimization loop neokapi needs is GEPA-shaped, not gradient-shaped: run the triage/remediation skill on N formats → collect failure annotations *as text* (which assertion failed, why, with citation) → a reflection pass proposes edits to the skill prompt / rubric wording → keep a small archive of rubric versions scored against a **fixed held-out set of formats** and only promote a version that Pareto-dominates ([GEPA](https://arxiv.org/abs/2507.19457), [dspy.ai GEPA](https://dspy.ai/getting-started/gepa-optimization/)). No DSPy dependency required — the mechanism is 4 files and a script.
- **Every metric/judge in the process should emit score + feedback text, never a bare number.** This is GEPA's API lesson and Hamel's "terse critiques" warning simultaneously: the textual critique is what powers the next round of rubric improvement.
- **Ground all self-improvement in the deterministic oracles neokapi already has.** Reflexion-style loops work where outcomes are checkable; round-trip byte-equality, Okapi parity, and spec assertions are exactly that. Reserve LLM judgment only for what code can't check (e.g. "is this divergence semantically faithful?"), per the assertion-first layering ([Hamel evals](https://hamel.dev/blog/posts/evals/), [Eugene Yan](https://eugeneyan.com/writing/eval-process/)).
- **Where the process uses an LLM judge (e.g. scoring rubric compliance or divergence-faithfulness), validate it with critique shadowing**: the maintainer is the single principal expert; grade ~30+ format-audit outputs binary pass/fail with critiques until no new failure modes appear; iterate the judge prompt to >90% agreement; store the graded set as a **golden agreement set** and re-run it whenever the judge's model changes; recalibrate if divergence exceeds ~20–25% ([hamel.dev judge guide](https://hamel.dev/blog/posts/llm-judge/), [Deepchecks](https://deepchecks.com/llm-judge-calibration-automated-issues/)).
- **Binary, per-criterion checks in the L0–L4 rubric; no 1–5 scoring anywhere.** Maturity levels should decompose into named binary gates (has malformed_test, has spec citations, round-trip byte-exact, …) — which neokapi mostly already does; resist any future "score 1–5 for documentation quality" criterion ([field guide](https://hamel.dev/blog/posts/field-guide/)).
- **Expect and embrace criteria drift in the rubric.** EvalGen's core finding says the L0–L4 rubric cannot be finalized a priori: each triage/audit session will surface criteria revisions. Make rubric edits a *first-class, versioned output* of every triage run (a `rubric-changelog` section), and re-score affected formats after a rubric change so the dashboard never mixes scores from different rubric versions ([arXiv:2404.12272](https://arxiv.org/abs/2404.12272)).
- **Adopt the vendor-grade model-migration runbook for the skill/prompt library** (directly from [openai/skills upgrade-guide](https://github.com/openai/skills/blob/724cd511c96593f642bddf13187217aa155d2554/skills/.curated/openai-docs/references/upgrade-guide.md) + [trim-before-rewrite guidance](https://knightli.com/en/2026/05/15/gpt-5-5-prompting-guide/)): (1) inventory every prompt-bearing file (.skills/*, harness demo.yaml narration prompts, judge prompts, CLAUDE.md guidance); (2) on a model upgrade, freeze prompts, swap the model, run the fixed eval set to isolate the model delta; (3) then **delete stale workarounds** (few-shot crutches, phrasing hacks for the old model) one at a time, re-running the eval after each edit; (4) record a migration summary. One change at a time, always against a pinned baseline.
- **Counter prompt rot structurally, not heroically**: move constraints out of prompts into validators wherever possible (schema checks on skill outputs, spec.yaml assertions, maturity_test.go guardrails); keep few-shot examples for *intent*, not rules; and pin model versions in any CI that runs prompts so upgrades are explicit events, never silent drift ([prompt rot](https://medium.com/@yadavvaibhav0104/prompt-rot-why-your-perfect-prompts-stop-working-and-how-to-fix-them-3d45dd79cc60), [Agenta](https://agenta.ai/blog/prompt-drift)).
- **Every prompt edit is a regression risk — gate it with an MVES.** Maintain a small (≈20–30 case) fixed suite per prompt surface (triage skill, audit script, judge) and run old-vs-new like promptfoo does (diff per case, fail on net regression); "generic improvements" to shared prompt boilerplate must be tested per-task because they are not monotonic ([arXiv:2601.22025](https://arxiv.org/pdf/2601.22025), [promptfoo](https://github.com/promptfoo/promptfoo)).
- **Make the runbook a living document with drift anchors.** Anchor runbook/skill claims ("the maturity script lives at scripts/audit-format.py", "49 formats") to file paths + symbols + git provenance and check them deterministically in CI, Fiberplane-drift style — cheap AST/path checks first, LLM doc-repair only as the second layer. The design slogan transfers verbatim: build tools that *yell at the agent* when the runbook and code disagree ([fiberplane drift](https://fiberplane.com/blog/drift-documentation-linter/)).
- **Accumulate lessons Voyager-style: verified, executable, indexed — then curate.** When a format audit discovers a reusable fix pattern (e.g. a skeleton-emitter repair recipe), capture it as an executable artifact (test + fixture + snippet) indexed by a one-line natural-language description, not prose alone; keep per-topic learnings files small, inject only the relevant one per task, prune on a schedule, and archive (never silently delete) superseded lessons ([Voyager](https://arxiv.org/abs/2305.16291), [Osmani](https://addyosmani.com/blog/self-improving-agents/)).
- **Durable multi-channel state for long-horizon format work**: the triage loop should persist (1) git commits per remediation, (2) a chronological progress log, (3) machine-readable per-format status (the dashboard dataset already is this), and (4) a semantic learnings file — and prefer fresh-context restarts with structured handoffs over ever-growing sessions ([Ralph loop / four channels, Osmani](https://addyosmani.com/blog/self-improving-agents/)).
- **Bottom-up error analysis on the triage transcripts themselves.** Periodically review real triage-run transcripts in a minimal custom viewer, tag failure modes open-endedly, and fix the categories covering the top ~60% of failures — improving the *process* (rubric wording, skill steps, tool ergonomics), not just the formats ([field guide](https://hamel.dev/blog/posts/field-guide/)).
- **Synthetic fixture generation along three axes** (feature × scenario × persona ⇒ for formats: construct × document-shape × producer-tool): generate *inputs* (documents), never expected outputs; verify each generated fixture actually triggers the targeted construct; keep them as permanent infrastructure — this is the cheapest way to push the 45/48 missing malformed_test gap ([field guide](https://hamel.dev/blog/posts/field-guide/)).
