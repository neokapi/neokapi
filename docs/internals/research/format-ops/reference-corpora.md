# Reference-file corpora for format testing: harvesting, licensing, storage, generation

Research stream for neokapi format ops. Date: 2026-06-11. All load-bearing claims carry URLs; primary sources preferred.

## Findings

### 1. Large harvested public corpora (the "real files" baseline)

**Govdocs1 (Digital Corpora, 2009)** — ~986,278 files (~1M originally, minus ~13,722 withdrawn California files) collected by issuing random dictionary-word and random-number queries against Yahoo/Google restricted to the US `.gov` domain. Files are "freely available for research and may be (to the best of our knowledge) freely redistributed" — US government works, so effectively public domain; this redistribution-safety is *the* reason govdocs1 became the canon for format-tool testing. Hosted as 1,000 directories × 1,000 files, plus 1,000 zipfiles, plus by-type tarballs (e.g. `files.jpeg.tar`) and ten 1,000-file "subset" zips for development/testing splits, at `https://downloads.digitalcorpora.org/corpora/files/govdocs1/` (S3-backed). >40 file types, but extensions often don't match real content (an artifact of harvest) — useful adversarial property for format detection tests. Source: https://digitalcorpora.org/corpora/file-corpora/files/

**NapierOne (2021–22)** — designed explicitly as "a modern mixed file data set alternative to Govdocs1": 44 popular file types × ~5,000 examples each, >500,000 files, type-validated and curated (govdocs1 is dated 2009 and skews to legacy formats). Licensed under the Edinburgh Napier University license, which permits copying, adaptation, redistribution and commercial exploitation with attribution. Distributed via GitHub index + AWS Open Data registry. Sources: https://github.com/simonrdavies/NapierOne , https://registry.opendata.aws/napierone/ , paper https://arxiv.org/abs/2201.08154

**SafeDocs / CC-MAIN-2021-31-PDF-UNTRUNCATED (2021, DARPA SafeDocs, JPL)** — ~8M PDFs / ~8TB, built from Common Crawl's July–Aug 2021 crawl: ~6M extracted directly from CC WARCs, plus ~2M *refetched from origin URLs* because Common Crawl truncates payloads at 1MB (a critical gotcha when harvesting binaries from CC). Packaged as 7,933 zipfiles of 1,000 PDFs each on Digital Corpora. This is the state-of-the-art recipe for harvesting one format at web scale. Sources: https://digitalcorpora.org/corpora/file-corpora/cc-main-2021-31-pdf-untruncated/ , construction notes https://digitalcorpora.org/corpora/file-corpora/cc-main-2021-31-pdf-untruncated/constructing-the-cc-main-2021-31-pdf-untruncated-corpus/ , tooling https://github.com/tballison/CC-MAIN-2021-31-PDF-UNTRUNCATED , PDF Association announcement https://pdfa.org/new-large-scale-pdf-corpus-now-publicly-available/

**UNSAFE-DOCS (CC-MAIN-2021-31-UNSAFE)** — companion corpus of documents that triggered AV/parser alarms; useful as a hostile-input stress set. https://digitalcorpora.org/corpora/file-corpora/unsafe-docs-cc-main-2021-31-unsafe/

**PDF Association corpus index** — `pdf-association/pdf-corpora` is a curated *index of indexes* for PDF-centric corpora; the pattern (a small git repo that catalogs external corpora with provenance, rather than vendoring bytes) is directly reusable. https://github.com/pdf-association/pdf-corpora

### 2. Curated small-example corpora + per-file provenance practice

**OPF format-corpus (`openpreserve/format-corpus`)** — "an openly-licensed corpus of small example files, covering a wide range of formats and creation tools"; **all items CC0 unless otherwise stated**, which is the cleanest possible redistribution posture for vendored test files. Files are deliberately *created by contributors* (e.g. the `office-examples/` tree explores the same simple document saved through many applications' "Save As…" variants) so copyright is clear by construction. It also hosts a curated set of govdocs1 *error PDFs* (~130k PDFs known to break PDF software) for stress testing. Sources: https://github.com/openpreserve/format-corpus , https://github.com/openpreserve/format-corpus/blob/master/README.md , https://github.com/openpreserve/format-corpus/tree/master/govdocs1-error-pdfs

Its per-file **metadata template** (`metadata-template.ext.md`, a sidecar markdown file per test file) records: `formatName`, `formatVersion`, `extensions`, `mimeType`, `mimeTypeAliases`, `pronomId`, `xmlNameSpace`, `creatorTool`, `creatorToolUrl`, `formatSpecUrl`. https://github.com/openpreserve/format-corpus/blob/master/metadata-template.ext.md — this is the closest existing thing to a "provenance manifest convention" for format test files, and notably it links each file to a **registry ID (PRONOM)** and a **spec URL**, mirroring what neokapi's spec.yaml already does.

A community re-factor exists (`ross-spencer/opf-format-corpus`, reorganized "for usability and maintainability") showing the original layout aged poorly — argument for machine-readable manifests over README prose. https://github.com/ross-spencer/opf-format-corpus

### 3. Format registries as ground truth and as *generators*

**PRONOM** (The National Archives UK) is the canonical technical registry of formats; **DROID** consumes XML *signature files* generated from PRONOM (binary magic + container signatures, versioned snapshots) — https://www.nationalarchives.gov.uk/aboutapps/pronom/droid-signature-files.htm , https://github.com/digital-preservation/droid

**Siegfried** (`richardlehane/siegfried`, v1.11.4, Jan 23 2026 — actively maintained) is a Go single-binary identifier implementing PRONOM signatures plus freedesktop.org MIME-info, Library of Congress FDD signatures (beta), and Wikidata (beta). Signatures update via `sf -update`; the `roy` companion tool builds *custom* signature files; output in YAML/JSON/CSV/DROID-CSV. https://github.com/richardlehane/siegfried — being Go, it (or its signature model) is embeddable/automatable in a neokapi corpus pipeline (e.g. CI assertion "every corpus file identifies as the PUID declared in its manifest").

**Skeleton Test Suite** (digipres community) *generates* minimal test files directly from PRONOM binary/container signatures — i.e., synthesizes the smallest byte stream that should identify as format X — used to test DROID-compatible identifiers. Referenced from https://github.com/digipres/awesome-digital-preservation ; aggregated registry search at https://www.digipres.org/formats/

**Library of Congress "Sustainability of Digital Formats"** — 470+ Format Description Documents (FDDs, `fdd000001`-style IDs), each covering sustainability factors (adoption, disclosure, patents), structure, and citations; authored in XML and downloadable in XML; actively updated (13 new FDDs reported June 2025). Best used as the authority to cite in per-format spec.yaml `spec_citations`, and FDD signatures flow into Siegfried. Sources: https://www.loc.gov/preservation/digital/formats/ , https://www.loc.gov/preservation/digital/formats/fdd/descriptions.shtml , https://blogs.loc.gov/thesignal/2025/06/new-file-format-research/

### 4. Bug-tracker harvesting and large-scale regression practice (closest analogues to neokapi's problem)

**Apache Tika / POI / PDFBox shared regression corpus (TIKA-1302)** — the three communities jointly assembled **~3M files / ~1TB** from govdocs1, Common Crawl, *and their own bug trackers*, and run candidate releases across it with the `tika-eval` module (single-VM, embedded H2 DB, designed for "a few million files"), diffing extraction output between versions to catch regressions pre-release. Sources: https://openpreservation.org/blogs/apache-tikas-regression-corpus-tika-1302/ , https://cwiki.apache.org/confluence/display/TIKA/TikaEval , https://cwiki.apache.org/confluence/display/tika/TikaEvalOnVM , ApacheCon slides http://events17.linuxfoundation.org/sites/events/files/slides/ApacheConMiami2017_tallison_v2.pdf . Key mechanism: **every attachment on a parser bug is corpus material** — the bug tracker is a self-replenishing harvest of exactly the files that break you.

**LibreOffice crash-testing** — continual automated import+re-save of a **75,000+ document corpus** (grown from ~24,500), harvested largely from Bugzilla attachments across doc/docx/odt/rtf/xls/ppt/etc.; a Python driver loads each document, round-trips it to multiple target formats, and monitors crashes/validation; trends are tracked over time. Sources: https://dev.blog.documentfoundation.org/2024/04/18/crash-fixes-part-3-testing-crashes/ , https://mmohrhard.wordpress.com/2013/04/19/automated-import-crash-testing-in-libreoffice/ . Note the *round-trip-and-revalidate* shape is exactly neokapi's read→write→compare maturity test, scaled to a harvested corpus and run out-of-band from PR CI (it took days per pass).

### 5. Common Crawl as a harvesting source — capability vs. terms

Capability: CC publishes WARC/WAT/WET per crawl; per-format harvesting = filter the columnar index by `content_mime_type` / URL extension, extract payloads from WARCs, and **refetch any payload marked truncated** (1MB cap) — the SafeDocs pipeline above is the worked example. https://commoncrawl.org/ , https://github.com/tballison/CC-MAIN-2021-31-PDF-UNTRUNCATED

Terms: CC's Terms of Use grant only a "limited, non-assignable, non-transferable, non-sublicensable, non-exclusive" license to the *service*; CC "do[es] not and cannot offer a license to the crawled page contents" — crawled content remains copyrighted by its origin owners and "may be subject to separate terms of use … from the owners of such Crawled Content." https://commoncrawl.org/terms-of-use , https://commoncrawl.org/faq . Practical consequence: **CC-derived files are fine to fetch-on-demand for local/CI testing but are not safely re-vendorable into a repo you redistribute**; even Digital Corpora's CC-derived corpora ship with their own terms rather than a clean license (the HF `pixparse/pdfa-eng-wds` card explicitly tells users to heed both Common Crawl's and Digital Corpora's terms: https://huggingface.co/datasets/pixparse/pdfa-eng-wds ).

### 6. Hugging Face datasets as both source and distribution channel

- **FinePDFs** (HuggingFaceFW, Sept 2025): 475M documents / ~3T tokens extracted from PDFs across 105 CC snapshots (2013–Feb 2025), processed with `datatrove`; released under **ODC-By**. It is *extracted text*, not original bytes — useful for content sampling, not for byte-exact format tests. https://huggingface.co/datasets/HuggingFaceFW/finepdfs , https://www.infoq.com/news/2025/09/finepdfs/
- **pixparse/pdfa-eng-wds**: filtered SafeDocs-derived PDF dataset (original bytes + OCR annotations) in WebDataset shards. https://huggingface.co/datasets/pixparse/pdfa-eng-wds
- **As a distribution channel**: HF Hub has no per-repo size limit (free public storage; >1TB needs a Team/Enterprise plan or a quota request), a 500GB hard per-file cap, Xet-backed content-defined-chunk dedup (so re-uploading near-identical corpora is cheap), and guidance to keep total file count <100k and ≤10k files/folder — i.e., ship corpora as tar/parquet shards, not loose files. Dataset cards carry a structured YAML `license:` field, enforcing license declaration at the repo level. https://huggingface.co/docs/hub/en/storage-limits , https://huggingface.co/docs/hub/en/datasets-upload-guide-llm

### 7. OSS-Fuzz seed-corpus practice (the operational gold standard)

- Ideal-integration doc: "The ideal corpus is a minimal set of inputs that provides maximal code coverage"; the seed corpus "should be available in revision control (can be the same as or different from the source code)" and should be "regularly extended with the inputs that (used to) trigger bugs and/or touch new parts of the code"; fuzz targets should also be run over the corpus as plain regression tests (not fuzzing) under sanitizers in normal CI. https://google.github.io/oss-fuzz/advanced-topics/ideal-integration/
- Packaging convention: archive named `<fuzz_target>_seed_corpus.zip`, placed in `$OUT` next to the fuzzer, zipped with `-j` so the archive is flat (each member = one input). Licensing guidance: "ensure your seed corpus has an appropriate and consistent license, and do not include proprietary or sensitive data." https://deepwiki.com/google/oss-fuzz/4.4-seed-corpora-and-dictionaries , https://github.com/google/oss-fuzz/blob/master/docs/faq.md
- Dedicated-corpus-repo pattern in the wild: Cisco Talos keeps ClamAV's seed corpus in its own repo, `Cisco-Talos/clamav-fuzz-corpus`. https://github.com/Cisco-Talos/clamav-fuzz-corpus
- Research: seed selection materially changes fuzzing outcomes; minimized/distilled corpora outperform both empty and "everything" corpora ("Seed Selection for Successful Fuzzing", ISSTA 2021, https://dl.acm.org/doi/pdf/10.1145/3460319.3464795 ; corpus-distillation comparison https://arxiv.org/pdf/1905.13055 ). Practical corpus-building walkthrough: https://blog.isosceles.com/how-to-build-a-corpus-for-fuzzing/
- **Go-native specifics** (relevant since neokapi is Go): seed inputs live in `testdata/fuzz/<FuzzTargetName>/` as versioned text-encoded corpus files (`go test fuzz v1` header + typed values), or are registered with `f.Add(...)`; `file2fuzz` (golang.org/x/tools/cmd/file2fuzz) converts raw binary files into `[]byte` corpus entries. https://go.dev/doc/security/fuzz/ , https://github.com/golang/go/issues/48717

### 8. Synthetic and adversarial corpus generation (when real files are unavailable)

- **FormatFuzzer** (CISPA): compiles 010 Editor *binary templates* (a large existing library of community-maintained format grammars) into C++ parser+generator+mutators; generates thousands of mostly-valid files/sec for formats like MP4/ZIP; usable standalone (pure generator) or as an AFL mutator. https://arxiv.org/html/2109.11277 , repo via paper (uds-se/FormatFuzzer)
- **Grammar-based / structure-aware fuzzing** canon: Google's structure-aware-fuzzing doc covers generation-based approaches and language-specific generators (csmith for C, wasm-smith for Wasm). https://github.com/google/fuzzing/blob/master/docs/structure-aware-fuzzing.md
- **libprotobuf-mutator pattern**: define a protobuf that models the format's structure, mutate the proto tree, and write a `proto → bytes` converter — protobuf as the intermediate representation for any complex format. https://github.com/google/libprotobuf-mutator , LLVM worked example https://llvm.org/devmtg/2017-10/slides/Serebryany-Structure-aware%20fuzzing%20for%20Clang%20and%20LLVM%20with%20libprotobuf-mutator.pdf
- **LLM-synthesized generators/seeds** (2025–26): generating *input generators* (not inputs) with LLMs for non-textual formats ( https://arxiv.org/pdf/2501.19282 ), and LLM-driven seed-corpus construction (SeedAIchemy, Nov 2025, https://arxiv.org/pdf/2511.12448 ). Directly applicable to a Claude-driven solo workflow: author a generator per format, keep the generator in-repo, regenerate corpora deterministically.
- **Registry-driven synthesis**: the digipres Skeleton Test Suite generates files from PRONOM signatures (minimal valid magic/container shells) — cheap way to get one guaranteed-identifiable specimen per format/version. via https://github.com/digipres/awesome-digital-preservation
- **Creation-tool matrix synthesis**: OPF `office-examples` shows the other axis — same logical document saved by N real applications — capturing producer variance that grammars miss. https://github.com/openpreserve/format-corpus/blob/master/office-examples/README.md

### 9. Licensing + provenance-manifest practice for redistributable test files

- **Safe-to-vendor licenses**, in descending order of safety: CC0 / public domain (OPF format-corpus default), US-government works (govdocs1), permissive-with-attribution open-data licenses (NapierOne's ENU license; ODC-By for FinePDFs). **Not safe to vendor**: anything pulled from Common Crawl or the open web without an explicit license (see §5); OSS-Fuzz's own guidance is a consistent license and no proprietary/sensitive data (§7).
- **REUSE/SPDX is the tooling-ready mechanism for per-file license declarations on binary fixtures**: REUSE Spec 3.3 requires Licensing Information for every covered file, and for binaries supports (a) an adjacent `<file>.license` sidecar, or (b) bulk declarations in `REUSE.toml` (which replaced `.reuse/dep5`; the two are mutually exclusive, dep5 deprecated but honored "at least until 2029"). Tags are `SPDX-FileCopyrightText:` + `SPDX-License-Identifier:` with valid SPDX expressions; `reuse lint` machine-verifies the whole tree. https://reuse.software/spec-3.3/ , https://reuse.software/faq/ , https://reuse.readthedocs.io/en/stable/man/reuse-lint.html
- **Field set observed across good corpora** (union of OPF metadata template §2, HF dataset cards §6, Digital Corpora pages §1): file id/hash (SHA-256), format name + version, declared extension vs. detected type, registry IDs (PRONOM PUID, LoC FDD), MIME type(s), creator tool + version + URL, spec URL(s), origin (URL / bug-tracker issue / generator + seed), harvest date, license (SPDX expression) + license evidence URL, redistribution status (vendor / fetch-on-demand / generate), and known defects (e.g. "truncated", "invalid xref" — the govdocs1-error-pdfs idea).

### 10. Storage tradeoffs

| Option | Limits / cost | Fit for test corpora |
|---|---|---|
| In-repo (git) | repo should stay well under 1GB (GitHub nudges >1GB, strongly encourages <5GB); 2GiB push cap | Best for the *curated tier*: small, CC0/owned files with sidecar manifests; corpus history bisects with code. https://docs.github.com/en/repositories/creating-and-managing-repositories/repository-limits |
| Git LFS | free tier 1GB storage + 1GB/month bandwidth, then $5/50GB; breaks offline `git bisect`; CI that forgets `lfs: true` fails confusingly | Generally poor for OSS test corpora (bandwidth-metered for every clone/fork); community workaround is LFS→S3/R2 proxying. https://testdouble.com/insights/why-use-git-large-file-storage , https://dbushell.com/2024/07/15/replace-github-lfs-with-cloudflare-r2-proxy/ , https://github.com/nicolas-graves/lfs-s3 |
| GitHub release assets | 2GiB/file, up to 1,000 assets/release, **no total-size or bandwidth limit** | Excellent fetch-on-demand tier — and neokapi already runs this pattern for docs assets (`docs-assets` release + `make fetch-docs-assets`). https://docs.github.com/en/repositories/releasing-projects-on-github/about-releases |
| S3 / R2 / AWS Open Data | egress costs (S3) vs. zero-egress (R2 10GB free); Open Data program hosts NapierOne, Digital Corpora | Right for multi-GB harvested corpora; what Digital Corpora itself does. https://registry.opendata.aws/napierone/ |
| Dedicated corpus repo | same git limits, but isolates churn/size from the code repo | Proven pattern (clamav-fuzz-corpus; OSS-Fuzz explicitly allows corpus in a different repo). https://github.com/Cisco-Talos/clamav-fuzz-corpus |
| HF dataset repo | free public storage, ~no repo cap (<1TB without a plan), 500GB/file, Xet dedup, structured `license:` card, keep <100k files via shards | Best for publishing a *large* redistributable corpus with enforced license metadata and cheap re-uploads. https://huggingface.co/docs/hub/en/storage-limits |

## Design implications for neokapi

1. **Three-tier corpus per format**, mirroring industry practice:
   - **Tier A "exemplars" (in-repo, vendored)**: ≤ a few MB per format; only CC0/self-created/US-gov files; each file carries a REUSE sidecar (`<file>.license` with SPDX tag) plus an entry in the per-format manifest. This is the tier maturity tests and spec.yaml assertions run on in PR CI.
   - **Tier B "harvested" (fetch-on-demand)**: hundreds–thousands of real-world files per format pulled from govdocs1 by-type tarballs, NapierOne per-type zips, SafeDocs zip shards, and Tika-style bug-tracker attachments; cached by hash; *never vendored* when license is unclear (Common Crawl rule, §5). Run in nightly/weekly jobs like LibreOffice crash-testing, not on every PR.
   - **Tier C "synthetic" (generated, deterministic)**: per-format generator (Go, seeded) + Go fuzz corpora in `testdata/fuzz/`; regenerate rather than store; promote any minimized crasher/regression input into Tier A with provenance `origin: fuzzer` (the OSS-Fuzz "extend corpus with inputs that used to trigger bugs" loop).

2. **Manifest schema (`corpus.yaml` per format**, sibling to spec.yaml): for each file record `sha256`, `size`, `format` + `version`, `puid` (PRONOM), `loc_fdd` (optional), `mime`, `ext_declared` vs `type_detected`, `creator_tool`/`creator_tool_url`, `spec_url`, `origin` (one of `vendored|url|archive-member|bug|generated`), `source_url` (+ `archive`, `member` for zip shards), `harvest_date`, `license` (SPDX expression; `LicenseRef-Unverified` allowed only in Tier B), `license_evidence`, `redistributable: true|false`, `defects: []`, `notes`. This is the OPF template (§2) made machine-readable + the REUSE/SPDX layer (§9) + content-addressing for cache integrity.

3. **Fetch-on-demand make target**: `make corpus FORMAT=docx` reads `corpus.yaml`, downloads Tier B sources into `~/.cache/neokapi/corpus/<sha256>` (XDG cache, consistent with the existing KAPI isolation contract), verifies SHA-256, extracts only the listed zip members (govdocs1 zipfiles and SafeDocs shards are 1,000-file zips — fetch the shard, extract the member, drop the shard). GitHub release assets are the natural mirror for a curated frozen snapshot (no bandwidth limits; neokapi already runs the `docs-assets` release + `make fetch-docs-assets` pattern — reuse it as `corpus-assets`).

4. **Identification as a corpus invariant**: run Siegfried (Go, embeddable, `sf -update` for current PRONOM) over every corpus file in CI and assert the detected PUID matches the manifest's `puid`; this simultaneously regression-tests neokapi's own format detection against the registry ground truth and catches corrupted/mislabeled corpus entries. Add `puid` to each format's spec.yaml if not already present.

5. **Bug-tracker = corpus inbox**: institutionalize the Tika/LibreOffice mechanism — every GitHub issue with an attached failing file gets that file (license permitting) minimized, manifest-ed (`origin: bug`, `source_url: <issue URL>`), and committed to Tier A in the fixing PR. Over years this is the highest-value corpus channel a solo maintainer has, and it's self-replenishing.

6. **Producer-variance matrix for office-like formats**: adopt the OPF office-examples axis — same canonical document authored via N producers (Word, LibreOffice, Google Docs export, Pages, python-docx, …) — since neokapi's moat is faithful round-trip and producer variance is where round-trips break. These are self-created → CC0 → Tier A vendorable.

7. **Synthetic generation strategy**: prefer (a) Go-native seeded generators per format (cheap for text formats: JSON/YAML/PO/ARB/strings); (b) for binary/container formats, consider FormatFuzzer's 010-template library or a libprotobuf-mutator-style proto IR; (c) skeleton-suite-style minimal magic shells for detection tests; (d) Claude-authored generators kept in-repo and regenerated deterministically (LLM writes the generator, not the files — reviewable, reproducible).

8. **Licensing red lines**: vendor only CC0/PD/US-gov/own-created files; NapierOne and ODC-By material may be vendored with attribution recorded in the manifest; Common Crawl / web-harvested / most bug-attachment material stays fetch-on-demand with `redistributable: false`. Enforce with `reuse lint` (REUSE 3.3 sidecar/`REUSE.toml`) in CI so an unlicensed fixture can't land.

9. **Scale realism for a solo maintainer**: PR CI = Tier A only (seconds); nightly = Tier C fuzz + Tier B sample (Siegfried-verified, e.g. 100/format); monthly/manual = full Tier B sweep (the LibreOffice multi-day pass is out-of-band for a reason). Corpus distillation research (§7) says a *minimized* corpus beats a huge one — cap Tier A per format and require each file to justify itself via a distinct assertion or coverage delta.

10. **If/when neokapi publishes its corpus**: a Hugging Face dataset repo (structured `license:` card, Xet dedup, sharded layout) or a dedicated `neokapi/format-corpus` GitHub repo (clamav-fuzz-corpus pattern) — not Git LFS (metered bandwidth penalizes every fork/CI clone).
