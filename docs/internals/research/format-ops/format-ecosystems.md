# How mature OSS projects engineer and maintain many-format support

Research stream: testing, regression-corpus, and format-adoption processes in Pandoc, Apache Tika, LibreOffice, FFmpeg FATE, ExifTool, tree-sitter, and the unified/remark ecosystem. Researched June 2026; source dates noted where they matter.

## Findings

### Pandoc — golden tests, command tests, and a deliberately high core-acceptance bar

**Architecture.** Pandoc is a set of readers producing a native AST and writers consuming it; the `native` format is the serialized AST itself, used directly as a test oracle ([Pandoc manual](https://pandoc.org/MANUAL.html), [contributing guide](https://pandoc.org/CONTRIBUTING.html)).

**Three test tiers** ([CONTRIBUTING](https://pandoc.org/CONTRIBUTING.html)):
1. **Golden tests** in `test/` — input file + committed expected output; regenerate with `--accept` ("You can rebuild the golden tests in `tests/` by passing `--accept` to the test script"). Docx reader tests use `Tasty.Golden` with the same `--accept` update flow. Legacy reader/writer roundtrip-style tests are wired through `test/Tests/Old.hs`: "If you are adding a new reader or writer... it is probably easiest to add some data files to the `test` directory, and modify `test/Tests/Old.hs`."
2. **Unit tests** under `test/Tests/` mirroring the module hierarchy.
3. **Command tests** under `test/command/` — markdown files containing a shell invocation plus expected output, named after the issue/feature they pin: "For example, `5474-tables.md` refers to both issue and feature." This is the bug→regression-test flywheel: every fixed issue leaves a named, executable artifact.

**Hard rule:** "If you are adding a new writer or reader, you must include tests" ([CONTRIBUTING](https://pandoc.org/CONTRIBUTING.html)). CI runs the whole suite on every PR via GitHub Actions.

**Acceptance bar for new formats.** jgm (John MacFarlane) is explicit that the binding constraint is *his* long-term maintenance load, not contributor enthusiasm. On the 2025/26 HWPX writer proposal ([discussion #11569](https://github.com/jgm/pandoc/discussions/11569)): "This is a format I don't know at all... In the end, I'm often the one who ends up having to respond to bug reports, even if someone else contributed the writer," and "a writer/reader for a format that includes embedded resources in a zip-like structure is going to be more complicated to support... it would be a fairly heavy patch, when you include the source files, tests, reference doc, etc." He declines to commit before seeing code ("I wouldn't want to make a final decision on this before seeing the code") and points to escape valves:
- **Custom Lua readers/writers** — pandoc embeds Lua so anyone can ship a format without touching core ([custom readers](https://pandoc.org/custom-readers.html), [custom writers](https://pandoc.org/custom-writers.html); added via [#7669](https://github.com/jgm/pandoc/issues/7669)).
- **Standalone-library-first** — recent core readers (Djot, Typst) "delegate a lot of the work to separate Haskell packages" and only the AST-mapping layer lives in pandoc; a contributor can also ship "a standalone anything -> hwpx converter" built on the pandoc library ([#11569](https://github.com/jgm/pandoc/discussions/11569), [duckrabbit mdoc account](https://duckrabbit.tech/articles/mdoc.html)).

**Real-world corpus as the readiness gate.** The contributor of the mdoc reader (shipped in pandoc 3.6, Dec 2024) tested against "all the man pages written in mdoc shipped in OpenBSD's base system" — 3,500 pages — and self-declared done "once I was seeing parse errors on just over a dozen of the 3500 man pages," alongside unit tests, command tests, and cases extracted from mandoc's own regression suite ([duckrabbit.tech](https://duckrabbit.tech/articles/mdoc.html)). A wild-corpus pass rate, not spec coverage alone, was the de facto bar.

### Apache Tika — 1TB harvested corpus + statistical (not byte-exact) regression via tika-eval

**Corpus.** Tika, PDFBox, and POI jointly maintain a shared regression corpus of ~3M files / ~1TB+ ("more than 1TB of documents from govdocs1 and from Common Crawl") hosted on a dedicated ASF VM at `https://corpora.tika.apache.org/base/` ([TikaEvalOnVM wiki](https://cwiki.apache.org/confluence/display/tika/TikaEvalOnVM)). Sources: the govdocs1 corpus, Common Crawl pulls, and files harvested from bug trackers; the corpus deliberately skews toward "more diverse and more recent file types" than govdocs1 alone ([ASF LEGAL-696](https://issues.apache.org/jira/browse/LEGAL-696), [ApacheCon 2017 tika-eval slides](https://events.static.linuxfound.org/sites/events/files/slides/ApacheConMiami2017_tallison_v1.pdf)). The corpus lives entirely outside the source repo; access for full runs is "committers/PMC members with access to the VM."

**Release-gating workflow** ([TikaEvalOnVM](https://cwiki.apache.org/confluence/display/tika/TikaEvalOnVM)): "Before a release, we'll run the last release against the candidate release to identify potential regressions." A driver script (`appBatchExecutor.sh`) runs version A (current release) and version B (candidate) over the corpus into separate extract directories, then `tika-eval-app Compare` loads both into an H2 database and `Report` emits Excel workbooks.

**What tika-eval measures** ([TikaEval wiki](https://cwiki.apache.org/confluence/display/TIKA/TikaEval)). The key insight: at 3M heterogeneous files you cannot do byte-exact goldens, so the comparison is *statistical and categorical*:
- exceptions — "how many and of what category? Are these regular catchable exceptions, evil OOMs or permahangs?"
- extracted-text length deltas; number of metadata values; number of embedded files/attachments found
- MIME detection distribution shifts between runs
- text *quality* proxies: counts of "common words" per language (junk/garbled text scores low) and automatic language identification sanity
- **Profile mode** (one run, corpus health stats) vs **Compare mode** (A/B between versions/tools).

**MIME registry.** Detection is data-driven via `tika-mimetypes.xml` in tika-core resources (globs + magic byte patterns + hierarchy), covering 1,000+ types; users/downstreams extend it by overlaying `org/apache/tika/mime/custom-mimetypes.xml` on the classpath, so the registry itself is a maintained, separately-extensible artifact rather than code ([tika-mimetypes.xml in repo](https://github.com/apache/tika/blob/main/tika-core/src/main/resources/org/apache/tika/mime/tika-mimetypes.xml), [MimeTypes API](https://tika.apache.org/3.0.0/api/org/apache/tika/mime/MimeTypes.html)).

### LibreOffice — bug-attachment harvesting + crash/validity testing at ~100k-document scale

**Corpus = the bug tracker.** The corpus is harvested by `bin/get-bugzilla-attachments-by-mimetype` in the core repo, which "digs through a pile of bugzilla's and populates the current working directory with a collection of bug-docs, organised into per-filetype directories with bug-ids as names with prefixes to indicate which bug-tracker" (fdo, rhbz, moz, …); downloads are idempotent (existing files skipped) and filtered by MIME type from HTTP headers ([script](https://github.com/LibreOffice/core/blob/master/bin/get-bugzilla-attachments-by-mimetype), [moggi 2013](https://mmohrhard.wordpress.com/2013/04/19/automated-import-crash-testing-in-libreoffice/)). Every document that ever crashed or misrendered for a user becomes permanent regression ammunition. Scale: ~25,000 docs at first (2013) growing to "about 54000 documents in many more formats" by March 2014 ([moggi 2014](https://mmohrhard.wordpress.com/2014/03/24/update-about-our-importexport-crash-testing/)); it has kept growing since (formats include cdr, doc(x), fod*, od*, ppt(x), pub, rtf, vdx, vsd, wpd, xls(x)).

**What a run does** ([moggi 2014](https://mmohrhard.wordpress.com/2014/03/24/update-about-our-importexport-crash-testing/)):
1. Import every document headlessly; record crashes/hangs.
2. "Every document that we successfully import is now exported to a number of formats depending on the application that opens it" — i.e., systematic re-export fan-out (~120,000 exported documents, ~90GB per run; full run ≈ 3 days on a TDF server).
3. Exported files are run through **external validators** — officeotron for OOXML and ODF Validator for ODF — and "a validation error in the exported files will generate a test failure." Crash-freedom is not the only oracle; spec validity of *output* is enforced by independent implementations.

**Publication and workflow.** Per-commit-range results are published under [dev-builds.libreoffice.org/crashtest/](https://dev-builds.libreoffice.org/crashtest/) (directories keyed by git hash; an ASan variant exists at [crashtest_asan](https://dev-builds.libreoffice.org/crashtest_asan/)); the [Crashtesting wiki page](https://wiki.documentfoundation.org/Development/Crashtesting) tracks each analysis and links every found issue to Bugzilla bugs and Gerrit fixes. The 2024 dev blog confirms the practice is alive and extended to crash-report-driven test writing ([documentfoundation.org dev blog, 2024-04-18](https://dev.blog.documentfoundation.org/2024/04/18/crash-fixes-part-3-testing-crashes/)).

### FFmpeg FATE — committed reference hashes, out-of-tree append-only samples, volunteer client fleet

**Split storage.** Tests and small reference outputs live in-tree (`tests/ref/fate/`); the large binary samples live in the out-of-tree **fate-suite**, synchronized via rsync (`make fate-rsync SAMPLES=...`; path configured with `--samples=` or `FATE_SAMPLES`) and recommended to be cron-refreshed about daily ([ffmpeg.org/fate.html](https://ffmpeg.org/fate.html), [fate.texi](https://github.com/FFmpeg/FFmpeg/blob/master/doc/fate.texi), [Running FATE — multimedia.cx wiki](https://wiki.multimedia.cx/index.php/Running_FATE)).

**Comparison model.** Tests are make targets that run a decode/encode command and compare against committed reference output — exact diff of framecrc/MD5-style digests for bit-exact codecs, with tolerance comparators (one-off/stddev) where float math makes outputs platform-dependent. `GEN=1` regenerates "the missing or mismatched references" — the same accept-and-commit ergonomic as pandoc's `--accept` and tree-sitter's `-u` ([fate.texi](https://github.com/FFmpeg/FFmpeg/blob/master/doc/fate.texi)).

**Sample governance (append-only)** ([fate.texi](https://github.com/FFmpeg/FFmpeg/blob/master/doc/fate.texi)):
- "please make sure they are as small as possible" — minimize before upload; bandwidth and client disk are shared costs.
- "generally do not replace, remove or overwrite files" because "older checkouts use existing sample files" — the suite is effectively immutable/append-only so historical revisions remain testable.
- Samples are uploaded ~24h before the patch lands; uploads go through `samples-request`, frequent contributors get accounts via ffmpeg-devel.

**Distributed CI.** FATE is also a reporting network: volunteers on many architectures/OSes run `tests/fate.sh` (checkout → build → test → submit over SSH) and results aggregate at [fate.ffmpeg.org](https://ffmpeg.org/fate.html), giving cross-platform regression visibility no single CI could.

### ExifTool — solo-maintainer format sprawl sustained by community samples + a self-contained golden suite

**Scale and cadence.** One maintainer (Phil Harvey) supports hundreds of file types and metadata formats (EXIF, GPS, IPTC, XMP, JFIF, GeoTIFF, ICC, Photoshop IRB, ID3, …) plus maker notes for 30+ camera brands; current version 13.59 (May 27, 2026), with near-weekly development releases for ~20 years ([exiftool.org](https://exiftool.org/)).

**Upstream change tracking is crowdsourced.** New camera models/firmware are decoded from user-submitted files: "ExifTool is very much a collaborative effort... development relies heavily on the input from camera owners to help decode new meta information... If you manage to figure out what any of it means, send me an e-mail and I'll add your new discoveries" ([exiftool.org](https://exiftool.org/), [Wikipedia](https://en.wikipedia.org/wiki/ExifTool)). The maintainer is the integrator of a distributed reverse-engineering effort, and the site advertises "tested with images from thousands of different camera models."

**Test suite.** The Perl distribution ships a full golden suite runnable on any user machine via `make test`: tiny sample files in `t/images/`, per-module test scripts in `t/`, and committed expected outputs (`TESTNAME_#.out`); `t/TestLib.pm` provides `testCompare()` (diff actual vs. stored `.out`) and `writeCheck()` (write tags, re-read, verify) — so both read and write paths are golden-tested per format ([t/TestLib.pm](http://web.mit.edu/graphics/src/Image-ExifTool-6.99/t/TestLib.pm), [exiftool repo](https://github.com/exiftool/exiftool)). Note the corpus discipline: test images are deliberately minimized/truncated so the entire regression suite ships inside the tarball.

### tree-sitter — a corpus DSL with update tooling and guard rails

Grammar repos keep tests in `test/corpus/*.txt`, a plain-text DSL: test name between `===` lines, source code, a `---` divider, then the expected parse tree as an S-expression ([Writing Tests](https://tree-sitter.github.io/tree-sitter/creating-parsers/5-writing-tests.html)). Notable mechanisms:
- **Header attributes** parameterize cases: `:skip`, `:error` (assert input is invalid without spelling out the tree), `:fail-fast`, `:language(LANG)` for multi-parser repos, `:platform(...)`; `:cst` switches the expected form.
- **Escaping by lengthening**: if the content itself contains `===`/`---`, you suffix or lengthen the delimiters — the format stays writable for any language.
- **Update flag with guard rails**: `tree-sitter test -u` rewrites expected trees from current parser output, but tests containing `ERROR`/`MISSING` nodes are *not* auto-updated — error-recovery expectations can't be silently clobbered by accept-mode ([docs source](https://github.com/tree-sitter/tree-sitter/blob/master/docs/src/creating-parsers/5-writing-tests.md)).
- Guidance is to "be comprehensive... test all the permutations of each language construct"; each grammar repo owns its corpus (the ecosystem is federated — core ships the runner, formats live in their own repos).

### unified / remark / micromark — formats-as-plugins with a spec-compliant minimal core

The unified collective ([governance announcement](https://dev.to/murderlon/crowdsourcing-the-evolution-of-text-parsing-with-unified-5ao1)) splits the problem into layers: **micromark** is a "small, safe... commonmark compliant" tokenizer validated against the CommonMark spec test suite, and *every* syntax extension (GFM, MDX, frontmatter, math, directives) is a separate `micromark-extension-*` + `mdast-util-*` + `remark-*` package rather than a core flag ([micromark repo](https://github.com/micromark/micromark), [remark](https://github.com/remarkjs/remark), [plugin list](https://github.com/remarkjs/remark/blob/main/doc/plugins.md)). Consequences:
- The core's regression surface is the external spec's test corpus; dialect drift is contained in versioned extension packages (150+ remark plugins, in-org "official" vs community-maintained).
- Cross-format conversion is itself a plugin (`remark-rehype` bridges mdast→hast), mirroring pandoc's AST-pivot but with the pivot points packaged independently ([remark-rehype](https://github.com/remarkjs/remark-rehype)).
- Sustainability is explicit: Open Collective funding, an org-level governance/maintenance system created specifically "to reduce the number of open issues" across hundreds of packages ([dev.to](https://dev.to/murderlon/crowdsourcing-the-evolution-of-text-parsing-with-unified-5ao1)).

### Cross-project patterns (synthesis)

1. **Two-tier corpora everywhere.** Small, minimized, *committed* golden fixtures (pandoc `test/`, ffmpeg `tests/ref/fate/`, exiftool `t/images`, tree-sitter `test/corpus`) + a huge, *out-of-repo*, harvested wild corpus for release gating (Tika's 1TB VM, LibreOffice's ~100k bugzilla docs, ffmpeg's rsync fate-suite). No project puts the big corpus in git; all make it append-only or idempotently re-harvestable.
2. **Three regression oracles, used in combination:** (a) byte/hash-exact golden diff (pandoc, FATE, exiftool); (b) external spec validators on *written output* (LibreOffice's officeotron + ODF Validator); (c) statistical A/B profiling where exactness is impossible at scale (tika-eval's exceptions/text-length/common-words/langid deltas).
3. **The bug→corpus flywheel.** LibreOffice harvests every bugzilla attachment; Tika harvests bug-tracker files into its corpus; pandoc names command tests after issue numbers. A fixed bug that doesn't leave a corpus artifact is considered unfinished.
4. **Accept-mode ergonomics with guard rails.** `--accept` (pandoc), `GEN=1` (FATE), `-u` (tree-sitter) make intentional golden churn cheap — and tree-sitter shows the refinement: never auto-update error-expectation tests.
5. **Acceptance bar = maintainer burden, relieved by an extension tier.** Pandoc (Lua custom readers/writers, library-first), unified (everything's a plugin), tree-sitter (grammars in separate repos) all route "new format energy" through a plugin tier first; core adoption requires tests, a reviewable patch, and an implicit wild-corpus pass rate (mdoc: 3,500 real pages, ~dozen failures tolerated).
6. **Upstream format-change tracking is mostly inbound, not polled:** ExifTool's user-sample mail loop, LibreOffice's bugzilla harvest, Tika's MIME-registry patches. The corpus *is* the change-detection sensor — new wild files that fail are how a format's evolution is noticed.

## Design implications for neokapi

- **Add a second corpus tier above spec.yaml.** Keep `spec.yaml` (examples + assertions + spec citations) as the committed golden tier; create an out-of-repo **wild corpus** per format (harvested real-world files), hosted append-only the way docs assets already are (GitHub release / object storage, rsync-or-fetch into `$KAPI_CACHE`), never in git. FATE's rules apply verbatim: minimize files before adding, never mutate or delete, fetch via a `make fetch-format-corpus` target.
- **Institutionalize the bug→corpus flywheel.** Every format bug fix must land (a) a minimized failing document into the corpus and (b) a spec.yaml case named with the issue number (pandoc's `5474-tables.md` pattern). Make the format-triage skill check this invariant. A `kapi`-equivalent of `get-bugzilla-attachments-by-mimetype` could harvest attachments from neokapi's own GitHub issues by extension, idempotently.
- **Build a tika-eval-style A/B profiler for release gating.** Byte-exact goldens don't scale to thousands of wild files. Run previous release vs. candidate over the wild corpus and diff *statistics per format*: parse failures/panics by category (catchable error vs. hang vs. OOM), block/run/word counts, roundtrip byte-equality rate, detection (format-ID) distribution shifts, extracted-text common-words score. Gate the release on "no unexplained deltas," emit a per-format report — this naturally extends the existing /format-maturity dashboard.
- **Adopt LibreOffice's dual oracle for writers: crash-freedom + independent validators.** For every format with an external validator (XLIFF schemas, JSON Schema, xmllint/RelaxNG for ODF-ish XML, officeotron-class OOXML validation, gettext `msgfmt -c` for PO), run the validator over *neokapi-written* output across the whole corpus; a validation error is a test failure even when the roundtrip "succeeded." This is orthogonal to the Okapi parity harness (parity says "same as Okapi"; validators say "actually legal per spec").
- **Make the maturity rubric the acceptance bar, pandoc-style.** Codify: a new format enters as L0 only with committed golden tests ("you must include tests"); promotion to higher levels requires a wild-corpus pass-rate threshold (the mdoc precedent: ~99.7% of 3,500 real files). Publish the threshold per level so format adoption is self-service and burden-aware — the rubric answers jgm's "who answers the bug reports" question with measurable exit criteria instead of maintainer vibes.
- **Keep the plugin tier as the pressure valve.** Pandoc's Lua readers and unified's plugin-everything model both confirm the existing neokapi stance (configured JSON/YAML readers and subprocess plugins before bespoke core formats). Document explicitly: new-format requests get routed to (1) configured generic reader, (2) plugin, (3) core — core only with spec.yaml + corpus + maturity target, mirroring pandoc's "standalone library first, core later if review passes."
- **Accept-mode with guard rails.** neokapi's golden/spec assertions should have a single `-update`/accept flow (pandoc `--accept`, FATE `GEN=1`, tree-sitter `-u`) — but copy tree-sitter's exception: cases asserting *error handling or lossy/malformed behavior* (the `malformed_test` gap, 45/48 formats) must never be auto-updated, only hand-edited.
- **Treat format detection as a maintained data artifact.** Tika's `tika-mimetypes.xml` + `custom-mimetypes.xml` overlay shows detection (globs + magic + priority) living as testable data with a user extension point — neokapi's detection-collision rules should be serialized similarly and regression-tested as their own suite, with detection-distribution drift checked in the A/B profiler.
- **Statistical corpus health ("profile mode") even without an A/B pair.** tika-eval's single-run profile (exception census, junk-text scoring, langid sanity) is a cheap nightly: run the current build over the wild corpus and chart per-format health on the dashboard; regressions show up as trend breaks between releases.
- **Per-commit-range publication.** LibreOffice publishes crashtest results keyed by git hash with wiki triage linking each finding to a bug + fix. For a solo maintainer, the equivalent is: nightly corpus run → JSON delta artifact keyed by commit → dashboard panel + auto-filed GitHub issues for new failure signatures (the self-improving loop: Claude can triage the delta).
- **Crowdsource upstream format drift, ExifTool-style.** For long-tail formats (subtitle dialects, vendor JSON/ARB variants, CAT-tool XLIFF flavors), the cheapest change-tracking sensor is inbound samples: a documented "submit a file that kapi mishandles" path (issue template that asks for the file + expected behavior) feeding directly into the corpus + spec.yaml flywheel.
