
# neokapi Framework + Core — Senior Code Review

## Executive summary

The neokapi framework and shared modules are architecturally sound: the streaming content model, capability-typed tool views, functional-options executor, centralized resilient HTTP client, hardened plugin installer, and parameterized SQL are all idiomatic and well-considered. The defects that surface are concentrated, not pervasive, and cluster around three structural roots: (1) the per-block tool dispatch path drops `context.Context`, forcing every AI/MT/external tool into `context.Background()` or a ctx-in-struct workaround and breaking mid-block cancellation; (2) several streaming output interceptors and gRPC plugin readers perform unguarded channel sends that deadlock or leak goroutines on the cancel/error path; and (3) viper config layering is mis-wired so the env-var precedence layer is silently unreachable. A 859-line proto bridge and a hand-written Run-union mapping (triplicated) have no round-trip tests, and the lint job silently skips all cgo/fts5/parity-tagged code. None are exploitable from untrusted network input, but the concurrency and context defects are real correctness bugs under cancellation/timeout, and the testing gaps mask exactly the silent-corruption hotspots discovery flagged.

## Top priorities

These are the findings that genuinely matter most — correctness under cancellation, goroutine leaks, a security DoS surface, and untested silent-corruption hotspots.

1. **`core/tool/base.go:171` — tool dispatch drops `ctx`, so no per-block handler can honor cancellation.** `Process(ctx,…)` only selects on `ctx.Done()` *between* parts; the per-block dispatch path (`dispatch`→`handleBlock`→`runTranslate/runAnnotate/runTransform`) and the capability handlers (`Annotate(BlockView)`, `Translate(TargetView)`, `Transform(SourceView)`) carry no ctx. This is the architectural root cause of the `context.Background()` and ctx-in-struct workarounds across `core/ai/tools`, `core/mt/tools`, `core/tools/redact.go`, `core/tools/externalcommand.go`, `core/tools/segmentation.go`. **Fix:** thread ctx into the dispatch path and add it to the handler signatures (`Annotate(ctx, BlockView)` …) or expose `v.Context()`. `ParallelBlockTool.Process` already has ctx in scope and can pass it straight through. This unblocks ~10 dependent context findings below.

2. **`core/flow/metrics.go:167` and `core/flow/trace.go:345` — output interceptors deadlock the pipeline on a downstream error.** Both spawn an output interceptor that does a bare `out <- part` with no `select{ case <-ctx.Done() }`, while the matching *input* interceptor does guard. When a downstream tool errors, the errgroup cancels ctx and the downstream stops draining; the interceptor blocks forever, `interceptors.Wait()` never returns, and the whole run hangs. `kapi run <flow> --trace` is exactly the path that wraps tools in `TracingTool`. **Fix (mechanical, autofix):** `select { case out <- part: case <-ctx.Done(): return }` at both sites.

3. **`core/ai/tools/translate.go:357` — orphaned `translated`-channel goroutine leaks one goroutine per session-backed translate call.** Nothing ever sends to `translated`; the real path uses `batchOut`. The drainer goroutine blocks forever on `for part := range translated`. The trailing `_ = filterErr` (line 401) confirms a half-finished refactor. **Fix:** delete the dead channel + drainer (lines 319, 356–362); either propagate `filterErr` or remove it. Add a goleak test around `processBatchedWithSession`.

4. **`cli/pluginhost/exec.go:195` and `cli/pluginhost/contributions.go:203` — `os.Exit()` inside `RunE` leaks Mode-C daemons and skips `App.Shutdown`.** `runSubprocess`/`runContributionSubprocess` call `os.Exit(exitErr.ExitCode())` directly, bypassing `PersistentPostRun` (app.Shutdown), the `cli.Run` cleanup func, and `DaemonClient.close` (which SIGTERMs the daemon pool and removes the unix socket). Any plugin invocation that spawned a daemon and then exits non-zero orphans the process and leaks its socket. The in-code comment claiming cobra surfaces the code "through SilenceErrors" is false for `os.Exit`. **Fix:** `return cli.WithExitCode(exitErr.ExitCode(), err)` at both sites — the machinery in `cli/exitcode.go` exists for exactly this.

5. **`cli/pluginhost/registry/index.go:82,179` — unbounded `io.ReadAll` on registry index + plugin artifact downloads (memory-exhaustion DoS).** Both reads are uncapped against a config-supplied URL (`$KAPI_REGISTRY_URL`); a compromised/redirected registry can OOM the client. SHA256/cosign verification happens *after* the full read, so it offers no protection. The cosign path already caps via `io.LimitReader(resp.Body, maxBundleBytes+1)` (`cosign.go:219`). **Fix:** wrap both in `io.LimitReader` with `maxArtifactBytes`/`maxIndexBytes`; reject on `Content-Length` over cap before reading.

6. **`core/plugin/protoconvert/protoconvert.go` — the 859-line model↔proto bridge has zero tests.** 40 hand-written bidirectional converters (Run union, Block targets/overlays, skeleton) with no round-trip coverage; discovery flags this as the spot where "model drift silently corrupts all Java-filter outputs." **Fix:** add `protoconvert_test.go` with table-driven `ProtoToX(XToProto(v)) == v` per Run variant + Blocks with multiple targets/overlays/segments, plus `FuzzRunRoundTrip`/`FuzzBlockRoundTrip`.

7. **`cli/config/config.go:32` and `:59` — viper config layering is inverted/unreachable.** (a) `AutomaticEnv()` without `SetEnvKeyReplacer(strings.NewReplacer(".","_"))` makes every dotted key env-unreachable (`flow.channelBuffer`, `plugins.directory`, `plugins.registry`, `formats.priorities`); only flat keys like `language` resolve. (b) `NewOverlayAppConfig` copies the overlay file in via `cfg.v.Set(key,…)` — viper's *top* precedence layer — so a key present in `bowrain.yaml` becomes unoverridable by `BOWRAIN_SERVER_URL`, inverting the expected flag>env>file order. **Fix:** add the env-key replacer before `AutomaticEnv()`; merge the overlay via `MergeConfigMap(overlay.AllSettings())` and reserve `Set()` for true CLI overrides.

## Architecture & encapsulation

- **`bowrain/core/store/store.go:12`** — `ContentStore` is a ~60-method god interface spanning eight concerns (project, stream, collection, item, block, version, notes, history, change-log/sync, asset). Every consumer and test double must satisfy all 60 to use one slice. **Fix:** split into role interfaces (`ProjectStore`, `StreamStore`, `BlockStore`, `VersionStore`, `AssetStore`, `ChangeFeed`, `CollectionStore`) and compose them into one umbrella type the concrete server impl still satisfies; let handlers/tests depend on the narrow role.
- **`sievepen/tm_store.go:14`** — `TMStore` embeds `TranslationMemory` and adds ~20 more methods across three responsibilities (faceted search/stats, bulk export, import-session CRUD). `cli/app.go:56` types `TMBackend sievepen.TMStore`, coupling the whole CLI to all three. **Fix:** segregate `TMSearcher`/`FacetProvider` and `ImportSessionStore`; let callers depend on the narrowest (flow leverage only needs `TranslationMemory`).
- **`bowrain/core/client/client.go:460`** *(also flagged by design-patterns, context)* — package-level REST helpers (`CreateAnonymousProject`, `CreateAuthenticatedProject`, `ListWorkspaces`, `GetProject`, `DeleteProject`, `ClaimProject`, `JoinWorkspace`, `CreateWorkspace`, `DownloadBlob`) hardcode `http.DefaultClient`/`http.Post`, bypassing the injectable `httpClient` and (mostly) dropping context. No test seam without a live listener; no caller timeout/transport honored. `auth/device.go` in the same module already shows the right pattern (injectable `HTTPClient` field). **Fix:** move these onto a constructor-injected facade carrying `httpClient *http.Client` defaulting to `http.DefaultClient`, thread `context.Context` via `http.NewRequestWithContext`.
- **`bowrain/core/client/tokens.go:42`** — token-management helpers (`CreateToken:56`, `ListTokens:83`, `DeleteToken:110`) repeat the same `http.DefaultClient` + `http.NewRequest` (no context) pattern in a security-relevant file; no injectable boundary. **Fix:** fold into the same injected-client facade; short-term add an overridable package-level `httpClient` var.
- **`cli/app.go:243`** *(cobra lens)* — `App.Init`/`InitPluginHost`/`InitRegistries` return no error, so `a.Config.Load()`'s error is discarded (`_ =`, line 264) and the only callers must use the non-E `PersistentPreRun` (`kapi/cmd/kapi/root.go:22`, `cli/toolbox.go:90`). A malformed `--config` proceeds with a half-initialized App. **Fix:** change `Init() error`, return the load/credential/discovery error, and wire `PersistentPreRunE`.
- **`sievepen/tm_store.go:33`** *(code-style lens)* — `SearchEntriesForStream(query, anyLocale, requireLocale, stream string, …)` presents four adjacent string params (mirrored in `sievepen/sqlite.go:1310`, `memory.go:403`, `termbase/sqlite.go:447`); positional call sites make a query/locale transposition compile cleanly and fail silently. **Fix:** introduce a `SearchParams` struct.

## Simplicity & duplication

- **`bowrain/core/client/sync_convert.go:266`** — the 7-arm Run-union mapping (Text/Ph/PcOpen/PcClose/Sub/Plural/Select with recursive Plural.Forms/Select.Cases) is hand-written three times: `protoconvert.RunToProto/ProtoToRun` (`protoconvert.go:229`), `client/sync_convert.go:266`, and `sync/convert.go:261`, each targeting a different wire format. A new Run kind requires edits in all three — the same drift hazard called out at the protoconvert hotspot. **Fix:** extract the recursive Plural/Select walk into one shared generic helper at minimum; ideally parameterize the 7-arm switch by a target factory/interface so the body is written once. Pairs with finding 6 above (add round-trip tests once consolidated).

## Concurrency & context

(Most of the per-tool context findings below collapse once the `core/tool/base.go:171` root fix lands — they are reported individually so each call site is tracked.)

- **`core/flow/metrics.go:167`**, **`core/flow/trace.go:345`** — output-interceptor deadlock; see Top priority 2. *(autofix)*
- **`core/ai/tools/translate.go:357`** — leaked `translated`-channel drainer; see Top priority 3.
- **`cli/pluginhost/format_client.go:142`** *(safety lens)* — after `ctx.Done()`, each emit site does an unconditional blocking send `ch <- model.PartResult{Error: ctx.Err()}` on the unbuffered `ch` (lines 142/150/159). The downstream consumer has already stopped ranging, so the send blocks forever and the reader goroutine + open gRPC stream leak for the process lifetime — fires on every fail-fast-aborted multi-document run. **Fix:** drop the post-cancel send (`return`), or use a buffered(1) channel / non-blocking select; the error is already surfaced via the cancelled context.
- **`cli/pluginhost/format_client.go:299`** *(gRPC lens)* — the Write send goroutine ranges over `parts` with no `ctx.Done()` select; on early recv-error return (line 319) `parts` is never drained and the goroutine parks until the caller closes the channel, which may never happen. **Fix:** cancel a child stream ctx on Write return and select on it in the send loop.
- **`core/tool/parallel.go:154`** *(safety lens)* — `Process` returns immediately on `sp.err` without cancelling its own ctx, leaving the dispatcher and in-flight workers blocked on the `concurrency*2`-buffered `results` channel; leak-freedom depends on a non-local external errgroup cancel. **Fix:** `ctx, cancel := context.WithCancel(ctx); defer cancel()` and `cancel()` before the error returns (154, 181), or drain `results`.
- **`core/ai/tools/translate.go:445,511`** — `translate()` passes `context.Background()` to `translateBlock` → `provider.Translate`/`ChatStream`, detaching the LLM call from the pipeline; a cancelled flow keeps spending tokens. **Fix:** plumb the handler ctx.
- **`core/mt/tools/translate.go:100,112`** — `provider.Translate(context.Background(), …)`; MT network call detached from cancellation. **Fix:** pass pipeline ctx.
- **`core/ai/tools/brandvoice.go:176,192`** — `ResolveProfile`/`ChatStructured` on `context.Background()`. **Fix:** thread ctx (same fix covers `qualitycheck.go:141`, `review.go:103`, `terminology.go:136`).
- **`core/ai/tools/entity_extract.go:124,134`** — `DetectEntities`/`extractWithLLM` on `context.Background()`. **Fix:** thread ctx.
- **`core/ai/tools/qualitycheck.go:141`**, **`review.go:103`**, **`terminology.go:136`** — three LLM handlers on `context.Background()`; same root + fix.
- **`core/tools/externalcommand.go:110`** — `context.WithTimeout(context.Background(), …)` feeds `exec.CommandContext`; a cancelled flow can't kill the child, only its own timeout. **Fix:** derive the timeout ctx from the pipeline ctx.
- **`core/tools/segmentation.go:171`** — `ctx context.Context` stored as a struct field, seeded with `context.Background()` (line 195), overwritten in `Process` (line 206), read from the annotate handler (255/274) — the canonical ctx-in-struct anti-pattern, racy if the tool is reused. **Fix:** remove the field; pass the handler ctx to `seg.Segment` once handlers carry ctx.
- **`termbase/sqlite.go:143`** *(database lens)* — the entire SQLite termbase uses non-context DB calls (`db.Begin`, `db.Query`, `db.QueryRow`, `tx.Exec` throughout), unlike its sibling `sievepen/sqlite.go`. Termbase lookup is on the concurrent flow hot path, so a cancelled run can't interrupt an in-flight `queryFuzzyFullScan LIMIT 500` + per-row Levenshtein scan. **Fix:** thread ctx and switch to `QueryContext`/`ExecContext`/`BeginTx` (interface change required).
- **`bowrain/core/client/client.go:540`** — `DownloadBlob` (and the package-level helpers) use `http.DefaultClient.Do` instead of the configured `c.httpClient`/`doRequest`: no timeout (hung server blocks forever), no retry, no 401-refresh. The critical Merkle-diff push path uses `doRequest` correctly; exposure is in admin/bootstrap + `DownloadBlob`. **Fix:** route everything through one `doRequest` bound to `c.httpClient` with a real timeout.
- **`bowrain/core/client/client.go:547`** — requests built with `http.NewRequest` (no context) + zero-timeout default client are uncancellable/undeadlinable. **Fix:** `http.NewRequestWithContext` with caller ctx (same consolidation as above).

## Error handling

- **`bowrain/core/sync/convert.go:132`** *(also `client/sync_convert.go:196–214`)* — sync converters blank-assign `json.Unmarshal` errors for optional extension fields and return no error; a malformed pull silently yields a zero `Skeleton`/`Annotations`/`DisplayHint`/`ContentRef`, corrupting round-trip. **Fix:** return and wrap the unmarshal error in both converters.
- **`termbase/sqlite.go:726`** *(database lens)* — `scanConcept` does `if err != nil { return c, nil }` on the term query, discarding a real DB error and returning a concept with zero terms; it's the building block for `GetConcept`/`Search`/`Concepts`/all match paths, so a transient error makes a populated concept appear empty. The per-row `continue` (line 735) also drops corrupt rows silently. **Fix:** `return c, fmt.Errorf("query terms for concept %s: %w", id, err)`; propagate/log per-row scan errors.

## Data & performance

- **`core/tools/searchreplace.go:195`** — `buildEffectivePairs` + `regexp.Compile` per regex pair, per Block, in `Transform` (per-block hot path). **Fix:** precompile pairs once in `NewSearchReplaceTool` (as `patterncheck.go`/`segmentation.go` already do).
- **`core/tools/qacheck.go:657`** — `regexp.Compile(p.Source)` (and the target pattern at 686) recompiled per block inside `Annotate`. **Fix:** precompile `conf.Patterns` into `{sourceRe,targetRe}` at construction; drop invalid patterns once.
- **`core/tools/tagprotect.go:104`** — `findProtectedTags` compiles the 4 fixed tag patterns on every block. **Fix:** precompile into `[]*regexp.Regexp` in `NewTagProtectTool`.
- **`core/model/run.go:405`** *(also flagged by data-structures + performance lenses)* — `flattenRunsTo`/`runsTextTo` accumulate into `*[]rune` (`append(*buf, []rune(r.Text.Text)...)` per text run, then `string(buf)`), a per-run alloc + final rune→string copy. The sibling `RenderRunsWithData` in the same file already uses `strings.Builder`, and `run_keys.go` helpers do too — this is the inconsistent slow one, on 71 per-block call sites (TM lookup, word count, QA, brand, segmentation). **Fix:** take a `*strings.Builder`, `WriteString` text and `WriteByte` the `{}[]` delimiters; return `b.String()`.
- **`core/tool/base.go:224`** — `EnforceImmutability` (on by default) renders+FNV-hashes full source and all targets twice per block per annotate tool via `RenderRunsWithData` + per-target `MarshalText`. **Fix:** hash runs directly into a reused FNV hasher (Reset between before/after) instead of materializing strings; hash target keys in stable order without the keys-slice+index-map.
- **`sievepen/sqlite.go:837`** — fuzzy TM scoring re-converts the fixed query keys to `[]rune` up to 6× per candidate (`LevenshteinRatio` converts both args, then `LevenshteinDistance` converts both again). **Fix:** convert query keys to `[]rune` once before the candidate loop; add a `[]rune`-based distance/ratio core.
- **`cli/pluginhost/format_client.go:91`** *(autofix)* — `daemonReader.Read` returns an unbuffered `make(chan model.PartResult)` while every native reader buffers at 64; each Part requires a full rendezvous, so the gRPC reader can't run ahead. Affects every doc parsed through a Mode-C plugin (all 57+ Java-bridge filters). **Fix:** `make(chan model.PartResult, 64)`.
- **`cli/pluginhost/daemon.go:678`** — `dialUnixSocket` sets no `Max*MsgSize`; inline `ContentRef_Inline` send and `Complete.Output` recv trip gRPC's 4MB default `ResourceExhausted` on whole-document streaming. **Fix:** `grpc.WithDefaultCallOptions(MaxCallRecvMsgSize/MaxCallSendMsgSize)` or chunk.

## Testing

- **`core/plugin/protoconvert/protoconvert.go`** — zero round-trip tests on the 859-line bridge; see Top priority 6.
- **`providers/mt/provider_test.go:95`** *(+ 158, 225, 280, 344)* — `t.Fatal("API should not be called…")` runs inside the httptest handler goroutine; `runtime.Goexit` only kills that goroutine, the client gets a truncated response, and the test continues (often passing). The "skip non-translatable blocks / never hit API" assertion isn't actually enforced. **Fix:** record hits in an atomic counter inside the handler and `require.Zero(t, hits)` on the test goroutine (or use `t.Errorf`).
- **`bowrain/core/client/anonymous_test.go:21`** *(testify lens)* — `assert.NoError(decode)` doesn't halt, then line 24 does an unchecked `req["target_languages"].([]any)`; on decode failure the type assertion panics on the handler goroutine and crashes the test binary. **Fix:** `targets, ok := …; if assert.True(t, ok) {…}` or decode into a typed struct.
- **`providers/ai/gemini_test.go:42`** *(testify lens)* — `assert.Len(req.Contents, 1)` (non-stopping) then `req.Contents[0].Role` unconditionally; empty `Contents` panics on the handler goroutine. **Fix:** `if assert.Len(t, req.Contents, 1) { … }`.
- **`core/ai/tools/tools_test.go:47`** *(+ 73, 100; also `core/tool/retry_test.go:34`, `core/mt/tools/translate_test.go:54`)* — `result := <-out` then `result.Resource.(*model.Block)` with no nil guard; if the tool drops the part, the receive returns a nil `*Part` and the dereference panics instead of failing the assertion. The mt `processPart` helper already does `require.NotNil`. **Fix:** add `require.NotNil(t, result)` (or route through a shared helper).
- **`sievepen/sqlite.go:1506`** — `FacetStatsFiltered` fans out 5 goroutines each swallowing query errors silently (`if rows, err := …; err == nil`), and no `TestSQLiteTM_*` exercises concurrent access (no `go func`/errgroup/WaitGroup in the sievepen sqlite tests), so the WAL-parallel-reader assumption and the fan-out never run under `-race`. **Fix:** add an errgroup-driven concurrent `Lookup`/`Search`/`FacetStatsFiltered` test run under `-race`; assert facets populate so a swallowed error is caught.
- **`core/flow/executor.go`** — the goroutine-per-tool executor (+ `streaming_collector.go`, `filerunner.go`, `batch.go`, `trace.go`, `metrics.go`) has no goleak detection; `go.uber.org/goleak` is absent from the framework go.mod and there's no `TestMain`. `TestFlowExecutorContextCancellation` asserts `wait()` errors but never that spawned goroutines terminate — it would pass even with the metrics/trace leaks above. **Fix:** add `goleak.VerifyTestMain(m)` in `core/flow` (and `sievepen`, `core/blockstore`); `defer goleak.VerifyNone(t)` in the cancellation test.

## Tooling & deps

- **`.golangci.yml:2`** *(lint lens)* — no `run.build-tags`, and the `lint-framework` CI job (`ci.yml:538`) installs no `libicu-dev`, so 100+ files behind `fts5`/`parity`/`cgo`/`onnx` constraints — including the exact review hotspots `sievepen/sqlite.go`, `core/storage/icu_tokenizer.go`, `driver_cgo.go`, `core/storage/migrate.go`/`Open` — are silently excluded from staticcheck/govet/errcheck. **Fix:** set `run.build-tags: [fts5, parity]` (or pass `--build-tags`), install `libicu-dev` in `lint-framework` mirroring the test job, and run a second pass with the `parity` tag so both constraint branches are analyzed.
- **`.github/workflows/security.yml:4`** *(deps lens)* — govulncheck runs only on a weekly `schedule` + `workflow_dispatch`; no `pull_request`/`push` trigger, so a PR introducing a newly-vulnerable dependency merges and the CVE surfaces up to a week later. **Fix:** add `pull_request:` (and optionally `push: branches:[main]`), scoping to touched modules to keep latency low.
- **`kapi/go.mod:129`** *(deps lens)* — four `bowrain*` replace directives with no corresponding `require` entries (dead no-ops). Worse than harmless: if a bowrain import is ever accidentally added, the replace silently resolves it against local sibling source instead of failing `GOWORK=off`, masking the very isolation-contract violation it should expose. **Fix:** remove the four bowrain replaces (keep only `=> ../` and `.../cli => ../cli`).
- **`.github/dependabot.yml:1`** *(low, deps lens)* — neither dependabot nor the govulncheck matrix covers `bowrain/plugin` or `bowrain/plugin/schema`, though `schema` feeds directly into in-scope `bowrain/core` and kapi-desktop. **Fix:** add gomod entries for both modules to dependabot.yml and the security.yml matrix.

## Coverage & method

**Lenses applied** (cc-skills-golang, all three in-scope modules target Go 1.26.0): code-style, naming, modernize, design-patterns, structs-interfaces, project-layout, dependency-injection, safety, security, error-handling, concurrency, context, lint, deps, cobra, viper, cli, grpc, database, testing, testify, performance, data-structures. The framework baseline is strong — the functional-options executor, resilient HTTP client centralization, hardened plugin tar/zip installer (safeJoin/O_NOFOLLOW/cosign), parameterized SQL + FTS5 escaping, capability-typed views, and OS-keychain credential handling were all examined and confirmed sound (explicit non-findings recorded per lens).

**Gated out / not run:** none of the lenses were blocked by absent libraries; the codebase uses no DI framework (wire/dig/fx), samber/* helpers, or alternate logging stacks, so those skills had no surface. goleak is *absent* (reported as a testing finding, not a gate).

**Scope not covered (honest limits):** the review did not build the full cgo/fts5/okapi-bridge stack (used `go list -tags fts5 -deps` for the import graph instead); the main `bowrain` server module, `kapi`, and `apps/kapi-desktop` were out of scope (JWT-secret sourcing and content-store HTTP handlers live there); the 49 format packages were sampled (openxml/idml/rtf/srt/json), not exhaustively read field-by-field; generated `*.pb.go`, `*_test.go` (except where a test was itself the finding), `bench/`, `examples/`, and the SQLite amalgamation were excluded per the scope's vendored/generated globs. `cli/parity/**` and `harness/` tooling were reviewed only in aggregate. golangci-lint was not executed (and, per the first tooling finding, would not have covered the cgo/fts5 hotspots even if it had been).
