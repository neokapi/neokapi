# Keycloak Custom Theme with Keycloakify v11

This note documents the custom Keycloak login theme built with Keycloakify v11. The theme replaces the default PatternFly-based login pages with glassmorphism-styled React pages that match the Bowrain application.

## Project Structure

The theme lives in `bowrain/apps/keycloak-theme/`:

```
bowrain/apps/keycloak-theme/
├── package.json                # Keycloakify v11 + React 19 + Tailwind v4
├── vite.config.ts              # Vite + React + Tailwind + Keycloakify plugin
├── tsconfig.json
├── src/
│   ├── main.tsx                # React entrypoint with ThemeProvider
│   ├── kc.gen.tsx              # Auto-generated by `keycloakify update-kc-gen`
│   ├── vite-env.d.ts
│   └── login/
│       ├── KcContext.ts        # Keycloak context type extensions
│       ├── KcPage.tsx          # Page router (login, register, error, default)
│       ├── KcPageStory.tsx     # Storybook mock for dev testing
│       ├── i18n.ts             # i18n customization
│       ├── main.css            # Theme CSS with OKLCH token bridge
│       └── pages/
│           ├── Login.tsx       # Custom login page
│           ├── Register.tsx    # Custom registration page
│           └── Error.tsx       # Custom error page
└── dist_keycloak/              # Built JAR (gitignored)
    └── keycloak-theme-for-kc-all-other-versions.jar
```

## Build Pipeline

The theme is built in two stages via the `build-keycloak-theme` npm script:

```bash
npm run build-keycloak-theme
# Equivalent to: tsc && vite build && keycloakify build
```

This produces a JAR file in `dist_keycloak/` that Keycloak loads as a provider. The Makefile provides:

```makefile
keycloak-theme: ui-deps  ## Build Keycloak login theme JAR
	cd $(KC_THEME_DIR) && $(NPM) ci && $(NPM) run build-keycloak-theme
```

Note that `keycloak-theme` depends on `ui-deps` because the theme imports components from `@neokapi/ui`.

## Vite Configuration

```ts
export default defineConfig({
  plugins: [
    react(),
    tailwindcss(),
    keycloakify({
      themeName: "bowrain",
      accountThemeImplementation: "none",
    }),
  ],
  resolve: {
    alias: {
      "@neokapi/ui": path.resolve(__dirname, "../../../packages/ui/src"),
    },
  },
});
```

Key points:
- `themeName: "bowrain"` -- the theme is registered in Keycloak under this name
- `accountThemeImplementation: "none"` -- only the login theme is customized; the account management pages use Keycloak defaults
- The `@neokapi/ui` alias points to the shared UI package source for direct TypeScript consumption

## doUseDefaultCss=false

A critical configuration in `KcPage.tsx` prevents PatternFly CSS from loading:

```tsx
<DefaultPage
    kcContext={kcContext}
    i18n={i18n}
    classes={classes}
    Template={Template}
    doUseDefaultCss={false}
    UserProfileFormFields={UserProfileFormFields}
    doMakeUserConfirmPassword={true}
/>
```

Setting `doUseDefaultCss={false}` is essential because:
- Keycloak's default theme loads PatternFly CSS, which conflicts with Tailwind utility classes
- PatternFly's reset styles override Tailwind's base styles
- The glassmorphism effects (backdrop-filter, alpha backgrounds) break under PatternFly's opaque surface styles

The custom pages (Login, Register, Error) never load PatternFly at all. The `doUseDefaultCss={false}` flag is only relevant for the `DefaultPage` fallback that handles uncustomized page types.

## Custom Pages

### Page Router (KcPage.tsx)

The router dispatches by `kcContext.pageId`:

```tsx
switch (kcContext.pageId) {
    case "login.ftl":    return <Login ... />;
    case "register.ftl": return <Register ... />;
    case "error.ftl":    return <ErrorPage ... />;
    default:             return <DefaultPage doUseDefaultCss={false} ... />;
}
```

All pages render inside an `AnimatedBackgroundGlass` component for the floating-orb glassmorphism background.

### Login Page

Uses `@neokapi/ui` components (`Card`, `Button`, `Input`, `Label`) with the `glass-surface` class for backdrop blur. Features:
- Email/username + password form
- "Remember me" checkbox (conditional on realm config)
- Password reset link
- Social login provider icons (GitHub, Google, Microsoft, Apple) rendered from `social.providers` in the Keycloak context
- A Bowrain logo SVG above the card

### Register Page

Dynamic form fields rendered from `profile.attributesByName` (Keycloak's declarative user profile). Password fields are handled separately with explicit confirm. Field icons are mapped by attribute name (`username` -> User icon, `email` -> Mail icon, etc.).

### Error Page

Minimal card displaying the error message with an optional "Back to Application" button when `client.baseUrl` is available.

## CSS Token Bridge

Keycloakify's CSS processing can strip or reorder `@layer` blocks, which causes `shadcn-glass-ui` tokens (defined inside layers) to be lost. The `main.css` file works around this by re-declaring all required tokens in `:root` outside any layer:

```css
@import "@neokapi/ui/styles/globals.css";
@source "../../../../../node_modules/shadcn-glass-ui/dist";

:root {
  /* OKLCH primitives (~80 tokens: white opacity variants, black, color ramps) */
  --oklch-white-8: oklch(100% 0 0 / 0.08);
  --oklch-copper-500: oklch(62% .16 48);
  --oklch-copper-600: oklch(54% .16 46);
  --oklch-bronze-500: oklch(58% .14 35);
  /* ... */

  /* Semantic tokens (~45 tokens) */
  --semantic-primary: var(--oklch-copper-500);
  --semantic-secondary: var(--oklch-bronze-500);
  --semantic-surface: var(--oklch-white-8);
  /* ... */

  /* Component tokens (~35 tokens: card, input, button, text) */
  --card-bg: var(--semantic-surface-muted);
  --btn-primary-bg: linear-gradient(135deg, ...);
  /* ... */

  /* Glass-specific tokens (~20 tokens: blur, glow, shadow, duration, radius) */
  /* ... */

  /* Background gradient + orb tokens (5 orbs) */
  --bg-from: var(--oklch-slate-900);
  --orb-1: var(--oklch-blue-500-20);     /* hue 255° */
  --orb-2: var(--oklch-copper-600-20);   /* hue 46° */
  --orb-3: var(--oklch-cyan-500-20);     /* hue 195° */
  --orb-4: var(--oklch-copper-600-15);   /* hue 46° */
  --orb-5: transparent;                   /* center, dark theme only */
}
```

The file defines approximately 207 CSS custom properties in total.

The `@source` directive ensures Tailwind scans the `shadcn-glass-ui` dist directory for runtime class names like `pl-10` and `left-2.5` that components reference.

A Tailwind v4 specificity fix is also included for icon inputs:

```css
input.pl-10 {
  padding-inline-start: calc(var(--spacing) * 10);
}
```

This corrects a cascade issue where Tailwind v4's logical `px-4` shorthand overrides the physical `pl-10` longhand, causing text to overlap input icons.

## i18n Customization

Custom translations are added via Keycloakify's i18n builder in `login/i18n.ts`:

```ts
const { useI18n, ofTypeI18n } = i18nBuilder
    .withThemeName<ThemeName>()
    .withCustomTranslations({
        en: { backToLogin: "Back to Login", backToApplication: "Back to Application" },
        fr: { backToLogin: "Retour \u00e0 la connexion", ... },
        de: { backToLogin: "Zur\u00fcck zur Anmeldung", ... },
        es: { backToLogin: "Volver al inicio de sesi\u00f3n", ... },
    })
    .build();
```

Standard Keycloak message keys (`loginAccountTitle`, `password`, `doLogIn`, `doRegister`, etc.) are used unchanged. Only navigation labels specific to the custom pages are added.

## JAR Volume Mount in compose.yaml

The built theme JAR is mounted directly into the Keycloak container as a provider:

```yaml
keycloak:
  image: quay.io/keycloak/keycloak:26.1
  command: start-dev --import-realm
  volumes:
    - ./docker/keycloak/realm.json:/opt/keycloak/data/import/realm.json
    - ./bowrain/apps/keycloak-theme/dist_keycloak/keycloak-theme-for-kc-all-other-versions.jar:/opt/keycloak/providers/keycloak-theme.jar
```

The realm import (`realm.json`) activates the theme:

```json
{
  "realm": "bowrain",
  "loginTheme": "bowrain",
  ...
}
```

The JAR must be built before starting Docker Compose (`make keycloak-theme`). Keycloak detects the provider JAR at startup and registers the "bowrain" login theme.

## Identity Providers in realm.json

The development realm configures two social identity providers:

```json
"identityProviders": [
    {
        "alias": "google",
        "providerId": "google",
        "config": {
            "clientId": "placeholder-google-client-id",
            "clientSecret": "placeholder-google-client-secret",
            "defaultScope": "openid email profile"
        }
    },
    {
        "alias": "github",
        "providerId": "github",
        "config": {
            "clientId": "placeholder-github-client-id",
            "clientSecret": "placeholder-github-client-secret",
            "defaultScope": "user:email"
        }
    }
]
```

These use placeholder credentials for development. The Login page's `SocialIcon` component renders branded SVG icons for `github`, `google`, `microsoft`, and `apple` aliases, with a generic globe fallback for unknown providers.

The realm also configures:
- `registrationAllowed: true` with email-as-username
- `verifyEmail: true` with SMTP pointing to Mailpit (`mailpit:1025`)
- OAuth2 device authorization grant enabled for CLI authentication
- A pre-seeded admin user (`admin@example.com` / `password`)
