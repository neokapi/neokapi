# Add i18n to a project

Adds i18n the right way for the stack. Two paths: **adopt the kapi-react stack**
(zero-wrapper React i18n), or **plug kapi into** whatever i18n catalogs the
project already uses. The source text is always the key — no message IDs.

## Step 1 — detect the stack

```bash
kapi presets list --framework        # known stacks + their catalog layout
kapi presets show <stack> --json     # the exact source glob + target path
```

| Signal in the project | Path |
|---|---|
| React + a bundler (Vite/webpack), greenfield i18n, wants zero wrappers | **Path A: adopt kapi-react** |
| `@neokapi/kapi-react` already present | **Path A** (already adopted) |
| `react-i18next` / `i18next`, `react-intl`, `next-intl`, `vue-i18n`, `@angular/localize`, Flutter `pubspec.yaml`, `.po`, Android `strings.xml`, iOS `.strings`/`.xcstrings` | **Path B: plug in** |

Ask the user which they prefer if a React project could go either way.

## Path A — adopt the @neokapi/kapi-react stack (React)

Zero-wrapper: write plain JSX, a bundler plugin extracts it at build time. See
the [kapi-react quickstart](https://neokapi.github.io/web/neokapi/docs/kapi-react/quickstart).

1. Install: `npm install -D @neokapi/kapi-react` (+ `kapi` CLI on PATH).
2. Add the bundler plugin (Vite shown; webpack/rollup/esbuild also exported):
   ```ts
   import neokapi from "@neokapi/kapi-react/vite";
   export default defineConfig({ plugins: [neokapi({ mode: "runtime" }), react()] });
   ```
3. Scripts in `package.json`:
   ```json
   { "scripts": {
       "extract": "kapi-react extract",
       "compile": "kapi-react compile i18n/ --out public/translations"
   } }
   ```
4. Translate the extracted KLF with the `kapi` CLI (this is the pipeline):
   ```bash
   npm run extract
   kapi pseudo-translate i18n/ --target-lang qps     # readiness check first
   kapi ai-translate i18n/ --target-lang fr           # real translation (brand-aware if a profile is bound)
   npm run compile
   ```
5. Load at runtime in `main.tsx`:
   ```ts
   import { loadTranslations } from "@neokapi/kapi-react/runtime";
   await loadTranslations("fr", "/translations/fr.json").catch(() => {});
   ```
6. Add `@neokapi/kapi-react-lint` (oxlint/eslint) so the `t()` escape hatch is used correctly for the rare non-JSX strings.

## Path B — plug kapi into an existing stack's catalogs

kapi reads and writes the catalog format the stack already uses.

**One-command project setup** (for repeatable runs / CI): scaffold a `.kapi`
project pre-wired for the detected stack —

```bash
kapi init --framework flutter --target-locale fr --target-locale de
# writes a recipe mapping lib/l10n/app_en.arb → lib/l10n/app_{lang}.arb
```

Supported: `react-i18next`, `react-intl`, `nextjs`, `vue-i18n`, `flutter`,
`angular`. (`kapi-react` is rejected here — it uses Path A's bundler workflow.)

**Or translate per file** (no project needed):

1. Find the catalogs (`kapi presets show <stack>` gives the layout), e.g.:
   - react-i18next / i18next → `public/locales/{lng}/translation.json`
   - react-intl → `src/lang/{lng}.json`
   - next-intl → `messages/{lng}.json`
   - vue-i18n → `src/locales/{lng}.json`
   - Angular → `src/locale/messages.{lng}.xlf`
   - Flutter → `lib/l10n/app_{lng}.arb` (ARB is JSON)
   - gettext → `locales/{lng}/LC_MESSAGES/*.po`
2. Translate the source catalog into each target locale (format auto-detected):
   ```bash
   kapi ai-translate public/locales/en/translation.json --target-lang fr \
     -o public/locales/fr/translation.json --json
   kapi run ai-translate-qa -i messages/en.json --target-lang de --json   # translate + QA
   ```
   Structure, placeholders (`{{name}}`, `{0}`, ICU), and keys are preserved; only values translate.
3. **Android / iOS** use okapi-bridge filters (install with `kapi plugin install okapi-bridge`):
   ```bash
   kapi ai-translate app/src/main/res/values/strings.xml --map '*.xml=okf_android' --target-lang fr \
     -o app/src/main/res/values-fr/strings.xml
   kapi ai-translate en.lproj/Localizable.strings --map '*.strings=okf_macstrings' --target-lang fr \
     -o fr.lproj/Localizable.strings
   ```
4. Readiness first: `kapi pseudo-translate <catalog> --target-lang qps` surfaces hardcoded/untranslated strings before any real translation.

## How to apply

- Keep the English source text as the key — don't introduce message IDs.
- Bind a brand voice profile + termbase so translations are on-brand and consistent (see [localize.md](localize.md)).
- Verify by building/running the app in the target locale; `kapi pseudo-translate` is the fast pre-flight.
