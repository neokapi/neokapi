# Add i18n to a project

Adds i18n the right way for the stack. Two paths: **adopt the kapi-react stack**
(zero-wrapper React i18n), or **plug kapi into** whatever i18n catalogs the
project already uses. The source text is always the key — no message IDs.

## Step 1 — detect the stack

```bash
kapi presets list --framework        # known stacks + their catalog layout
kapi presets show <stack> --json     # the exact source glob + target path
```

| Signal in the project | Path |
|---|---|
| React + a bundler (Vite/webpack), greenfield i18n, wants zero wrappers | **Path A: adopt kapi-react** |
| `@neokapi/kapi-react` already present | **Path A** (already adopted) |
| `react-i18next` / `i18next`, `react-intl`, `next-intl`, `vue-i18n`, `@angular/localize`, Flutter `pubspec.yaml`, `.po`, Android `strings.xml`, iOS `.strings`/`.xcstrings` | **Path B: plug in** |

Ask the user which they prefer if a React project could go either way.

## Path A — adopt the @neokapi/kapi-react stack (React)

Zero-wrapper: write plain JSX, a bundler plugin extracts it at build time. See
the [kapi-react quickstart](https://neokapi.github.io/web/neokapi/docs/kapi-react/quickstart).

**kapi-react in a nutshell — change as little as possible:**

- **Leave translatable JSX text exactly as written.** The plugin extracts plain
  text nodes (`<h1>Your notes</h1>`, `<p>Welcome back, {name}.</p>`) at build time.
  Do **not** wrap JSX text in `t()` — that's the whole point of zero-wrapper.
- **Use `t("…")` only for strings the plugin can't see as JSX** — text held in
  variables/data arrays, `alert()`/toast messages, and translatable attributes
  (`placeholder`, `aria-label`, `title`, `alt`). The English source is the key.
  **Don't eyeball which strings these are — the linter finds them (step 4).** They
  render reactively only inside `<NeokapiProvider>` (below), so wrap the app.
- **The source English text is the key — never invent message IDs.**
- The components otherwise stay as they were: the diff after adoption should be the
  config (plugin, provider, scripts), `"use client"` on components that render
  translated text, and `t()` on the handful of non-JSX strings — nothing more. If
  you find yourself rewriting component bodies, you're doing too much.

The translation pipeline (step 4) is **CLI-driven** — `kapi-react` and `kapi` run
directly on the source files. You don't need to build or run the app to translate,
and there is **no `.kapi` project**, so `kapi verify` (which is project-scoped)
does not apply here — the readiness gate is `kapi pseudo-translate`. The npm
install, bundler plugin, and runtime loader (steps 1, 2, 5) only wire the compiled
translations into the running app; add them as config and don't block translation
on them.

**Next.js App Router:** the runtime is client-side, so any component that renders
translated JSX must be a Client Component (`"use client"` at the top of the file).
Server Components can't use the runtime. Add the bundler plugin via the `webpack`
hook in `next.config`:

```js
import neokapi from "@neokapi/kapi-react/webpack";
export default { webpack: (config) => { config.plugins.push(neokapi({ mode: "runtime", translationsDir: "./public/translations" })); return config; } };
```

**Wrap the app in `<NeokapiProvider>` and load the dict before first paint.** The
dictionary is fetched on the client. `<NeokapiProvider>` makes the subtree react to
locale changes (so a switch after load repaints); gating the first render on the
load avoids a flash of the source language. A client provider that resolves the
locale (here from `?lang`, but a route segment or cookie works the same), loads its
dict, and holds children until it's in place:

```tsx
"use client";
import { Suspense, useEffect, useState } from "react";
import { useSearchParams } from "next/navigation";
import { NeokapiProvider, loadTranslations } from "@neokapi/kapi-react/runtime";

function Gate({ children }: { children: React.ReactNode }) {
  const lang = useSearchParams().get("lang") ?? "en";
  const [ready, setReady] = useState(lang === "en"); // English is the source — no load needed
  useEffect(() => {
    if (lang === "en") return;
    loadTranslations(lang, `/translations/${lang}.json`).finally(() => setReady(true));
  }, [lang]);
  if (!ready) return null; // or a spinner — never the un-translated frame
  return <NeokapiProvider>{children}</NeokapiProvider>;
}

export default function I18nProvider({ children }: { children: React.ReactNode }) {
  return <Suspense fallback={null}><Gate>{children}</Gate></Suspense>;
}
```

Wrap the app in `<I18nProvider>` in `app/layout.tsx`. `<NeokapiProvider>` is what
makes a later locale switch repaint; the `ready` gate is what stops the first paint
from flashing the source language. Components that render translated text must be
Client Components.

1. Install (for the app's runtime): `npm install -D @neokapi/kapi-react` (+ `kapi` CLI on PATH).
2. Add the bundler plugin (Vite shown; webpack/rollup/esbuild also exported):
   ```ts
   import neokapi from "@neokapi/kapi-react/vite";
   export default defineConfig({ plugins: [neokapi({ mode: "runtime" }), react()] });
   ```
3. Scripts in `package.json`:
   ```json
   { "scripts": {
       "extract": "kapi-react extract",
       "compile": "kapi-react compile out/ --out public/translations"
   } }
   ```
4. **Run the linter to find the strings the extractor can't see — don't hunt by
   hand.** `@neokapi/kapi-react-lint` (oxlint or ESLint 8.57+/9/10) flags exactly the
   escape-hatch cases: user-facing strings in data arrays (`{ title: 'Q3 plan' }`
   rendered as `{title}`), translatable attributes (`placeholder`/`aria-label`/
   `title`/`alt`), and bare `{'literal'}` JSX expressions. Use the **strict** preset
   for the adoption pass (it enables `prefer-t-for-label-props`, the data-array
   detector), autofix what it can, then wrap the rest in `t()`:
   ```jsonc
   // .oxlintrc.json  → { "jsPlugins": ["@neokapi/kapi-react-lint/oxlint"], "rules": { … } }
   // or eslint.config.js → import { recommendedStrict } from "@neokapi/kapi-react-lint/eslint";
   ```
   ```bash
   npx oxlint --fix          # autofixes bare {'literal'} JSX; reports the rest
   # or: npx eslint . --fix
   ```
   `no-string-literal-jsx-expr` autofixes; `prefer-t-for-label-props`/`-expr` report
   (wrapping in `t()` + the import is left to you). After this pass every user-facing
   string is either plain JSX (auto-extracted) or `t()`-wrapped — nothing slips
   through as a hardcoded literal.
5. Translate the extracted KLF with the `kapi` CLI (this is the pipeline):
   ```bash
   npm run extract                                   # JSX + t() calls → source KLF in i18n/
   kapi pseudo-translate i18n/ --target-lang qps     # readiness check first
   kapi ai-translate i18n/ --target-lang fr          # translated KLF → out/ (brand-aware if a profile is bound)
   npm run compile                                   # compile out/ → public/translations
   ```
   `kapi ai-translate <dir>` writes the translated `.klf` files to `out/`, so
   `compile` reads `out/` (where the targets are), not the source `i18n/`. Add the
   `extract`/`compile` scripts to `package.json`.
6. Load at runtime: in Vite, `await loadTranslations("fr", "/translations/fr.json")`
   in `main.tsx` before render; in Next.js App Router, use the `<NeokapiProvider>` +
   gate pattern shown above.

## Path B — plug kapi into an existing stack's catalogs

kapi reads and writes the catalog format the stack already uses.

**One-command project setup** (for repeatable runs / CI): scaffold a `.kapi`
project pre-wired for the detected stack —

```bash
kapi init --framework flutter --target-locale fr --target-locale de
# writes a recipe mapping lib/l10n/app_en.arb → lib/l10n/app_{lang}.arb
```

Supported: `react-i18next`, `react-intl`, `nextjs`, `vue-i18n`, `flutter`,
`angular`. (`kapi-react` is rejected here — it uses Path A's bundler workflow.)

**Or translate per file** (no project needed):

1. Find the catalogs (`kapi presets show <stack>` gives the layout), e.g.:
   - react-i18next / i18next → `public/locales/{lng}/translation.json`
   - react-intl → `src/lang/{lng}.json`
   - next-intl → `messages/{lng}.json`
   - vue-i18n → `src/locales/{lng}.json`
   - Angular → `src/locale/messages.{lng}.xlf`
   - Flutter → `lib/l10n/app_{lng}.arb` (ARB is JSON)
   - gettext → `locales/{lng}/LC_MESSAGES/*.po`
2. Translate the source catalog into each target locale (format auto-detected):
   ```bash
   kapi ai-translate public/locales/en/translation.json --target-lang fr \
     -o public/locales/fr/translation.json --json
   kapi run ai-translate-qa -i messages/en.json --target-lang de --json   # translate + QA
   ```
   Structure, placeholders (`{{name}}`, `{0}`, ICU), and keys are preserved; only values translate.
   Flutter `.arb` is auto-detected natively (ICU placeholders + `@`/`@@` metadata
   preserved). For i18next projects, add `--format i18next` so plural-suffix keys
   (`key_one`/`key_other`) and `{{interpolation}}`/`$t()` nesting are grouped and
   protected.
3. **Android / iOS** are handled by native, offline readers — no `kapi plugin
   install` and no JVM. Pass `--format` for the formats whose file extension is
   shared, so the native reader is used even when okapi-bridge is installed (the
   bridge would otherwise claim `.xml`/`.strings`):
   ```bash
   # Android strings.xml — .xml is shared, so name the native format
   kapi ai-translate app/src/main/res/values/strings.xml --format androidxml --target-lang fr \
     -o app/src/main/res/values-fr/strings.xml
   # iOS String Catalog — Xcode 15+ default, one multi-locale file, auto-detected by .xcstrings
   kapi ai-translate Localizable.xcstrings --target-lang fr -o Localizable.xcstrings
   # Legacy iOS .strings / .stringsdict
   kapi ai-translate en.lproj/Localizable.strings --format applestrings --target-lang fr \
     -o fr.lproj/Localizable.strings
   ```
   `<plurals>` / `<string-array>`, `xliff:g` do-not-translate spans,
   printf / `%@` / `%#@var@` placeholders, and CLDR plural variations are
   understood and preserved.
4. Readiness first: `kapi pseudo-translate <catalog> --target-lang qps` surfaces hardcoded/untranslated strings before any real translation.

## How to apply

- Keep the English source text as the key — don't introduce message IDs.
- Bind a brand voice profile + termbase so translations are on-brand and consistent (see [localize.md](localize.md)).
- `kapi pseudo-translate` is the fast catalog pre-flight (surfaces hardcoded/missed strings).
- **Confirm the target locale actually renders — don't stop at "translated the catalog."**
  Run the app at the target locale (e.g. open `/?lang=ja`, or the route/cookie your
  app uses) and check that the visible UI is in that language. If it still shows the
  source language, the wiring is wrong, not the translation — usual causes: the
  string wasn't extracted (verify it's in `i18n/` and the compiled
  `public/translations/<lang>.json`), the rendering component is a Server Component
  (must be `"use client"`), or the dictionary loads *after* first paint (gate the
  render, above). Fix and re-check until the target locale renders, then you're done.
