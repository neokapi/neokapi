# specs/ — the spec knowledge base

The pin-or-don't-cite registry of the format specifications this repository
cites: which specs, at which pinned version, under which rights class, and —
where the publisher's license permits storing text at all — the immutable
snapshots and per-clause section files those citations resolve against.

Contracts: [format-maturity.md §2.4](../docs/internals/format-maturity.md)
(Knowledge axis) and [format-ops.md §7](../docs/internals/format-ops.md)
(artifact map). Evidence base:
[research/format-ops/SYNTHESIS.md](../docs/internals/research/format-ops/SYNTHESIS.md)
(§4, D9) and
[followup-spec-knowledge-base-acquisition.md](../docs/internals/research/format-ops/followup-spec-knowledge-base-acquisition.md).

## Purpose: pin or don't cite

A citation that cannot be resolved against a pinned snapshot is not a
citation. Every spec reference in a `spec.yaml` or `dossier.yaml` names a
catalog-registered spec id and pinned version;
`scripts/format-ops/check-citations.mjs` resolves the anchor and quote hash
against the pinned snapshot, never the live network (PDF-only specs resolve
by quote-hash). Live URLs are re-crawled only by the scheduled, non-blocking
watch job, which diffs the publisher's anchor/heading indexes and proposes
`{new anchor, new clause}` relocations as auto-PRs when something moves.

## Layout

| Path | Contents |
|---|---|
| `catalog.yaml` | one entry per governing spec: `{id, name, version, url, rights, watch, notes}` |
| `snapshots/<spec>/<version>/` | vendor-class snapshots — unmodified publisher text, committed |
| `snapshots/cache/<spec>/<version>/` | cache-class snapshots — gitignored, fetched/regenerated by tooling |
| `sections/<spec>/<version>/<anchor>.md` | per-clause retrieval units: one numbered clause per file, provenance frontmatter |

The `sections/` files are the retrieval substrate for AI test generation
([format-spec-cases.md](../docs/internals/format-spec-cases.md)) and the
per-format context pack — section-anchored clauses beat embedding chunks on
standards text. Sections of cache-class specs are themselves cache-class:
they are regenerated deterministically from the cached snapshot and never
committed.

## Rights matrix

Verbatim from the research synthesis,
[SYNTHESIS.md §4](../docs/internals/research/format-ops/SYNTHESIS.md) (last
paragraph):

> Rights matrix: IETF/WHATWG/Ecma/Unicode-data/Google = vendorable;
> W3C-TR/OASIS/MS-* = unmodified copies + implementation excerpts with
> notice; Unicode prose/Apple = cache-or-link only; **ISO/IEC never**
> (single-user license — use the free Ecma-376/OASIS-ODF twins,
> https://www.iso.org/files/live/sites/isoorg/files/store/en/PUB100206.pdf,
> https://ecma-international.org/policies/by-ipr/ecma-text-copyright-policy/).

And the red lines, [SYNTHESIS.md D9](../docs/internals/research/format-ops/SYNTHESIS.md):

> Rights red lines: ISO text never enters repo/cache/prompts (use Ecma/OASIS
> twins); Apple link-only; CC-BY-SA files separately licensed; W3C/OASIS
> excerpts ride the implementation carve-outs
> (https://www.w3.org/copyright/document-license-2023/,
> https://www.iso.org/copyright.html).

`catalog.yaml` records this as four rights classes:

| `rights` | Matrix bucket | Storage |
|---|---|---|
| `vendor` | IETF / WHATWG / Ecma / Unicode-data / Google (verbatim-vendorable); community specs per repo LICENSE (CommonMark, TOML, Fluent, JSON Schema, DTCG); **and** W3C-TR / OASIS / MS-* — these last three only as *unmodified copies + implementation excerpts with notice retained*, the condition recorded in the entry's `notes` | committed under `snapshots/<spec>/<version>/` |
| `cache` | Unicode prose (UAX/UTS/LDML); any publisher whose license is unverified | gitignored under `snapshots/cache/<spec>/<version>/`; the repo commits only the catalog entry and the citations |
| `link-only` | Apple | no stored text in repo or shared cache; citations carry the URL plus a short fair-use quote only |
| `never` | ISO/IEC | no repo, no cache, no prompts — cite and snapshot the free twins instead: Ecma-376 for ISO/IEC 29500, OASIS ODF for ISO/IEC 26300 |

CC-BY-SA material (CommonMark) is vendor-class but stays in clearly
separately-licensed files with its LICENSE alongside — never blended into
Apache-2.0-licensed files beyond short attributed quotes.

## Citation shape

Every citation in `spec.yaml` / `dossier.yaml` carries:

```yaml
{spec, version, url#fragment, clause, heading, quote, quote_sha256}
```

- `spec`, `version` — catalog id + pinned version; both must match `catalog.yaml`
- `url#fragment` — anchor-addressable wherever the publisher supports it
- `clause`, `heading` — publisher-native clause numbering and the heading
  text at capture time (the clause carries the citation for PDF-only specs,
  where anchors are weak)
- `quote` — at most one normative sentence; never a section dump
- `quote_sha256` — hash of the normalized quote, resolved against the pinned
  snapshot

Where the target is not anchor-addressable the recorded resolution mode is
`pinned-version-only` (see format-maturity.md §2.4).

## Who writes here

Per the format-ops artifact map ([format-ops.md §7](../docs/internals/format-ops.md)),
this tree is written by the **upstream-watch** and **case-gen** rituals:

- `upstream-watch` maintains the `catalog.yaml` pins and watch feeds, lands
  new snapshots when a pinned version moves, and proposes citation
  relocations when a live anchor index drifts.
- `case-gen` reads the `sections/` clause files as its retrieval substrate
  and lands the section extractions its per-clause case generation needs.

Snapshot population for the specs already cited is owned by the bootstrap
backfill plus upstream-watch — this scaffold deliberately ships no
snapshots. `catalog.yaml` is hand-curated (entries are reviewed like code);
everything under `snapshots/` is unmodified publisher text plus a provenance
record, and `sections/` files are deterministic extractions — neither is
hand-edited.
