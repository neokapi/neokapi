## Parser bugs fixed in this commit

1. **Trailing inline `# â€¦` comments** on structural lines confused the
   line classifier. MIF emits comments like `> # end of Color` after
   container close-`>` and `<MIFFile 9.00> # Generated by â€¦` after
   single-line statements. The original `parseMIF` matched only bare
   `>` and `>` at trimmed-end-of-line, so neither pattern closed any
   container â€” every fixture except `1187_crlf.mif` parsed to one big
   unclosed root statement and the writer fell back to a 47-byte stub.
   Fix: introduced `stripTrailingComment` which trims a trailing
   un-quoted `# â€¦` from the structural prefix before classification,
   correctly skipping `#` inside backtick-quoted string values
   (`<String \`width: #ffffff'>`).

2. **Whole-line `# â€¦` comments** at top level (the `# Options:` /
   `# Paragraph Text` block at the head of every FrameMaker-emitted
   MIF) were silently appended to `rawBuilder` but never re-emitted;
   the writer's skeleton path simply dropped them. The fix above
   classifies them as comments and threads them into the surrounding
   statement's raw text, so the skeleton round-trips them verbatim.

3. **Writer used SourceText for skeleton refs** â€” `writeFromSkeleton`
   always wrote `block.SourceText()` even when a target locale was
   set and a translation existed. Fixed to mirror `writeBlock`'s
   `HasTarget` / `TargetText` check; this is what unblocks the
   pseudo-translated round-trip from being byte-identical to the
   source.

## Divergence clusters (still open)

### Cluster A â€” translatable `<VariableDef>` strings inside `<VariableFormats>`

**Affects**: 28 of the 41 fixtures (every one whose first divergence is
`Continued)'>\\n > # end of Variablâ€¦` near offset 30kâ€“66k).

**Symptom**: Native treats `<VariableFormats>` as a leaf `Data` part
(it's not in `alwaysSkipTags`, but `processContainer` only descends
into `TextFlow` / `Tbls` / `Notes`). The okapi reference, with
`extractVariables: true` (default), pseudo-translates the
`<VariableDef \`(Continued)'>`-style values, e.g.
"`(Continued)`" â†’ "`Ä†ĹŤĹ„ĹŁÄ©Ĺ„Ĺ©Ä“ÄŹ)`".

**Root cause**: `processContainer` and `collectTranslatableParas` in
`reader.go` only handle the `TextFlow` family (TextFlow / Notes /
Tbls / Cell / CellContent / Row / TblBody / Tbl). MIF carries
translatable text in many more container types:
- `<VariableFormats><VariableFormat><VariableDef \`â€¦'>`
- `<XRefFormats><XRefFormat><XRefDef \`â€¦'>`
- `<PgfCatalog><Pgf><PgfNumFormat \`â€¦'>` (gated by
  `extractPgfNumFormatsInline`)
- `<PgfCatalog><Pgf><PgfNumString \`â€¦'>` (paragraph-prefix numbering
  text like `(5)\t`)

**Fix shape**: Extend the recursive descent in `processContainer` to
recognise these container subtrees and yield their translatable
leaves as `Block` parts. The skeleton emitter
(`findStringPositions`) needs the matching extension so the writer
can replay them via the existing block-ref mechanism.

**Reference**: see okapi
`net.sf.okapi.filters.mif.MIFFilter#processVariables`,
`#processXRefs`, `#processPgfCatalog`. The Java filter walks the
top-level statement list and dispatches by tag rather than
hard-coding a TextFlow-only descent.

### Cluster B â€” `<PgfNumFormat>` / `<PgfNumString>` translatable text

**Affects**: 4 fixtures (`904.mif`, `909-3.mif`, `Test02-v9.mif`,
`TestEncoding-v9.mif`, `TestEncoding-v10.mif` â€” first-diff snippet
`Table <n+\\>: '>\\n  <PgfNumberFontâ€¦`).

**Symptom**: The okapi reference pseudo-translates the literal "Table"
inside `<PgfNumFormat \`T:Table <n+\>: '>` to "`Ĺ˘Ă Ć€ÄşÄ“`"; the
auto-numbering placeholder `<n+\>` and the `T:` prefix come through
as inline codes / passthrough.

**Root cause**: Same as Cluster A â€” `<Pgf><PgfNumFormat>` lives under
`<PgfCatalog>`, which native treats as a single `Data` blob via
`alwaysSkipTags`. Removing PgfCatalog from `alwaysSkipTags` is *not*
the right fix â€” most of its content is non-translatable; only the
`PgfNumFormat`/`PgfNumString` leaves are. Need targeted descent
plus an inline-code finder for `<n+\>`, `<\>`, `<n=1\>` etc. (these
are FrameMaker auto-number building blocks, equivalent to the okapi
`MIFFilter` "extract pgf num format inline" path).

### Cluster C â€” `1187_crlf.mif` `<Char HardReturn>` between `<String>` runs

**Affects**: 1 fixture (`1187_crlf.mif`, the smallest one â€” also the
parity test's `first divergent fixture`).

**Symptom**: Source has
```
<ParaLine
 <String `ParaLine 1'>
 <Char HardReturn>
> # end of ParaLine
```
The okapi reference renders the pseudo-translated paragraph with
`ParaLine 1\n` collapsed into a single `<String>` value (the
`HardReturn` is folded into the string text as `\n` and then the
`<Char HardReturn>` line is dropped from the skeleton). Native
preserves the raw `<Char HardReturn>` line and writes the pseudo
back into the `<String>` only â€” so the output has both the
pseudo-translated text *and* the leftover `<Char HardReturn>` line,
making the file 25 bytes longer than the reference.

**Root cause**: When `extractHardReturnsAsText: true` (default) the
reader correctly inlines `<Char HardReturn>` into the block source as
`\n` â€” but the writer's skeleton refs don't know they should
delete the matching `<Char HardReturn>` skeleton line. Need to
either emit a `<Char>`-elision marker in the skeleton or have
`findStringPositions` widen each ref's `endOffset` to swallow any
trailing `<Char â€¦>` lines that contributed to the block text.

**Reference**: okapi rebuilds the paragraph from the textual model on
output (`MIFFilter.writeParagraph`), so `<Char>` statements that
were inlined as text never reappear as separate skeleton lines.

### Cluster D â€” output-encoding mismatch for high-codepoint pseudo

**Affects**: at least `1187_crlf.mif`, `1188_crlf.mif`,
`TestParaLines.mif` (where the divergence snippet shows mojibake-like
bytes such as `\xa4\xe0U\xe09)D\x13` instead of the expected
`Ć¤Ă Ĺ•Ă ÄąÄ©Ĺ„Ä“`).

**Symptom**: The `escapeMIF` writer escapes per-rune over a Go
`string`, then writes `byte(r)` â€” which truncates any rune â‰Ą 256 to
a single byte and silently drops the high bits. The
pseudo-translated text is full of latin-extended codepoints
(U+0100â€“U+024F) that all alias to the same byte.

**Root cause**: `core/formats/mif/writer.go` `escapeMIF`:
```go
default:
    out = append(out, byte(r))
```
This is wrong for any non-ASCII rune. MIF files are stored in their
declared encoding (commonly UTF-8 for modern MIF, but some older
fixtures are FrameRoman or shift-jis); the writer should encode the
rune in the declared encoding. The shortest fix that works for the
parity suite is to write the UTF-8 bytes of the rune
(`utf8.AppendRune`), since okapi's reference output is also UTF-8 for
the test target locale `fr-FR`. A complete fix would honour the
`<MIFFile>` line's declared encoding and the `<MIFNOTE>` charset
hints used by `JATest.mif` / `TestEncoding-v9.mif`.

### Cluster E â€” `<MIFEncoding>` declaration round-trip

**Affects**: `TestEncoding-v9.mif`, `TestEncoding-v10.mif`,
`JATest.mif` (and probably `1188_crlf.mif`'s smaller diff at offset
6251).

**Symptom**: Out of scope for this round, but flagged: MIF files can
declare a non-UTF-8 character set via `<MIFEncoding â€¦>`. The native
reader/writer have no charset awareness â€” they pass bytes through as
Go strings via `bufio.Scanner`, which assumes UTF-8 and corrupts
non-UTF-8 input silently. The bridge has full charset handling via
okapi `Statements` lexer.

## Triage suggestion for the next iteration

The single highest-impact fix is **Cluster A + B** (extend
`processContainer` to descend into `VariableFormats`, `XRefFormats`,
and `PgfCatalog`'s translatable leaves, and update
`findStringPositions` to record their byte spans). That alone moves
~32 of 41 fixtures from "div" to "byte-equal" once the encoding fix
(Cluster D) is also in.

The encoding fix (Cluster D) is small and orthogonal â€” `escapeMIF`
should use `utf8.AppendRune` instead of `byte(r)`. Without it,
byte-equality is unreachable even after the extraction work because
the pseudo-translated text always includes high-codepoint runes.

Cluster C is fixture-specific and Cluster E is a larger encoding
project (at least a day) that mostly affects two fixtures â€” both
should be deferred behind the higher-impact work.
