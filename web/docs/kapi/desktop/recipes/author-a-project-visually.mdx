discovery), see [Modes &amp; bindings](/kapi/modes) and
[Your first project](/kapi/get-started/first-project); this recipe walks creating

**Goal:** stand up a project — its languages, content patterns, and flows — through
the app, ending with the same committable `.kapi` recipe you'd get from

## Steps

1. **Create the project.** Choose a folder; the app writes a `<name>.kapi` recipe
   and the sibling `.kapi/` state dir — the same layout `kapi init` produces.
2. **Declare languages.** Pick the source language and the locales you translate
   into. These drive every flow's defaults.
3. **Group content into collections.** Add glob patterns (docs, UI strings,
   contracts) as named collections. The Content view shows the patterns alongside
   the real files they match and the format Kapi detected for each.
4. **Bind flows.** Attach flows from the [flow editor](/kapi/desktop/recipes/build-a-flow-visually)
   so the project runs them with its declared languages and defaults.
5. **Commit the recipe.** The `.kapi` file is portable YAML — commit it, and the
   project runs identically from the CLI and for teammates.

## Why the desktop

Visual authoring keeps the project's shape in view — which globs match which files,
what format each is, which flows are bound — and validates as you go. The artifact
is an ordinary `.kapi` recipe: nothing is desktop-only, so the project is fully
drivable from `kapi` on the command line and in CI.
