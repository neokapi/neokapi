# Localize audio & video

kapi localizes a media asset two independent ways, and an asset can use both:

- **Translate the extracted content** — the speech in audio, the spoken track and
  on-screen text in video, the text OCR'd from an image. The recognized text
  translates and round-trips into a text-carrying artifact (a subtitle track, an
  alt-text sidecar).
- **Replace the whole asset** — the file itself is the deliverable: you supply a
  per-locale version and the project treats it as authoritative.

The recognition runs locally through plugins, mirrored exactly by the
[in-browser labs](/lab/audio). This is the architecture in
[AD-030](/contribute/architecture/030-multimodal-extraction-and-llm-refinement).

## Install the engine plugins

The readers are built in, but the recognition/demux engines ship as plugins.
Install only what a format needs:

```bash
kapi plugin install asr      # speech recognition (audio + the video speech track)
kapi plugin install av       # video demux (audio track + sampled frames)
kapi plugin install vision   # image / frame OCR
```

They are also available from Homebrew (`brew install neokapi/tap/kapi-asr`,
`kapi-av`, `kapi-vision`). Without the engine a media file still opens — as an
opaque, replaceable `Media` asset — so the *replace-asset* path needs no plugin.

## Audio → subtitles

`kapi-asr` transcribes the speech into timing-anchored cues; `ai-translate` fills
the target; the subtitle writer (chosen by the output extension) emits a localized
`.vtt`/`.srt`/`.ttml`:

```bash
kapi run audio-to-subtitles -i talk.m4a -o talk.fr.vtt --target-lang fr
```

## Video → subtitles

Video extraction yields *two* streams — the spoken track and on-screen frame
text. `video-to-subtitles` runs a `subtitle-filter` step first so only the
timing-anchored speech reaches the subtitle track; on-screen text (geometry
anchored) is dropped from the spoken subtitles:

```bash
kapi run video-to-subtitles -i clip.mp4 -o clip.fr.vtt --target-lang fr
```

When neither `kapi-av` nor `kapi-asr` is installed the video opens as an opaque
`Media` asset instead of erroring — ready for the replace-asset path below.

## Image text → translated

`kapi-vision` OCRs the image into geometry-anchored blocks; `ai-translate`
translates them; the image writer round-trips the localized caption to an
`<image>.alt.txt` sidecar (the pixels are not re-rendered):

```bash
kapi run image-ocr-translate -i banner.png --target-lang fr
```

Low-confidence lines can be re-read by a configurable multimodal LLM with the
`media-refine` tool — see
[AD-030](/contribute/architecture/030-multimodal-extraction-and-llm-refinement).

## Replace the whole asset

When the deliverable is the file itself, pair the source with a per-locale variant
and the project treats it as authoritative — medium-agnostic across image, audio,
and video. In a [`.kapi` project](/kapi/modes) the variant lives beside the source
on the locale path (for example `assets/fr/promo.mp4`); `kapi pull`/`kapi push`
and the writer carry that file through unchanged. The engine never *synthesizes*
localized media (no TTS, no re-encode) — a replacement is a file you or a
connector provide.
