# Modes & bindings

**New here?** Start with [Projects vs ad-hoc](/kapi/get-started/project-vs-adhoc)
for the plain-language version. This page is the **mechanics** ‚Äî for when you want
to know exactly how a run resolves.

Kapi doesn't have separate "ad-hoc commands" and "project commands." It has **one
set of commands**, and what changes is the **binding** ‚Äî where content comes from
and where results go. Ad-hoc vs project is just which binding gets used.

## The binding model: source ‚Üí flow ‚Üí sink

A **flow** is just an ordered set of tools ‚Äî it owns no input or output. Where the
content enters (the **source**) and where results go (the **sink**) are *bindings*
resolved when you run it:

| Binding | As source | As sink |
| --- | --- | --- |
| `file` (default) | read + parse a file | write the file back (round-trip via skeleton) |
| `store` | the project store (blocks + overlays) | commit overlays ‚Äî **no file** |
| interchange | import XLIFF / PO / a bilingual `.klz` | emit one for a translator |
| `none` | ‚Äî | discard (checks/analysis only) |

The mode you're in is just which binding gets resolved:

- **Ad-hoc** = the `file` binding ‚Äî you name `-i`/`-o`.
- **In a project**, the default *sink* becomes the **store** (a *process-only* run);
  pass `-o` to force a file instead.

A process-only run persists only *working* output ‚Äî the target overlays in the
block store, pre-merge. It does **not** touch the committed **state store** (the
review decisions in `.kapi-state.json`): those are authored by `kapi apply`, not
by a flow's sink. Caches are rebuildable; the state store is the one committed,
authored artifact. See [the project store](/kapi/project-store).

Run `kapi run <flow> --explain` any time to print the resolved `source ‚Üí sink`
without executing:

```bash
kapi run translate-qa -i app.json -o app.fr.json --explain
#   ‚Üí flow translate-qa: file(app.json) ‚Üí file(app.fr.json)
kapi run translate-qa -i app.json --explain          # inside a project
#   ‚Üí flow translate-qa: file(app.json) ‚Üí store
```

## Ad-hoc: loose files

```bash
# One tool on one file
kapi pseudo-translate app.json --target-lang qps         # ‚Üí app_qps.json
kapi translate -i app.xliff --target-lang fr -o fr.xliff
kapi qa app.xliff -o none --report qa.json         # checks only, no output

# A composed flow
kapi run translate-qa -i app.json -o app.fr.json --target-lang fr
```

## Project: two loops, both ending at `merge`

Set up and manage the tracked content (these are core commands ‚Äî no plugin needed):

```bash
kapi init
kapi add "src/**/*.json"   # format auto-detected from the extension
kapi ls --stats            # list tracked files + block/word counts
```

**1 ‚Äî Interchange loop** (hand bilingual files to a translator or CAT tool):

#   ‚Ä¶translate the bilingual files‚Ä¶
kapi merge -i out/                          # apply back onto sources, absorb into TM
```

```

## The `.klz` parcel: move work without a server

A `.klz` is a portable, single-file carrier ‚Äî the way you hand work between
machines or people without a server. It comes in two profiles:

```bash
# Whole-project snapshot ‚Äî a runnable "project in a file"
kapi pack -o snapshot.klz        # carries the recipe + content + TM/termbase
kapi unpack snapshot.klz         # rehydrate elsewhere

# Bilingual interchange ‚Äî neokapi's native format for a kapi-equipped reviewer
kapi extract --format klz --target-lang fr   # ‚Üí one bilingual .klz per pair
kapi merge bundle.fr.klz                      # ingest the returned work
```

Source travels as identity + a round-trip skeleton by default; add `--with-source`
to embed the raw originals (for offline re-extraction).

## Do ad-hoc and project mode mix?

Yes ‚Äî because they're the same commands over different bindings, not rival
toolsets. The guidance:

| You want‚Ä¶ | Use |
| --- | --- |
| A throwaway job on one file | **ad-hoc** (`kapi <tool>` or `run -i -o`) |
| A repo you localize repeatedly (accumulating TM) | **project** (`add` / `extract` / `run`‚Üístore / `merge`) |
| To move work between machines/people | **`.klz` parcel** (`pack` or `extract --format klz`) ‚Üí `merge`/`unpack` |

**Coherent mixes:** a tool run *inside* a project (`kapi translate src/app.json`)
still picks up the project's defaults and TM while working on a single file; the
same flow runs ad-hoc or process-only with no change; a `.klz` made in a project
is consumed ad-hoc by a reviewer and `merge`d back.

**The one thing to avoid:** running **two stateful stores over the same content** ‚Äî
an ad-hoc `.klz` workspace *and* a `.kapi` project for the same files. Use the
`.klz` workspace when there is no project; use the project store when there is.

**Worth knowing:** inside a project, an explicit `-o` writes a file and skips the
store (process-only) ‚Äî that's intentional, and `--explain` shows you which you got.
