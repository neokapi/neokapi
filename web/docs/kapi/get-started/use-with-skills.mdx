# Use the Kapi Agent Skills

kapi ships a portable **Agent Skill** (a `SKILL.md`). With it installed, you don't
run `kapi` yourself — you describe the outcome in plain language, and your AI
coding assistant loads the skill, runs the right `kapi` commands, reads the
results, and iterates until the work passes kapi's gates. kapi supplies the
guardrails (faithful round-trip, brand voice, terminology); the assistant does the
writing and translating; you stay in the conversation.

`SKILL.md` is an [open standard](https://github.com/agentskills/agentskills), so
the skill works in **Claude Code, GitHub Copilot, Cursor, Windsurf, VS Code,
Codex**, and the rest of the ecosystem. This page is written around **Claude
Code** — the richest integration — but the loops apply to any tool that runs the
skill.

## Install

Pick the path for your tool. All paths need the `kapi` CLI on your `PATH`
(`kapi version`; see [installation](/kapi/get-started/installation)) — keep it
updated so the skill matches your command surface.

| Your tool | Install | Updates |
|---|---|---|
| **Claude Code** | the plugin (below) — skill **+ hooks** | automatic |
| **Copilot · Cursor · Windsurf · VS Code · Codex · …** | `npx skills add neokapi/agent-skills` | `npx skills update` |
| **Any MCP client** (structured tool calls) | run `kapi mcp` — see [Use Kapi with MCP](/kapi/get-started/use-with-mcp) | — |

### Claude Code — the plugin

The richest path: the **Claude Code plugin** bundles the skill *and* two hooks
that make the loop a guarantee (see [Hooks](#what-the-hooks-add)):

```text
/plugin marketplace add neokapi/claude-plugins
/plugin install kapi@neokapi-plugins
```

Restart Claude Code. The plugin **updates itself** through Claude Code's plugin
manager — republished on each kapi release — so your guidance stays current with
no manual step.

### Other tools — the portable skill

The same skill installs into any `SKILL.md`-aware tool with the
[`skills`](https://github.com/vercel-labs/skills) CLI:

```bash
npx skills add neokapi/agent-skills                    # all detected tools
npx skills add neokapi/agent-skills -a github-copilot  # one tool
```

It copies the skill into your tool's skill directory (`.github/skills/`,
`.cursor/skills/`, …); run `npx skills update` to refresh it. You get the skill's
guidance, but not the Claude-Code hooks (there is no portable equivalent). For
kapi's structured tools in these clients, also run the MCP server (`kapi mcp`).

Either way, the skill is a small router the assistant loads progressively, so it
stays cheap in context — the assistant **discovers it automatically** and invokes
it when a task fits its description; you never name the skill or the commands.

### What the hooks add

Installing the **plugin** (not the skill alone) also registers two Claude Code
hooks, both scoped to the `.kapi` project in the session and both **fail-open**
(outside a project, or if they can't run, Claude proceeds normally):

- **Stop hook** — when Claude tries to finish, it runs `kapi verify`; if a brand,
  terminology, or translation-QA gate is failing, it keeps Claude working with the
  findings. The skill makes the verify loop the default; the hook makes it a
  guarantee.
- **PreToolUse hook** — denies direct hand-edits to files the project *generates*
  as translation targets, steering the change through the source and the
  round-trip instead.

No AI-provider credential is needed for the attended work below: Claude writes the
content and `kapi` applies and checks it, with no second model in the loop. A
credential (`kapi credentials add`) is only for the **unattended** paths —
`kapi translate` run without you, or the optional `--ai` checks.

## Start here: let Claude set up the project

The first thing to ask for is the project itself. The guardrails and your prompts
both get simpler once the source files, brand voice, and glossary are captured in
a committed `.kapi` recipe.

> Set kapi up for this project.

**What Claude does.** It uses kapi together with its own reading of the repo to
bootstrap everything:

1. **Detect the stack and find the source.** It looks at the layout —
   `package.json`, config, where strings live — to identify the project type. For
   a known i18n stack it runs `kapi init --framework <stack>` (it can list the
   supported stacks with `kapi presets list --framework`), which pre-fills the
   content mapping; otherwise `kapi init` plus `kapi add "<glob>"` for the
   translatable files it found.
2. **Detect the brand voice.** It drafts a `brand.yaml` from signal you already
   have — README, landing copy, UI strings, or a site you link — starting from the
   `kapi brand new` template, and binds it in the recipe. You confirm or refine the
   draft.
3. **Extract terminology.** It surfaces candidate terms — recurring domain phrases,
   product names — from its own reading of the content (kapi's `term-extract` tool
   does the same unattended, via a provider), then lands the ones you approve in
   the project glossary through `kapi apply` (a `{"kind":"term", …}` change-set
   entry, which writes the committed glossary source and compiles it into the
   cache).

The result is committed to git: a recipe, a brand profile, and a glossary —

```yaml
# myapp.kapi
version: v1
name: myapp
defaults:
  source_language: en
  target_languages: [fr, de, ja]
  brand_voice:
    profile_file: l10n/brand-voice.yaml
content:
  - path: "src/locales/en/*.json"
    target: "src/locales/{lang}"
```

— so from here your prompts stay short. "Translate the docs" or "check this is on
brand" needs no flags or file lists: Claude resolves the languages, voice, and
glossary from the recipe, and `kapi check` / `kapi verify` enforce them
automatically. See [Create your first project](/kapi/get-started/first-project)
for the recipe model in full.

## How brand voice and terminology are applied

This is the part that keeps the output on-brand and consistent, and it flows in
**two directions** around Claude:

- **Context in — *before* Claude writes.** Claude pulls your brand voice with
  `kapi brand guide` (kapi renders it as a short markdown guide — tone, do/don't,
  preferred wording — which Claude injects into its own context) and the approved
  term for anything ambiguous with `kapi termbase lookup`. So the draft or
  translation starts on-voice and on-term, rather than being corrected afterward.
- **Gate out — *after* Claude writes.** kapi checks the result against those same
  sources: `kapi check` (or `kapi verify` in a project) scores the text against the
  brand profile and flags forbidden/competitor terms and off-glossary wording,
  each finding anchored to the exact block. Claude fixes what's flagged and
  re-checks until the gate is green.

When the voice and glossary are bound in a `.kapi` project ([above](#start-here-let-claude-set-up-the-project)),
both directions happen **automatically** — Claude resolves them from the recipe,
so you don't restate a profile or glossary on each request. And for localization,
a bound brand voice is injected into the translation prompt itself, so
target-language output is on-brand at generation time, not merely checked after.

This left-intake / right-gate loop is the same whether you're authoring or
localizing; the next two sections show each in use.

## The author path — write and fix content, on brand

Ask Claude to produce or revise content and hold it to your rules. For example:

> Draft release notes for v2.1 from `CHANGELOG.md`, and make sure they're on brand.

> Check `README.md` against our brand voice and fix anything off-brand.

> Tighten the intro of `report.docx` to two sentences — keep the formatting.

> Go through `docs/` and fix anything that fails our content rules.

**What Claude does.** It loads the brand voice with `kapi brand guide` (so it
writes on-voice in the first place), drafts or edits the content, then runs the
loop kapi is built around:

1. `kapi check <file>` — kapi returns a `kapi.check/v1` Report: a 0–100 score, a
   pass/fail gate, and one finding per block (which rule, where, a suggested fix).
2. For each flagged block, Claude rewrites the text itself and writes it back with
   `kapi apply` — kapi reconstructs the file in its original format, **structure
   and inline codes intact**, and rejects any edit that would corrupt markup. No
   model is called for the rewrite; Claude is the writer.
3. `kapi check` again. When the gate is green (exit 0), Claude stops and tells you.

Because the check is the gate, Claude keeps iterating on its own until the content
passes — you don't have to review every step, just the result. This works on any
format kapi reads, including ones your editor can't open (more below). To drive
the same loop by hand, see
[Edit content in any format](/kapi/recipes/rewrite-content) and
[Keep your source on brand](/kapi/recipes/keep-source-on-brand).

### Editing a document you can't open

The author path shines on binary formats. Ask:

> What does `pitch.pptx` say on slide 3, and can you make the headline punchier?

Claude reads the deck's text with `kapi inspect pitch.pptx --jsonl` (each block as
`id` + `content_hash` + text, with links and styling shown as `<x id="…"/>`
placeholders), rewrites the headline, and applies it with `kapi apply`. The
`.pptx` keeps every style, master, and shape — only the headline text changes.

## The localization path — translate and publish

Ask Claude to ship the content in other languages. For example:

> Translate `announcement.docx` into Japanese.

> Localize `src/locales/en.json` into French and German, using our glossary.

> Re-translate the docs that changed since the last release into Norwegian.

> Get `report.docx` ready for a translation vendor in French.

**What Claude does.** It translates the content itself and routes it **through
kapi** so the guardrails apply — it does not read the file, translate in its head,
and write the target directly (that would skip terminology, placeholder integrity,
and format). The shape depends on the ask:

- **A single file, translated in place.** Claude pulls the approved wording with
  `kapi termbase lookup`, translates each block, and writes the localized file
  back through the same faithful round-trip — the `.docx`/`.json` keeps its
  structure, only the text is in the target language.
- **Project localization.** In a `.kapi` project it runs `kapi extract` (a
  bilingual file of source + empty targets), fills the targets, runs `kapi merge`
  to materialize the localized files, then `kapi verify` — the gate that checks
  terminology, placeholder integrity, and that nothing was left untranslated.
- **Hand-off to a vendor or human reviewer.** `kapi extract` produces a standard
  bilingual interchange file (XLIFF/PO); `kapi merge` consumes the returned
  translations back into the original format.

Whichever shape, Claude finishes by verifying, and fixes what the gate flags
before telling you it's done. A provider credential is only needed if you want
`kapi translate` to run **unattended** (a CI job, no assistant in the loop). To
drive localization by hand, see
[Translate and publish](/kapi/recipes/translate-content) and the
[Bilingual workflow](/kapi/bilingual-workflow).

## Gate it in CI

The same checks Claude runs locally gate a pull request — `kapi check` (or
`kapi verify` in a project) exits non-zero when the gate fails, so on-brand,
correctly-localized content is enforced even when no assistant is in the loop.
