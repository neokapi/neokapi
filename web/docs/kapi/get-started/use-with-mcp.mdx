# Use Kapi with MCP

`kapi mcp` exposes kapi's content operations as **structured MCP tools**, so any
MCP-compatible client — Claude Desktop, Claude Code, Cursor, Windsurf, VS Code —
can read a document's blocks, check them against rules, and edit them while
preserving the file's structure. The assistant calls a tool when a task needs it;
you describe the work in plain language.

The `kapi` CLI must be on your `PATH` — confirm with `kapi version` (see
[installation](/kapi/get-started/installation)). The edit loop —
`extract_content` to read, `apply_edits` to write — needs no provider credential:
the assistant supplies the edits and kapi applies them, with no model. A provider
added with `kapi credentials add` is only for unattended translation
(`kapi translate`) and the optional `--ai` checks.

:::tip Driving the CLI from Claude Code instead?
If you use **Claude Code**, the [Kapi Agent Skills](/kapi/get-started/use-with-skills)
let Claude drive the `kapi` CLI by natural language — no tool wiring. The two
approaches can be used together.
:::

## Set up the MCP server

`kapi mcp` launches a stdio JSON-RPC server; your client starts it as a
subprocess — you do not run it manually.

**Claude Desktop** — add to
`~/Library/Application Support/Claude/claude_desktop_config.json` on macOS or
`%APPDATA%\Claude\claude_desktop_config.json` on Windows, then restart Claude
Desktop:

```json
{
  "mcpServers": {
    "kapi": {
      "command": "kapi",
      "args": ["mcp"]
    }
  }
}
```

**Claude Code** — create or edit `.mcp.json` at the repository root:

```json
{
  "mcpServers": {
    "kapi": {
      "command": "kapi",
      "args": ["mcp"]
    }
  }
}
```

Claude Code discovers and connects automatically.

**GitHub Copilot** — Copilot speaks MCP (GA in VS Code, the Copilot CLI, Visual
Studio, and JetBrains IDEs). In VS Code, add `.vscode/mcp.json` at the repository
root:

```json
{
  "servers": {
    "kapi": {
      "command": "kapi",
      "args": ["mcp"]
    }
  }
}
```

Copilot then calls kapi's tools automatically — `extract_content`, `apply_edits`,
`check_file`, `brand_check`. For the **guidance** that drives the loop (when to
reach for which tool), also install kapi's Agent Skill — Copilot reads `SKILL.md`
natively: `npx skills add neokapi/agent-skills -a github-copilot`. See
[Use the Kapi Agent Skills](/kapi/get-started/use-with-skills) for the skill across
all tools.

See [MCP server](/reference/mcp) for Cursor, Windsurf, Visual Studio, and
JetBrains configurations and the full tool list.

## The tools

`kapi mcp` registers a focused set of structured tools:

- **`extract_content`** — the read leg: read a file's blocks into context, one
  record per block with its text (inline codes rendered as `<x id="…"/>`
  placeholders), role, a stable `id`, and a `content_hash`.
- **`apply_edits`** — the one write verb: apply a typed change-set. Content edits
  land through the byte-faithful round-trip (structure and inline codes
  preserved, drift-guarded by `content_hash`); asset edits (glossary term, TM
  pair, brand rule, recipe field) are written to their committed source and
  compiled into the cache. No provider is used.
- **`check_file`** / **`check_text`** — run the configured checkset over a file
  path or a plain string and return a `kapi.check/v1` Report.
- **`brand_check`** — score a text snippet against a brand voice profile without
  reading a full file.

`extract_content` returns one record per block, so the assistant can reason about
a document before editing it:

```json
[
  {
    "file": "README.md",
    "number": 1,
    "id": "p0",
    "content_hash": "e3b0c44...",
    "role": "heading",
    "level": 1,
    "text": "Getting started"
  },
  {
    "file": "README.md",
    "number": 2,
    "id": "p1",
    "content_hash": "a1b2c3...",
    "role": "body",
    "text": "Install the CLI and run your first check."
  }
]
```

`check_file` / `check_text` return a Report with a 0–100 score, a pass/fail gate,
and one finding per block — its location, the rule it violated, and a suggested
fix. The stable `rule` id in each finding is the loop's primary key: the
assistant tracks it across iterations to confirm a fix and detect regressions.
See [Checks](/framework/checks) for the full rule model and Report structure.

## The author → check → revise loop

The tools compose into a fix loop with no scaffolding:

1. The assistant calls `extract_content` to read the blocks, or `check_file` to find
   what failed — the Report returns which block, which rule, and where.
2. It rewrites the flagged blocks itself and calls `apply_edits` with a
   change-set of `content` entries (each naming the block's `id`, the
   `content_hash` it read, and the new text). No second model is called.
3. It calls `check_file` again. When the gate is green, it stops.

If an edit drifted (`content_hash` no longer matches) or would corrupt an inline
code, `apply_edits` returns `ok: false` and leaves that block unchanged, so the
assistant re-inspects and retries. This works for any file type kapi reads —
Markdown, JSON, DOCX, HTML, YAML, and the rest.

## Brand voice

Bind a brand voice profile to a `.kapi` project file and every check picks it up
automatically:

```yaml
# myapp.kapi
defaults:
  brand_voice:
    profile_file: brand.yaml
```

With a profile bound, `check_file` adds vocabulary and style findings across every
block in a document; `brand_check` scores a snippet against a profile without
reading a full file. See [Brand voice](/framework/checks/brand-voice) for the
profile format, pack model, and vocabulary rule syntax.
