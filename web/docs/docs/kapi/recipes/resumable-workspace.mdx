# Work in progress with `.klz`

Some work doesn't finish in one place or one shot. You might pre-translate now and
review later, or segment on one machine and translate on another. A **`.klz`** is
the portable in-progress form of a flow: write it instead of a finished file, then
resume from it whenever you like.

## `.klz` as in-progress output

Any tool or flow can write a `.klz` instead of a translated file — just give the
output a `.klz` extension:

```bash
kapi pseudo-translate app.json -o work.klz      # writes the working state, not app.fr.json
```

`work.klz` carries the original source plus the work done so far (the per-block
overlays). To produce the finished document, run again **from** the `.klz`:

```bash
kapi pseudo-translate work.klz -o app.fr.json   # resumes, writes the finished file
```

The result is byte-identical to a one-shot `kapi pseudo-translate app.json -o
app.fr.json` — the `.klz` round-trip just lets you stop and continue. This needs
**no project**: the `.klz` *is* the workspace.

Resuming reuses the work already in the package instead of recomputing it, so the
expensive steps (AI translation, MT) don't run again. You can also continue into a
new `.klz` to carry the work forward, or hand it to another machine in between:

```bash
# Machine A — start the work, save the in-progress package
kapi run ai-translate-qa -i app.json -o work.klz

#   ... copy work.klz to machine B (no server, no account needed) ...

# Machine B — resume and finish
kapi run ai-translate-qa -i work.klz -o app.fr.json
```

Several inputs pack into one `.klz`, and resume writes them back out — into a
directory (one file per source, keeping each name) or via an `-o` template. Each
document's work stays isolated:

```bash
kapi pseudo-translate src/*.json -o batch.klz
kapi pseudo-translate batch.klz  -o qps/                # → qps/<name>.json per source
kapi pseudo-translate batch.klz  -o 'out/{name}.json'   # or a template
```

(When you read a `.klz`, pass `-o` for the destination — there's nothing to infer
it from. A single input with a plain path, e.g. `-o app.fr.json`, writes that one
file.)

## Project snapshots: `pack` / `unpack`

Inside a `.kapi` project, the working state is bigger than one document's overlays
— it also includes the project's translation memory and termbase. `kapi pack`
snapshots all of it into a `.klz`, and `kapi unpack` rehydrates it into another
machine's `.kapi/` state dir:

```bash
kapi pack   -o snapshot.klz   # snapshot the project's working state
kapi unpack snapshot.klz      # rehydrate elsewhere, then run to resume
```

`pack` excludes regenerable caches and secrets, so a snapshot is the source of
truth, not a pile of derived files. For a custody trail, `kapi pack --log` stamps
a tamper-evident, hash-chained provenance line into the snapshot (`unpack` verifies
it). A plain `pack` is byte-deterministic.

Within a project, re-running a flow also reuses the project's persistent cache
(`.kapi/cache/blocks.db`), so resume there is simply running again — no `.klz`
needed unless you're moving the state somewhere.

## How resume stays correct

Progress is derived from the content itself — the per-block overlays — never from
a separate status file. Each block is addressed by a hash of its source, so a
block you've already translated is found and reused, and a block whose source you
changed gets a new hash and is redone. That's what makes resume idempotent
(re-running a finished step is a no-op) and safe across a crash (work that didn't
commit simply isn't there, so the next run redoes it).
