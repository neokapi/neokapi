# Hand off a project mid-flow with `.klz`

Some work can't finish in one place. You might segment and pre-translate on CI,
do the licensed machine translation on an air-gapped machine, and finish back on
CI. kapi supports this with two pieces that need almost no new command surface:
**cached resume** (re-running skips work already done) and **`pack` / `unpack`**
(move the working state as a portable `.klz`).

## Resume is just running again

Caching is a **project** feature. Inside a `.kapi` project — passed with `-p` or
auto-discovered by walking up from the current directory, like git — a run
executes against the project's persistent block store (`.kapi/cache/blocks.db`).
Tools cache their per-block results there, so the *next* run reuses them instead
of recomputing:

```bash
# run from inside a .kapi project (or pass -p recipe.kapi)
kapi run ai-translate-qa -i app.json -o app.fr.json   # translates, caches results
kapi run ai-translate-qa -i app.json -o app.fr.json   # reuses the cache — no re-translation
```

The second run is byte-identical and doesn't call the translator again for blocks
it already did. There's no "resume" flag — the store *is* the workspace, and
running again continues from it. This is correct because the store is
content-addressed: a block you've already translated is found by its hash, and a
block whose source you changed gets a new hash, so only the changed content is
redone.

Outside a project (a standalone `kapi run` with no `.kapi` anywhere up the tree),
the run uses an ephemeral in-memory store: it produces the same output but caches
nothing between invocations, so there is no cross-run resume. The persistence
that makes resume and `pack`/`unpack` work lives in the project's `.kapi/` state
directory.

## Move the working state: `pack` / `unpack`

To continue on another machine, snapshot the working state into a `.klz` and
rehydrate it there:

```bash
# Machine A — pre-translate, then snapshot
kapi run ai-translate-qa -i app.json -o app.fr.json
kapi pack -o snapshot.klz

#   ... copy snapshot.klz to machine B (no server needed) ...

# Machine B — rehydrate and resume
kapi unpack snapshot.klz
kapi run ai-translate-qa -i app.json -o app.fr.json   # resumes against the warm cache
```

`pack` captures the block-store overlays plus the authoritative translation
memory and termbase. It deliberately **excludes** regenerable caches and secrets,
so a snapshot is the source of truth, not a pile of derived files.

## Tamper-evident provenance (optional)

For a hand-off you may want a custody trail. `pack --log` stamps a hash-chained
line into an advisory `history.jsonl` inside the snapshot recording the pack;
`unpack` verifies the chain and warns if it was altered:

```bash
kapi pack -o snapshot.klz --log
```

The log is **purely advisory**: it's excluded from the package's content hash,
nothing reads it back to decide anything, and deleting it loses no work. A plain
`pack` (without `--log`) is byte-deterministic — packing the same state twice
produces identical bytes.

Progress is always derived from the content itself — the overlays present in the
store — never from a separate status file that could drift from it.
