# Run a bilingual round-trip

**Goal:** send source content out for translation and get it back in, without
losing track of what changed. The **extract → translate → merge** round-trip
through XLIFF is the canonical way to do this: extract emits one bilingual XLIFF
per target locale (pre-filled from your TM), a translator or CAT tool fills the
`<target>` elements, and merge applies them back onto the source files while
recording every new pair in the TM with full provenance.

## The full round-trip

```text
seed TM → kapi extract → translator fills <target> → kapi merge → kapi tm audit
```

The complete loop is shown below: seed the TM from a corporate TMX, extract
per-locale XLIFFs, simulate a translator filling them in, merge back, and trace
the result with `kapi tm audit`. The translator step is an external editor
action and `tm audit` reads the SQLite TM, so the full loop runs against the
native binary rather than the in-browser build.

## Steps

1. **Seed the TM** (one-time) so extract can pre-fill known matches:
   `kapi tm import corporate.tmx --file .kapi/tm.db --all-pairs`.
2. **Extract** one bilingual XLIFF per target locale: `kapi extract -p app.kapi`.
3. **Translate** — a person or CAT tool fills each `<target>`, preserving the
   kapi batch IDs the merge relies on.
4. **Merge** the returned XLIFFs back onto the source: `kapi merge -i out/`.
5. **Audit** what the merge wrote: `kapi tm audit --batch <id>`.

## Try the extract step

`kapi extract` runs entirely in the browser. A small recipe (`app.kapi`) and a
source catalogue are already in the in-browser project folder. Run the step,
then open the emitted XLIFF under `out/` in the files pane.
